## 简答版（1-2分钟）

巴西现金贷H5是面向巴西市场的在线信用贷款平台，SPA架构 + Node中间层，300万周活用户。我是前端负责人，带5人前端小组，在前后端人力比1:4的情况下保障业务敏捷迭代。

项目有三个核心挑战：授信流程因多渠道、多版本AB实验导致逻辑混乱；首页LCP一度高达10秒以上严重影响转化；线上问题缺乏有效监控、响应慢。

我分别用三套方案解决：引入可配置任务流引擎解耦授信逻辑；通过接口聚合、代码分割、Node层承接AB实验等系统性手段优化性能；整合集团监控平台并制定SOP流程。

结果：LCP从10秒优化到4秒，全年0起P3及以上事故，故障响应时间缩短50%，全年无前端原因导致的需求延期。

## 展开版（5分钟+）

### 背景与挑战

巴西现金贷H5是国际化金融业务的核心产品，SPA架构 + Express Node中间层，周活跃用户约300万。我作为前端负责人，带领5人团队，核心挑战有三层：

**1. 流程复杂度：** 授信环节（活体验证、证件上传、信息收集）在精细化运营下演变出多渠道（优质/普通用户）× 多版本（AB实验）× 多端的组合，页面逻辑耦合严重，新增一个渠道需要改动大量现有代码。

**2. 性能瓶颈：** 首页LCP（75分位）一度高达10秒以上，严重影响用户转化率。金融产品的用户留存对加载速度非常敏感。

**3. 可观测性缺失：** 线上问题靠用户反馈发现，排查无章法，风险逐步积累，随着业务规模增长风险越来越大。

### 方案选型

**问题一：授信流程混乱**
传统做法用 if-else 判断渠道和版本，代码越加越乱，新增渠道必然引入回归风险。

我的方案：设计可配置的任务流引擎，每条授信链路由一组有序任务组成，新渠道只需配置一条任务序列，不改现有代码。核心思想是「流程配置化，页面只负责渲染当前任务」，实现流程与页面组件的解耦。

**问题二：性能优化**
没有银弹，系统性诊断各个环节：
- 路由懒加载 + 代码分割，减少首屏 bundle 体积
- Node中间层聚合首页依赖的多个接口为一个，减少首屏请求数和往返时延（贡献最大）
- AB实验灰度逻辑从客户端移到 Node 层处理，避免首屏等待实验接口

**问题三：监控与SOP**
整合集团已有监控平台，接入核心链路的性能和错误指标；制定线上问题处理SOP：发现 → 定级 → 响应时限 → 排查路径 → 修复验证，让整个团队都能按流程处理，不依赖单个关键人。

### 关键实现

1. **任务流引擎**：定义任务接口（入参/出参/跳转条件），引擎持有任务序列配置，根据用户渠道和实验版本选取对应序列顺序执行，页面组件只渲染当前任务
2. **Node中间层**：在 Express 层实现 `/api/init` 接口聚合首页所有依赖、AB实验动态接入、首页HTML托管
3. **性能优化**：webpack-bundle-analyzer 分析体积后针对性拆分，配合 Node 层接口聚合减少首屏依赖
4. **监控体系**：接入集团平台设置核心指标告警，制定 SOP 文档并落地团队培训

### 结果与影响

| 指标 | 改善前 | 改善后 |
|---|---|---|
| 首页LCP（75分位） | 10秒以上 | 4秒以内 |
| 全年P3及以上事故 | — | 0起 |
| 故障平均响应时间 | — | 缩短50% |

## 高频追问 Q&A

**Q1：任务流引擎具体怎么设计的？**
核心是把"授信该走哪条链路"的决策从页面里抽出来，统一放在引擎里。每个授信环节定义为一个 Task（有唯一ID、执行函数、完成后的跳转目标）。引擎持有一份任务序列配置（JSON），根据用户渠道和实验版本选取对应序列，按顺序执行任务，任务完成后引擎决定下一步跳转。页面组件只关心"当前任务是什么"，不关心前后的流程逻辑。

**Q2：Node中间层做了哪些接口聚合？**
首页加载依赖多个接口（用户状态、贷款产品信息、AB实验分组等），原先在客户端串行/并行请求，任何一个慢都影响首屏。我在 Node 层把这些聚合成一个 `/api/init`，服务端并发请求、合并结果后返回给客户端，客户端一个请求拿到所有首屏数据。这是LCP改善最显著的单项优化。

**Q3：LCP从10秒到4秒，哪些手段贡献最多？**
接口聚合贡献最大，节省了多个串行请求的时间；其次是首屏代码分割，把非首屏路由的 JS 延迟加载；第三是将 AB 实验决策移到 Node 层，避免客户端等待实验接口返回才能渲染。

**Q4：带5人团队，遇到过什么管理挑战？怎么解决的？**
最典型的两个：代码质量参差不齐，以及需求估时不准。前者通过建立代码规范文档 + CR 流程解决，规范不是约束而是减少 CR 来回沟通的成本；后者通过迭代 buffer 机制（每个迭代预留弹性时间）+ 风险提前暴露来解决，全年做到无前端原因导致的延期。

**Q5：监控怎么衡量效果？**
核心指标是告警发现时长（从问题出现到告警触发）和 P3 事故数量。建立监控后，故障平均响应时间缩短了50%，全年无P3及以上。数据来自集团监控平台的月度报告。

## 容易答偏的地方

1. **不要一上来就说"LCP优化"**。这只是三个核心问题之一。应该先讲你的角色（前端负责人，5人团队，300万用户），再分层次讲三个挑战和对应方案，最后汇总数据。顺序：人 → 挑战 → 方案 → 结果。

2. **"前后端1:4"是个好的叙事入口**，它说明了资源约束下你做到了什么，比单纯说"我优化了性能"更有说服力，也自然引出了为什么要做任务流引擎和接口聚合（降低开发成本）。
