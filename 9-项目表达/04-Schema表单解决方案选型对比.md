# Schema 表单解决方案选型对比：Formily / XRender / form-create / Amis

- 创建日期: 2026-05-11
- 标签: [Formily, XRender, form-create, Amis, 选型, Schema驱动, 中后台]
- 状态: 已完成

---

## 一、选型背景与核心约束

**场景**：中后台运营平台，签约模块含 100+ 字段的复杂表单，多国业务规则差异大。
目标是引入"元数据平台 + 渲染引擎"组合：产品在元数据平台可视化配置表单结构并输出 JSON Schema，前端渲染引擎消费 JSON 完成渲染，彻底解耦业务逻辑与页面代码。

**由此确立 5 个硬约束：**

| # | 约束 | 原因 |
|---|---|---|
| C1 | **JSON Schema 原生驱动** | 元数据平台输出 JSON，渲染引擎必须能直接消费，否则配置化没有意义 |
| C2 | **声明式字段联动** | 100+ 字段场景，命令式 watch 极易写出蜘蛛网依赖，维护成本不可接受 |
| C3 | **可嵌入已有 Vue 工程，低侵入** | 项目已有成熟架构，渲染层不能接管状态管理、路由和 HTTP 层 |
| C4 | **支持渐进式引入** | 不允许全量迁移停工，新方案必须能与旧代码并存，逐场景替换 |
| C5 | **适配自研组件库** | 团队有自研 UI 组件库，表单控件必须能替换为自研组件，不能被方案的内置 UI 锁死 |

> **核心逻辑**：先用约束排除不可选的方案，剩下的再比优劣。

## 二、逐方案评估

### XRender（阿里）

**定位**：轻量级 Schema 表单渲染，API 简洁，上手门槛低。

| 约束 | 结论 | 说明 |
|---|---|---|
| C1 JSON Schema 驱动 | ✅ 支持 | 内置 FR-Schema 格式，可渲染表单 |
| C2 声明式联动 | ❌ 不满足 | 联动通过 `watch` 回调命令式触发，字段多时依赖追踪全靠人工维护 |
| C3 低侵入嵌入 | ❌ 不满足 | **官方只维护 React 版本，无官方 Vue 支持**；社区移植不成熟，引入已有 Vue 工程需自行承担维护风险 |
| C4 渐进式引入 | ❌ 不满足 | Vue 项目无法引入 |
| C5 适配自研组件库 | ❌ 不满足 | Vue 项目无法引入，自然无法对接自研组件库 |

**排除原因**：C2 + C3 + C4 + C5 均不满足。核心问题是 XRender 是纯 React 库，没有官方 Vue 版本，在 Vue 工程里根本无从引入。

---

### form-create（社区）

**定位**：Vue 生态较活跃的 JSON Schema 表单库，支持 Element Plus / Ant Design Vue / iView。

| 约束 | 结论 | 说明 |
|---|---|---|
| C1 JSON Schema 驱动 | ⚠️ 部分满足 | 使用自定义 `rules` 格式，**不是标准 JSON Schema**，与元数据平台输出的标准 Schema 需要转换层 |
| C2 声明式联动 | ⚠️ 部分满足 | `control` 系统只支持**单字段触发**；多字段联合条件、异步联动需要 `handle` 回调硬编码，无内置 loading 状态管理 |
| C3 低侵入嵌入 | ✅ 支持 | 渲染层组件，不影响应用架构 |
| C4 渐进式引入 | ✅ 支持 | 可按页面/场景逐步替换，旧代码不干扰 |
| C5 适配自研组件库 | ✅ 支持 | 支持注册自定义组件（`customComponents` API），可将自研控件挂入表单渲染流程 |

**C2 具体说明——form-create 联动能力边界：**

form-create 的联动基于 `control` 配置，结构是"**单字段触发 → 多目标响应**"：

```js
// form-create 的联动写法：country 变化 → 控制 city 显隐
{
  field: 'country',
  control: [{ value: 'CN', method: 'display', rule: ['city'] }]
}
```

这套机制对简单显隐、赋值够用，但有三个硬限制：

1. **无法多字段联合触发**：`control` 只能响应单个字段的变化，"字段 A 和字段 B 同时满足条件时才触发"需要拆成多个 `control` 块手动组合
2. **异步联动无内置支持**：`control` 的 `handle` 回调可以写异步函数，但没有内置的 `loading` / `error` 状态，异步期间的 UI 状态全靠手动管理
3. **动态注入有约束**：官方文档明确建议"提前把关联组件都定义在 rules 里"，动态字段注入路径不如 Formily 完整

同样的场景在 Formily 中：

```json
// Formily 的联动写法：在 Schema 里声明，city 的选项依赖 country + region 两个字段，且是异步加载
{
  "field": "city",
  "x-reactions": "{{fetchCityOptions}}"
}
// scope 里注册 fetchCityOptions(field) { field.loading=true; fetch(...).then(action.bound(data => { field.dataSource=data; field.loading=false })) }
```

`x-reactions` 支持 `dependencies: ['.country', '.region']` 多字段依赖，`field.loading` 内置异步状态，`@formily/reactive` 自动追踪依赖图——这些不需要手写。

**C1 具体说明——Schema 格式不兼容：**

form-create 使用自定义的 `rules` 数组格式（`{ field, type, props, control, ... }`），不是标准 JSON Schema（Draft-7）。如果元数据平台输出的是标准 JSON Schema，接入 form-create 需要额外写一层格式转换，这本身就是维护负担。

**排除原因**：C1 + C2 均不满足。核心问题是 Schema 格式不兼容（需要转换层）+ 联动系统无法覆盖多字段联合条件和异步场景，而这两个恰好是我们 100+ 字段、元数据平台驱动场景的硬需求。

---

### Amis（百度）

**定位**：完整 Low-code 渲染引擎，可从 JSON 渲染整个页面，包含组件库、状态管理、HTTP 层、路由。

| 约束 | 结论 | 说明 |
|---|---|---|
| C1 JSON Schema 驱动 | ✅ 支持 | 核心能力，整页 JSON 渲染 |
| C2 声明式联动 | ✅ 支持 | 内置表达式和事件系统，可声明式描述联动 |
| C3 低侵入嵌入 | ❌ 不满足 | 自带完整运行时（状态、HTTP、路由），嵌入已有 Vue 工程冲突严重；样式定制困难，难与现有 UI 风格对齐 |
| C4 渐进式引入 | ❌ 不满足 | Amis 需要接管渲染上下文，局部嵌入技术上可行但维护成本极高；旧代码与 Amis 管理的组件通信需要额外桥接层 |
| C5 适配自研组件库 | ❌ 不满足 | Amis 深度绑定自身组件体系，自定义渲染器可注册但样式隔离困难，与自研组件库对接成本极高 |

**排除原因**：C3 + C4 + C5 均不满足。我们只需要表单层，Amis 是整套 Low-code 平台，嵌入已有工程会把架构主权交出去，且无法平滑渐进迁移，也难以对接自研组件库。

---

### Formily（阿里）✅ 选定

**定位**：企业级表单解决方案，支持 React / Vue 2 / Vue 3，核心是 `@formily/reactive` 声明式响应系统。

| 约束 | 结论 | 说明 |
|---|---|---|
| C1 JSON Schema 驱动 | ✅ 原生支持 | `SchemaForm` 直接消费标准 JSON Schema，无需转换层 |
| C2 声明式联动 | ✅ 完整满足 | `reactions` 配置声明依赖关系，`@formily/reactive` 自动追踪依赖图，100+ 字段场景可维护 |
| C3 低侵入嵌入 | ✅ 满足 | 定位是渲染层，不干涉应用状态管理和路由；`x-component` 扩展点允许注册自定义 Vue 组件处理边缘场景 |
| C4 渐进式引入 | ✅ 满足 | `SchemaForm` 是组件级，可按场景逐步替换旧表单；新旧方案可在同一页面并存，不需要全量迁移 |
| C5 适配自研组件库 | ✅ 满足 | `x-component` 机制允许将任意 Vue 组件注册为表单控件，自研组件库中的每个组件都能挂入 Schema 渲染流程，无需修改组件本身 |

**加分项**：
- 阿里内部大规模中后台项目长期验证
- `x-component` 提供逃生通道——Schema 表达不了的特殊交互，可注册 Vue 组件挂入，不强行限制边缘需求
- Vue 2 / Vue 3 均有维护，社区活跃

## 三、决策汇总表

| 方案 | C1 Schema驱动 | C2 声明式联动 | C3 低侵入 | C4 渐进式引入 | C5 自研组件库 | 适合场景 |
|---|---|---|---|---|---|---|
| **Formily** | ✅ 原生 | ✅ 完整 | ✅ | ✅ | ✅ x-component | 中后台超复杂表单、需 Schema 驱动、要与低代码平台对接 |
| **XRender** | ✅ | ❌ 命令式 | ❌ 无Vue支持 | ❌ | ❌ | React 技术栈、中等复杂度表单、联动逻辑简单 |
| **form-create** | ⚠️ 自定义格式 | ⚠️ 单源触发 | ✅ | ✅ | ✅ customComponents | Vue 技术栈、字段数 < 30、联动逻辑简单、不需要对接标准 JSON Schema |
| **Amis** | ✅ | ✅ | ❌ 重侵入 | ❌ 难渐进 | ❌ 样式隔离难 | 从零搭建内部 Low-code 工具平台、不依赖已有工程架构 |

## 四、面试口述脚本

### 30 秒版（只讲约束 + 结论）

> 选型时我先确立了五个约束：JSON Schema 原生驱动、声明式联动、低侵入嵌入已有 Vue 工程、支持渐进式引入、能适配自研组件库。这五个约束把 XRender（纯 React，Vue 工程无法引入）、form-create（复杂联动要硬编码绕过）、Amis（整套运行时侵入性太强，自研组件库难对接）都排掉了。Formily 五个约束全部满足，所以选它。

### 90 秒版（约束 + 逐步排除 + 选定理由）

> 背景是中后台签约平台，单场景 100+ 字段，需要与元数据平台对接——产品可视化配置表单，输出 JSON Schema，渲染引擎消费。这个目标确定了五个硬约束：JSON Schema 原生驱动、声明式联动（命令式 watch 在 100+ 字段下维护成本不可接受）、低侵入嵌入已有 Vue 工程、支持渐进式引入（不允许停工全量迁移）、必须能适配我们团队的自研组件库。
>
> 逐一对比：XRender 是纯 React 库，没有官方 Vue 版本，在 Vue 工程里根本无从引入，直接排除。form-create Vue 生态活跃，但有两个硬伤：一是它用自定义 `rules` 格式而非标准 JSON Schema，元数据平台输出的标准 Schema 需要额外转换层；二是联动系统（`control`）只支持单字段触发，多字段联合条件和异步联动都要 `handle` 回调硬编码，没有内置 loading 状态管理——这两点恰好是我们场景的核心需求，所以排除。Amis 功能最强，但自带完整运行时——状态、HTTP、路由全包，嵌入已有工程侵入性极大；自研组件库对接也因 Amis 深度绑定自身组件体系而成本极高，排除。
>
> Formily 五个约束全部满足：SchemaForm 原生消费 JSON Schema，reactions 声明式描述联动依赖，定位是渲染层不干涉应用架构，组件级引入按场景逐步替换旧表单，最关键的是 x-component 机制可以把自研组件库里的任何控件注册进表单渲染流程，无需改动组件本身。最终选 Formily。

## 五、高频追问 Q&A

**Q1：你说的"声明式联动"和"命令式联动"有什么本质区别？**

命令式联动：你写 `watch('fieldA', (val) => { setFieldB(val) })`，字段多时每个依赖关系要手动声明，散落各处，改一个字段要排查所有相关 watch，很容易遗漏或形成循环依赖。

声明式联动（Formily reactions）：在字段 B 的配置里描述"我依赖字段 A"，`@formily/reactive` 自动建立依赖图，字段 A 变化时所有依赖它的字段自动更新，框架负责追踪和避免循环。100+ 字段场景下，声明式的可维护性远高于命令式。

**Q2：Formily 和 XRender 最本质的区别是什么？**

核心差异在响应式模型。XRender 本质是"Schema → UI"的静态渲染器，字段联动是渲染外的事，需要你自己 watch 并手动触发。Formily 内置了 `@formily/reactive` 响应式状态管理，字段间依赖关系在运行时自动追踪——这不只是 API 风格差异，而是解决问题的层次不同：一个是渲染库，一个是带状态管理的表单运行时。

**Q3：为什么不用 Amis？它功能更强不是更好吗？**

功能强不等于适合这个场景。Amis 给的是整套 Low-code 运行时——HTTP 层、路由、状态、组件库全包，嵌入已有工程时这些和已有架构冲突。更关键的是渐进迁移问题：局部嵌入 Amis 后，旧代码和 Amis 管理的组件通信需要额外桥接层，实际上制造了两套体系并存的维护负担。Amis 适合从零搭建内部工具平台，不适合作为嵌入已有工程的表单层。

**Q4：渐进式引入具体怎么操作的？**

Formily 的 `SchemaForm` 是组件级的，直接在需要改造的页面引入即可，不影响其他页面。我们的实际做法：先选一个复杂度中等的签约场景做试点，新方案写完后和原有页面双跑，功能对齐后正式切换，切换后原有代码再删除。这样每次只有一个场景处于迁移状态，风险可控，也不需要停工。

**Q5：form-create 在什么场景是合适的选择？**

Vue 技术栈 + 字段数量有限（约 30 以内）+ 联动逻辑简单（只涉及单字段触发的显隐/赋值）+ 不需要对接标准 JSON Schema 的普通业务表单。它上手成本低，与 Element Plus / Ant Design Vue 集成开箱即用，`control` 配置直观，不需要理解 `@formily/reactive` 那套概念。

如果业务复杂度在这个范围内，form-create 是性价比更高的选择；一旦需要多字段联合触发、异步联动状态管理、或者对接标准 JSON Schema 的元数据平台，form-create 的天花板会很快到来。

## 容易答偏的地方

1. **不要把对比说成"哪个更好"**。正确框架是"在我的约束下，哪个满足要求"——Amis 在 Low-code 平台场景是最佳选择，form-create 在简单 Vue 表单场景完全够用，这不是优劣排名，是场景适配问题。

2. **被追问"Formily 缺点"时要诚实**。学习曲线陡峭（`@formily/reactive`、`SchemaField`、`x-component` 体系需要时间理解）、Bundle 体积较大、Debug 体验不如简单库直观。说出来反而显得你真的深入用过。

3. **渐进式迁移是重要加分项**。很多候选人只讲"选了什么"，不讲"怎么落地的"。主动提到渐进迁移策略（试点→双跑→切换→清理）能体现工程判断力，不只是技术选型能力。
