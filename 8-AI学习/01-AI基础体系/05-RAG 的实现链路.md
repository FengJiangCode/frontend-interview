# 05-RAG 的实现链路

## 先说结论

如果只记一句话，可以先记这个：

- RAG 的核心不是“让模型变聪明”
- 而是“先把相关知识找出来，再让模型基于这些知识回答”

所以它的关键链路通常是：

1. 处理用户问题
2. 去知识库里检索相关内容
3. 召回最相关的片段
4. 把这些片段放进上下文
5. 再交给大模型生成答案

也就是说，RAG 的重点不是只有最后那一步生成，而是整条“找资料 -> 选资料 -> 喂给模型 -> 再回答”的链路。

## 1. 为什么需要实现链路视角

如果只从概念上理解 RAG，很容易停留在一句话：

```text
先检索，再生成
```

但真正落到系统里，问题会马上变多：

- 从哪检索
- 检索什么
- 怎么判断哪些片段最相关
- 片段太多怎么办
- 怎么拼接到上下文里
- 最终怎么生成答案

所以你要真正理解 RAG，必须看到它的实现链路，而不是只记定义。

## 2. 一条典型的 RAG 链路长什么样

一个最常见的 RAG 流程可以先记成：

```text
用户问题
-> 问题预处理
-> 检索知识库
-> 召回相关片段
-> 选择/排序片段
-> 拼接进上下文
-> LLM 生成答案
-> 输出结果
```

有些系统会再加：

- 查询改写
- 重排
- 引用来源
- 答案校验

但对初学阶段来说，先把这条主链搞懂就够了。

## 3. 第一步：处理用户问题

用户问题不会总是天然适合检索。

比如用户问：

```text
我们项目里登录失败的问题之前怎么处理过？
```

系统通常不会直接把整句话原封不动拿去检索，有时候会先做一些预处理：

- 提取关键词
- 改写问题
- 标准化表达
- 拆成更适合检索的查询

### 这一步解决什么问题

它主要解决：

- 用户表达太口语化
- 问题太长
- 检索关键词不够清晰

## 4. 第二步：检索知识库

这一步就是去外部知识里找相关信息。

知识库可能来自：

- 企业文档
- FAQ
- 产品说明
- 接口文档
- 项目复盘
- 历史工单

这里最重要的一点是：

模型不是直接知道这些知识，而是系统去查这些知识。

### 检索方式常见有两类

#### 1. 关键词检索

更像传统搜索：

- 看用户 query 里有哪些词
- 去文档中匹配这些词

优点：

- 简单直接
- 可解释性强

缺点：

- 对表达差异不够友好

#### 2. 向量检索

更像“按语义相似度查”：

- 先把 query 变成向量
- 再去找语义最接近的文档片段

优点：

- 能处理近义表达
- 语义召回能力更强

缺点：

- 结果有时没那么直观
- 调试门槛更高

现实里很多系统会把两者结合起来。

## 5. 第三步：召回相关片段

检索之后，不是整库都给模型，而是只拿最相关的一部分。

这一步就是召回。

比如：

- 先找到 top 5 文档片段
- 再从里面选 2 到 3 段最有价值的内容

### 为什么不能全塞给模型

因为：

- 上下文长度有限
- 信息太多会稀释重点
- 无关内容会干扰生成结果

所以 RAG 里非常重要的一点是：

不是“找到了资料就行”，而是“找到足够相关、足够干净、足够可用的资料”。

## 6. 第四步：切片与组织内容

知识库里的原始文档通常不会直接整篇拿去喂模型。

更常见的做法是先切片。

比如一篇很长的制度文档，可能会切成：

- 一段一段
- 按章节切
- 按标题切
- 按固定 token 长度切

### 为什么要切片

因为：

- 检索粒度更细
- 更容易找到真正相关的小片段
- 更方便控制上下文长度

### 切片也有代价

切得太碎，可能丢上下文。  
切得太大，又可能召回不准。

所以切片不是机械动作，而是会直接影响 RAG 效果。

## 7. 第五步：排序或重排

有些系统检索完后，还会再做一次排序。

原因是：

- 初次检索找到的片段不一定最优
- 可能有“有点相关但不是最重要”的内容排得太前

所以常见做法是：

- 先粗召回一批
- 再精排一遍

这样能提升最终放进上下文里的内容质量。

你可以把它理解成：

- 第一次是“先大致找出来”
- 第二次是“再挑最该给模型看的”

## 8. 第六步：把资料注入上下文

这是 RAG 真正和生成连接起来的关键一步。

系统会把：

- 用户当前问题
- 检索到的相关片段
- 可能还有系统约束

一起组成新的 Prompt / Context，然后交给大模型。

比如会变成：

```text
请基于下面资料回答用户问题。

资料 1：...
资料 2：...
资料 3：...

用户问题：...
```

这一步很关键，因为：

- 资料放得太乱，模型不容易用
- 资料太多，模型容易抓不住重点
- 资料没标来源，后续引用也不方便

## 9. 第七步：生成答案

到了这一步，大模型才开始真正生成答案。

但这时它不是“凭空回答”，而是“基于当前上下文里的检索资料来回答”。

这就是 RAG 和普通大模型问答的关键区别。

### 这一步的目标

通常包括：

- 给出更贴资料的答案
- 降低胡编乱造概率
- 尽量附上来源或依据

### 但要注意

即使做了 RAG，也不等于绝对不会错。

因为错误可能来自：

- 没检索到正确资料
- 召回片段不够相关
- 切片方式不合理
- 模型总结时仍然理解偏了

所以 RAG 是提高准确性的方法，不是绝对正确保证。

## 10. 一个最小可用的 RAG 思维模型

如果你现在不想记太多实现细节，可以先记这个最小模型：

```text
问题 -> 查资料 -> 选资料 -> 塞上下文 -> 再回答
```

只要你记住这 5 步，后面再看向量库、召回、重排、chunking 这些词，就不会完全乱掉。

## 11. 工作中的真实例子

### 例子 1：企业问答助手

用户问：

```text
报销流程是什么？
```

系统内部可能是：

1. 识别问题主题是“报销”
2. 去制度知识库里检索相关文档
3. 召回“报销流程说明”和“差旅规范”
4. 选出最相关的两段
5. 放进上下文
6. 让模型生成回答并附引用来源

这就是标准 RAG 流程。

### 例子 2：研发文档助手

工程师问：

```text
这个接口为什么会报 401？
```

系统内部可能是：

1. 检索鉴权文档
2. 检索接口说明
3. 检索历史问题记录
4. 把相关片段给模型
5. 模型结合资料生成排查建议

这里重点仍然是：先查知识，再生成。

## 12. RAG 最容易出问题的地方

### 1. 检索不到

如果知识库里没有，或者 query 不对，后面再强的模型也没用。

### 2. 召回不准

召回来的是“有点像”，但不是最关键的内容。

### 3. 切片不合理

重要信息被拆散，或者上下文丢了。

### 4. 上下文太乱

资料堆太多、结构不清晰，模型没法很好利用。

### 5. 生成阶段仍然总结错

资料是对的，但模型归纳时仍然可能出错。

## 13. 面试怎么说

如果面试官问你“RAG 的实现链路是什么”，可以这样回答：

```text
RAG 的核心链路通常是先处理用户问题，再去知识库检索相关内容，召回并筛选最相关的文档片段，然后把这些片段注入到当前上下文里，最后再交给大模型生成答案。
它的关键不只是最后的生成，而是前面的检索、召回、切片、排序和上下文组织，因为这些步骤会直接决定模型最后拿到的资料质量。
```

这个回答的重点是：

- 讲清主链路
- 讲清前半段的重要性
- 讲清 RAG 不是只有“查一下数据库”这么简单

## 14. 复习时建议重点自测

- RAG 为什么不是单纯“查资料后回答”这么简单
- 为什么切片方式会直接影响 RAG 效果
- 为什么不能把所有检索结果都塞给模型
- RAG 的错误通常可能出在哪几个环节
- 如果让你讲一遍 RAG 的实现链路，你能不能按顺序讲清楚
