# 03-Prompt、Context、Memory、Tool Use

## 先说结论

如果只记一句话，可以先记这个：

- Prompt 决定“你让模型做什么”
- Context 决定“模型当前看到了什么”
- Memory 决定“系统能不能记住过去”
- Tool Use 决定“模型能不能接触外部能力”

这四个概念经常一起出现，因为它们共同决定了一个 AI 系统到底能不能稳定完成任务。

如果只靠 Prompt，而没有足够的 Context、Memory 或 Tool Use，系统通常只能完成很短、很轻、很局部的任务。

## 1. Prompt 是什么

Prompt 可以理解为你给模型的任务说明。

它的核心作用是告诉模型：

- 你要做什么
- 应该按什么方式做
- 输出应该长什么样
- 有哪些限制条件

比如：

- 帮我总结这份需求文档
- 帮我分析这段代码的 bug
- 请按“问题、原因、修复建议”格式输出

### Prompt 解决什么问题

它主要解决：

- 任务目标不明确
- 输出格式不稳定
- 模型容易跑偏

### Prompt 的边界

Prompt 很重要，但它不是万能的。

如果系统缺少外部知识、缺少历史状态、缺少工具能力，只靠一段 Prompt 往往做不了复杂任务。

## 2. Context 是什么

Context 可以理解为模型当前这一次推理时所看到的全部上下文。

它通常包括：

- 用户当前的问题
- 系统提示词
- 前面对话内容
- 检索到的知识片段
- 工具调用返回结果

所以 Prompt 只是 Context 的一部分，而不是全部。

### Context 解决什么问题

它主要解决：

- 模型当前应该基于什么信息回答
- 当前这轮任务里哪些信息最重要

### 工作里怎么理解

如果你问模型：

```text
帮我分析这个 bug
```

但没有把日志、代码、接口文档给它，那 Context 就不够。

如果你把这些信息都提供进去，模型的判断才更有依据。

## 3. Memory 是什么

Memory 可以理解为系统对过去信息的记忆能力。

它和 Context 的区别是：

- Context 更偏“当前这一轮模型看到了什么”
- Memory 更偏“系统过去记住了什么，并在后面还能继续用”

Memory 常见形式有：

- 对话历史摘要
- 用户偏好记录
- 长期任务状态
- 某次执行中间结果的保存

### Memory 解决什么问题

它主要解决：

- 多轮对话里前后信息断裂
- 长任务过程中状态丢失
- 系统每次都像“失忆”一样重新开始

### 一个直观例子

如果用户前面说过：

- 这个项目是 Vue3
- 不要引入第三方库
- 输出尽量面试化

后面系统还能记得这些偏好并继续使用，这就体现了 Memory 的价值。

## 4. Tool Use 是什么

Tool Use 可以理解为模型调用外部工具或能力的方式。

这些工具可能包括：

- 搜索工具
- 文件读取工具
- 数据库查询
- 浏览器操作
- 调用某个 API

它的核心不是“模型自己会了这个能力”，而是“模型知道什么时候应该调用外部能力来完成任务”。

### Tool Use 解决什么问题

它主要解决：

- 模型本身看不到外部数据
- 模型不能直接执行动作
- 模型不能只靠脑补完成任务

### 工作里怎么理解

比如要排查一个 bug：

- 只靠模型推理不够
- 它得去读日志、读代码、查文档

这时候就需要 Tool Use。

## 5. 四者分别在解决什么问题

可以把它们放到四个角度来看：

### Prompt：解决任务表达问题

核心问题是：

- 我要模型做什么
- 输出应该怎么组织

### Context：解决当前信息充分性问题

核心问题是：

- 模型这次推理到底看到了哪些信息

### Memory：解决跨轮记忆问题

核心问题是：

- 系统能不能延续过去的状态和偏好

### Tool Use：解决外部能力接入问题

核心问题是：

- 模型能不能读文件、查数据、调工具

## 6. 四者最容易混淆的地方

### 1. 把 Prompt 和 Context 混为一谈

Prompt 是你给的任务指令。  
Context 是模型实际看到的全部信息。

所以：

- Prompt 是 Context 的一部分
- Context 不等于 Prompt

### 2. 把 Context 和 Memory 混为一谈

Context 更像“眼前看到的内容”。  
Memory 更像“过去记住的内容”。

Memory 里的信息，只有在被拿出来参与当前推理时，才会进入 Context。

### 3. 把 Tool Use 当成模型本身能力

模型能调用工具，不代表模型天生会这些事。

更准确地说，是系统允许它：

- 知道工具存在
- 知道什么时候该调
- 知道怎么把结果拿回来继续用

## 7. 四者怎么一起工作

现实里，这四者通常是一起协作的。

一个简化流程可以这样看：

1. 用户提出问题
2. 系统用 Prompt 规定任务目标和输出格式
3. 系统把当前问题、资料、历史信息拼成 Context
4. 如果有长期状态，就从 Memory 里拿出相关内容
5. 如果信息不够，就触发 Tool Use 去查外部资源
6. 再把工具结果放回 Context，继续让模型生成结果

所以一个更完整的系统，不是只靠某个点，而是靠四者协同。

## 8. 工作中的真实例子

### 例子 1：代码排查助手

如果用户说：

```text
帮我定位这个保存失败的问题
```

系统内部可能是这样：

- Prompt：要求按“问题、原因、修复建议”输出
- Context：当前问题描述、日志、代码片段、接口文档
- Memory：记住用户前面提过“这是 Vue3 项目，不引第三方库”
- Tool Use：去读日志、查文件、搜关键字

缺一个都可能让结果变差。

### 例子 2：企业知识助手

如果员工问：

```text
报销流程是什么
```

系统内部可能是这样：

- Prompt：要求基于公司制度文档回答，并附来源
- Context：用户当前问题 + 检索到的制度片段
- Memory：记住这个用户来自哪个部门、偏好回答风格
- Tool Use：去查知识库、搜索相关制度

## 9. 为什么只靠 Prompt 不够

这是很多人刚接触 AI 系统时最容易误判的地方。

只写一段 Prompt，确实能让模型做一些基础任务。  
但只要任务一复杂，很快就会暴露问题：

- 没有外部资料
- 没有历史状态
- 没有工具能力
- 当前上下文不完整

所以真正稳定的 AI 系统，通常都不是“只写一段 Prompt”这么简单。

## 10. 面试怎么说

如果面试官问你“Prompt、Context、Memory、Tool Use 分别是什么”，可以这样回答：

```text
我会把它们理解成 AI 系统运行时的四个关键层面。
Prompt 负责定义任务目标和输出要求；
Context 负责提供当前推理所需的信息；
Memory 负责保存和延续历史状态；
Tool Use 负责让模型接触外部工具和数据。
四者结合起来，系统才能从一次性问答，走向更稳定的多轮任务执行。
```

这个回答的重点是：

- 讲清四者分别解决什么问题
- 讲清它们不是同一个层次
- 讲清四者组合后的价值

## 11. 复习时建议重点自测

- Prompt 和 Context 的区别是什么
- Context 和 Memory 的区别是什么
- 为什么 Tool Use 不是模型本身的“天生能力”
- 为什么复杂任务只靠 Prompt 往往不够
- 如果要做一个代码排查助手，这四者分别会发挥什么作用
