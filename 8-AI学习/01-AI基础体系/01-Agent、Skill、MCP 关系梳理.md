# 01-Agent、Skill、MCP 关系梳理

## 先说结论

如果只记一句话，可以先记这个：

- Agent 负责“为了目标做事”
- Skill 负责“把某类事情做得更稳定、更可复用”
- MCP 负责“把外部工具、资源和上下文以统一方式接进来”

这三者不是并列替代关系，而是经常一起工作：

- Agent 是任务组织者
- Skill 是能力封装单元
- MCP 是外部连接层

## 1. Agent 是什么

Agent 可以理解为一种“面向目标完成任务”的 AI 形态。

它和普通问答型 AI 的区别在于，普通问答更像“你问我答”，而 Agent 更像“收到目标后，自己组织步骤、调用能力、逐步完成任务”。

一个 Agent 通常会具备这些特征：

- 有明确目标
- 会分解任务
- 会根据上下文做判断
- 可能会调用工具
- 可能会多轮执行，而不是一次性吐出答案

例如：

- 帮你分析一个代码仓库并给出修改建议
- 帮你整理会议纪要并生成后续待办
- 帮你读取知识库后回答带来源的问题

这些都比单纯“解释一个概念”更接近 Agent 的工作方式。

## 2. Skill 是什么

Skill 可以理解为对某一类任务处理能力的封装。

它的重点不是“有多聪明”，而是“能不能稳定复用”。  
如果某类任务你反复都要做，而且做法、约束、输出格式都比较固定，就很适合沉淀成 Skill。

例如：

- 代码评审 Skill
- 面试题生成 Skill
- 周报整理 Skill
- Bug 分析 Skill

Skill 通常会把这些内容固定下来：

- 任务目标
- 适用场景
- 执行步骤
- 输出格式
- 风险提示

所以 Skill 不只是“一段 Prompt”，而更像“带方法和边界的任务模板”。

## 3. MCP 是什么

MCP 可以理解为模型和外部世界之间的一层标准化连接方式。

它解决的问题不是“怎么提问”，而是“模型怎么稳定访问外部资源和工具”。

这些外部能力可能包括：

- 文件系统
- 数据库
- 文档资源
- 浏览器
- API 服务
- 内部工具平台

如果没有统一连接方式，每接一个工具都要单独适配，成本会比较高。  
MCP 的价值就在于：让模型或 Agent 能以更统一的方式理解“我现在能访问什么资源、能调用什么工具、这些工具怎么用”。

## 4. 三者分别解决什么问题

可以把它们放到三个不同层面来看：

### Agent：解决任务组织问题

核心问题是：

- 目标是什么
- 现在该做哪一步
- 要不要调用工具
- 调用后下一步怎么继续

### Skill：解决能力复用问题

核心问题是：

- 某一类任务能不能沉淀成稳定做法
- 能不能减少重复描述
- 能不能让输出更一致

### MCP：解决外部连接问题

核心问题是：

- 模型如何访问文件、数据、接口和工具
- 多种资源如何统一暴露给模型
- 怎么降低接入和协作成本

## 5. 三者是什么关系

一个比较好记的理解方式是：

- Agent 像“负责人”
- Skill 像“成熟工作手册”
- MCP 像“对接外部系统的标准接口”

它们的关系通常是：

1. Agent 收到目标
2. Agent 判断是否需要调用某个 Skill
3. Agent 或 Skill 在执行过程中，通过 MCP 访问外部工具和资源
4. Agent 汇总结果，继续后续步骤或输出最终结果

也就是说：

- Agent 决定“做什么、怎么推进”
- Skill 帮助“某一步做得更标准”
- MCP 负责“把外部能力接进来”

## 6. 一个具体例子

比如有一个“代码仓库分析 Agent”，目标是：

```text
阅读项目代码，找出登录模块的问题，并给出修复建议。
```

这个过程中：

### Agent 在做什么

- 理解总目标
- 决定先看目录结构，再定位登录模块
- 判断需要读取文件、搜索关键字、汇总风险
- 决定最后输出问题清单和修改建议

### Skill 在做什么

如果系统里有“代码审查 Skill”或“Bug 分析 Skill”，它可以帮助：

- 用固定步骤检查代码
- 按统一格式输出问题
- 自动关注边界情况、异常处理、命名和测试缺口

这会让分析过程更稳定，而不是每次都从零开始描述要求。

### MCP 在做什么

Agent 或 Skill 如果要真正读取仓库内容，就需要访问文件、搜索命令或其他资源。

这时 MCP 的作用就是：

- 暴露文件系统资源
- 暴露搜索工具
- 暴露项目文档或其他外部信息源

这样 Agent 才不只是“空想”，而是真能接触到外部上下文。

## 7. 为什么容易混淆

很多人刚接触这几个概念时，容易出现这几种混淆：

### 1. 把 Agent 和 Skill 混为一谈

原因是两者都看起来像“AI 能力”。

但它们关注点不同：

- Agent 关注任务推进
- Skill 关注能力复用

一个系统可以没有 Skill 也有 Agent，但有了 Skill 后，Agent 往往更容易稳定工作。

### 2. 把 MCP 当插件市场

MCP 当然可以承载工具接入，但它不只是“列一个可用插件清单”。

更关键的是：

- 统一协议
- 统一访问方式
- 统一资源暴露方式

所以它更像基础连接层，而不是单纯的功能目录。

### 3. 把 Skill 当 Prompt 模板

Prompt 模板只是 Skill 的一部分。

真正有价值的 Skill，通常还会包含：

- 适用边界
- 执行步骤
- 输出格式
- 风险提醒

所以 Skill 比单段 Prompt 更完整。

## 8. 工作里怎么理解更实用

如果你是从工作落地角度来理解，可以这样记：

- 需要“帮我完成一个多步骤任务”时，重点看 Agent
- 需要“把某类常见任务做成稳定套路”时，重点看 Skill
- 需要“把 AI 连接到文件、接口、知识库、工具”时，重点看 MCP

这三个概念放在一起时，你会更容易看懂现在很多 AI 工具为什么长成这样。

## 9. 面试怎么说

如果面试官问你“怎么理解 Agent、Skill、MCP 的关系”，可以这样回答：

```text
我会把它们看成三个不同层次的能力。
Agent 更像任务执行者，负责围绕目标组织步骤和调用能力；
Skill 更像对某类任务的经验封装，用来提升复用性和稳定性；
MCP 更像模型连接外部工具和资源的标准化方式。
三者结合起来，才能让 AI 不只是回答问题，而是更稳定地接触外部信息并完成任务。
```

这个回答的关键点是：

- 讲清职责分工
- 讲清不是替代关系
- 讲清组合后的价值

## 10. 复习时建议重点自测

- Agent 和普通问答型 AI 的核心区别是什么
- Skill 为什么不等于一段 Prompt
- MCP 解决的核心问题是什么
- Agent、Skill、MCP 三者分别位于哪一层
- 如果让你举一个工作例子，你能不能讲清三者如何协作
