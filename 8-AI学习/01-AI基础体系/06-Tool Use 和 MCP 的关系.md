# 06-Tool Use 和 MCP 的关系

## 先说结论

如果只记一句话，可以先记这个：

- Tool Use 解决“模型怎么调用外部能力”
- MCP 解决“这些外部能力怎么被标准化接进系统”

也就是说：

- Tool Use 更偏“调用行为”
- MCP 更偏“连接方式”

两者密切相关，但不是同一个层级的东西。

很多人会把它们混在一起，是因为在真实系统里，模型常常正是通过 MCP 暴露出来的工具去完成 Tool Use。

## 1. Tool Use 是什么

Tool Use 可以理解为模型使用外部工具或能力的过程。

这些工具可能包括：

- 文件读取
- 搜索
- 浏览器操作
- 数据库查询
- 调用 API
- 执行代码

它的关键不在“工具叫什么”，而在：

- 模型知道什么时候该调工具
- 模型知道该调哪个工具
- 调完以后知道怎么利用结果继续完成任务

### Tool Use 解决什么问题

它主要解决：

- 模型本身接触不到外部世界
- 模型不能只靠记忆完成任务
- 模型需要真实数据、真实资源、真实执行结果

所以 Tool Use 是“把模型从纯文本推理，扩展到可接触外部能力”的关键一步。

## 2. MCP 是什么

MCP 可以理解为一种标准化连接方式，用来把外部资源和工具能力暴露给模型或 Agent。

它的重点不是“模型是否要调用工具”，而是：

- 工具和资源以什么格式被描述
- 模型怎么发现有哪些能力可用
- 模型怎么统一访问这些能力

所以 MCP 更像一层连接协议或连接规范。

### MCP 解决什么问题

它主要解决：

- 工具接入不统一
- 不同资源访问方式太零散
- 多工具、多数据源场景下协作成本高

如果没有这种标准化层，每接一个外部能力都可能要单独适配一次。

## 3. 两者分别在什么层面

可以把它们放到两个层面看：

### Tool Use：能力使用层

核心问题是：

- 当前任务要不要调工具
- 调哪个工具
- 调完以后怎么继续

### MCP：能力接入层

核心问题是：

- 工具和资源怎么接进系统
- 系统怎么统一看见这些能力
- 模型或 Agent 怎么用一致方式访问它们

所以从层级上说：

- Tool Use 更靠近“执行动作”
- MCP 更靠近“底层接入”

## 4. 一个最直观的理解方式

你可以这样记：

- Tool Use 像“使用电器”
- MCP 像“统一插座标准”

如果只有电器，没有统一插座，接起来很麻烦。  
如果只有统一插座，没有人去真正使用电器，也发挥不出价值。

所以：

- MCP 负责把能力规范化接进来
- Tool Use 负责真正把这些能力用起来

## 5. 为什么它们经常一起出现

因为在很多现代 Agent 系统里，模型并不是直接乱调工具，而是：

1. 系统先通过某种标准方式暴露可用资源和工具
2. 模型或 Agent 知道“有哪些工具可以用”
3. 再在执行过程中进行 Tool Use

如果这些工具是通过 MCP 暴露出来的，那你就会同时看到：

- MCP 在底层提供工具和资源
- Tool Use 在上层使用这些工具和资源

所以二者经常一起出现，但它们的职责还是不同。

## 6. 一个具体例子

比如你做一个“代码仓库分析助手”。

这个系统里可能需要这些能力：

- 读文件
- 搜索关键字
- 查项目文档

### 从 Tool Use 角度看

模型会在某一步判断：

- 我现在需要读 `auth.js`
- 我现在需要搜索 `login`
- 我现在需要查看 `README`

这就是 Tool Use。

### 从 MCP 角度看

系统会先把这些能力以统一方式暴露出来：

- 有哪些资源可读
- 有哪些工具可调用
- 每个工具需要什么参数

这就是 MCP 在做的事。

所以：

- MCP 负责“把门打开”
- Tool Use 负责“真的走进去做事”

## 7. 和 Agent、Skill 的关系

这两个概念还经常和 Agent、Skill 一起出现。

### 和 Agent 的关系

Agent 通常负责判断：

- 什么时候该调工具
- 先调哪个工具
- 调完以后下一步怎么安排

所以 Agent 常常会触发 Tool Use。

### 和 Skill 的关系

Skill 通常会把某一类任务的工具调用方式固定下来。

比如一个 `Bug 分析 Skill` 可能会稳定地做这些事：

- 先读日志
- 再读源码
- 再读接口文档

这里的“读日志、读源码、读文档”本质上就是 Tool Use。  
而这些能力如果是通过 MCP 暴露出来的，那 Skill 其实是在“使用 MCP 提供的工具和资源”。

## 8. 最容易混淆的地方

### 1. 把 MCP 直接理解成工具本身

不完全对。

MCP 更像“把工具和资源标准化接进来”的方式，而不是工具本身。

### 2. 把 Tool Use 当成一种协议

也不对。

Tool Use 更像行为方式，不是协议定义。

### 3. 觉得有 MCP 就天然有 Tool Use

不一定。

系统即使接入了很多能力，如果模型或 Agent 不去调用，它们也只是“可用但未使用”。

### 4. 觉得 Tool Use 不需要标准化接入层

在简单场景下也许还能硬接，但一旦工具多了、资源多了，没有统一方式会非常乱。

## 9. 工作中的真实例子

### 例子 1：研发助手

一个研发助手想帮你排查 bug：

- Tool Use：读取日志、搜索代码、查看接口文档
- MCP：把日志资源、文件资源、搜索工具统一暴露出来

所以你看到的效果是：

- 上层模型在调工具
- 底层系统在通过 MCP 提供这些工具

### 例子 2：企业知识助手

一个企业知识助手要回答制度问题：

- Tool Use：检索知识库、读取相关文档
- MCP：把知识库资源、检索接口标准化接入系统

这样系统才能稳定地让模型去查知识，而不是每个知识源都单独写一套接法。

## 10. 面试怎么说

如果面试官问你“Tool Use 和 MCP 有什么关系”，可以这样回答：

```text
我会把它们看成两个不同层级的概念。
Tool Use 更偏模型对外部能力的实际调用行为，解决的是模型什么时候调工具、调什么工具、怎么利用工具结果继续完成任务；
MCP 更偏外部能力的标准化接入方式，解决的是工具和资源怎么统一暴露给模型或 Agent。
所以两者经常一起出现，但不是一个东西。可以简单理解成，MCP 负责把能力接进来，Tool Use 负责把这些能力真正用起来。
```

这个回答的重点是：

- 讲清一个偏“使用”，一个偏“接入”
- 讲清它们的层级不同
- 讲清它们为什么经常一起出现

## 11. 复习时建议重点自测

- Tool Use 和 MCP 分别解决什么问题
- 为什么 Tool Use 更像行为，而 MCP 更像连接方式
- 为什么它们在真实 Agent 系统里经常一起出现
- 如果一个系统能读文件和搜索代码，这里面哪些属于 Tool Use，哪些属于 MCP
- 如果让你一句话区分它们，你会怎么说
