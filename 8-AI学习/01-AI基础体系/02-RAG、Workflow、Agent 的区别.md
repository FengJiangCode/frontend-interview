# 02-RAG、Workflow、Agent 的区别

## 先说结论

如果只记一句话，可以先记这个：

- RAG 解决“知识不在模型里”的问题
- Workflow 解决“步骤明确、稳定重复执行”的问题
- Agent 解决“目标明确，但路径不固定，需要判断和调整”的问题

这三者不是互斥关系，而是经常组合出现：

- Agent 可以调用 RAG
- Agent 可以运行 Workflow
- Workflow 里也可以嵌入 RAG

很多人之所以会混淆，是因为它们都常出现在 AI 应用里，但它们解决的问题层次并不一样。

## 1. RAG 是什么

RAG 可以理解为“先找资料，再生成答案”。

它的核心思路不是只靠模型脑子里原本记住的知识，而是先从外部知识库里检索相关内容，再把检索到的内容放进上下文里，让模型基于这些信息回答。

所以 RAG 的重点是：

- 外部知识
- 检索
- 把检索结果注入上下文
- 让答案更贴近真实资料

工作里常见场景：

- 企业知识库问答
- 产品文档助手
- 面向内部制度、FAQ、操作手册的问答系统

### RAG 解决什么问题

它主要解决：

- 模型不知道最新知识
- 模型不知道企业内部私有知识
- 希望回答尽量基于已有资料，而不是纯生成

### RAG 不擅长什么

它不天然擅长：

- 复杂多步决策
- 长流程任务编排
- 需要大量动态工具调用的任务执行

所以 RAG 很重要，但它不是所有 AI 系统的终点。

## 2. Workflow 是什么

Workflow 可以理解为“把一套固定步骤编排好，让系统按步骤执行”。

它强调的是：

- 先做什么
- 再做什么
- 失败了怎么办
- 每一步输入输出是什么

Workflow 更像一条预先设计好的流水线，而不是临场自由发挥。

工作里常见场景：

- 先抽取内容，再分类，再总结，再输出结果
- 先读工单，再匹配模板，再生成回复建议
- 先收集日志，再清洗数据，再出报告

### Workflow 解决什么问题

它主要解决：

- 步骤本来就比较清晰
- 执行逻辑希望稳定
- 希望结果可控、可重复、可观测

### Workflow 不擅长什么

它不太适合：

- 路径不固定的问题
- 需要边做边判断下一步的问题
- 很依赖动态决策的开放任务

如果任务本身变化很大，硬写成固定 Workflow，反而会变得笨重。

## 3. Agent 是什么

Agent 可以理解为“围绕目标自己组织步骤的系统”。

它和 Workflow 的最大区别在于，Workflow 往往是你提前把步骤写好，而 Agent 更强调：

- 收到目标后，自己决定先做什么
- 根据中间结果调整后续动作
- 按需要调用工具、读取资源、重新规划

所以 Agent 更像“目标驱动”，Workflow 更像“流程驱动”。

工作里常见场景：

- Bug 排查助手
- 代码仓库分析助手
- 帮你完成多轮信息搜集、判断和总结的任务型助手

### Agent 解决什么问题

它主要解决：

- 目标明确，但路径不固定
- 任务需要多轮判断
- 需要动态调用工具或资源
- 需要根据中间结果调整计划

### Agent 的代价

Agent 的好处是灵活，但代价也明显：

- 复杂度更高
- 可控性更弱
- 调试和评估更难
- 如果设计不好，容易显得“很聪明但不稳定”

## 4. 三者分别在解决什么问题

可以把它们放到三个不同视角看：

### RAG：解决知识来源问题

核心问题是：

- 模型回答所需的知识从哪里来
- 如何把外部资料放进上下文

### Workflow：解决步骤编排问题

核心问题是：

- 任务应该按什么固定顺序执行
- 每一步该接收什么输入、产出什么输出

### Agent：解决动态决策问题

核心问题是：

- 下一步到底该做什么
- 什么时候该换策略
- 要不要调用工具、读资源、重试或继续拆解

## 5. 三者的关键区别

一个比较好记的理解方式是：

- RAG 更像“找资料”
- Workflow 更像“跑流程”
- Agent 更像“做决策”

再具体一点：

### 1. 对知识的依赖方式不同

- RAG 强依赖外部知识检索
- Workflow 不一定关心知识检索，它更关心步骤编排
- Agent 可能会用知识检索，但不等于它本身就是 RAG

### 2. 对流程的确定性不同

- Workflow 的流程通常是预先写好的
- Agent 的流程通常是运行中动态决定的
- RAG 本身不是流程系统，它只是提供知识增强能力

### 3. 对灵活性的要求不同

- Workflow 追求稳定和可控
- Agent 追求灵活和适应性
- RAG 追求知识更准、更贴资料

## 6. 三者可以怎么组合

现实里很少只用一个，常见组合其实很多。

### Agent + RAG

Agent 先判断要查什么，再调用检索能力，把相关资料拉进上下文，然后继续决策或输出。

适合：

- 知识库问答增强
- 研发助手查文档后再给建议
- 需要边查资料边完成任务的系统

### Workflow + RAG

在固定流程里插入检索步骤。

例如：

1. 接收用户问题
2. 从知识库检索资料
3. 生成答案
4. 输出引用来源

这种场景不一定需要 Agent，因为路径已经很明确。

### Agent + Workflow

Agent 负责决定要不要走某条流程，而具体某个子任务内部，仍然可以用固定 Workflow 完成。

例如：

- Agent 判断当前是“接口报错问题”还是“缓存问题”
- 如果是接口报错，就走一套固定排障 Workflow
- 如果是缓存问题，就走另一套固定排障 Workflow

### Agent + Workflow + RAG

这是很多真实系统会出现的组合。

例如一个企业内部研发助手：

- Agent 负责理解问题并决定路线
- Workflow 负责执行固定步骤
- RAG 负责从文档和知识库里拿资料

## 7. 工作中的真实例子

### 例子 1：知识库问答助手

如果你只是想做一个“问制度、问文档、问 FAQ”的助手，很多时候：

- RAG 就是核心
- 外层可以加简单 Workflow
- 不一定需要 Agent

因为这种任务通常是：

- 先检索
- 再回答
- 再附带来源

路径比较稳定。

### 例子 2：Bug 排查或研发助手

如果目标是“帮我排查这个线上问题”，情况就更复杂：

- 它可能需要先看日志
- 再看代码
- 再看接口文档
- 中间发现新线索后还要改变下一步

这时候更像 Agent 场景。

而在这个 Agent 里：

- 读取知识库文档时可能会用到 RAG
- 某些固定分析步骤又可以沉淀成 Workflow

所以真实系统往往不是三选一，而是组合使用。

## 8. 什么时候不该用 Agent

这是很重要的一点。

很多人一做 AI 应用，就想直接上 Agent，但其实下面这些情况，往往没必要：

- 任务步骤已经非常明确
- 你只需要“查资料后回答”
- 结果要求很稳定、很可控
- 你不希望系统自由决策太多

这时候：

- 只用 RAG 就够了
- 或者 `RAG + Workflow` 就够了

如果强行上 Agent，常见问题会是：

- 复杂度上升
- 成本更高
- 结果更难预测
- 调试更麻烦

所以不是 Agent 越多越高级，而是要看任务本身需不需要动态决策。

## 9. 常见误区

### 1. 把 RAG 当完整系统

RAG 很重要，但它主要解决知识注入问题，不等于任务执行系统本身。

### 2. 把所有多步骤系统都叫 Agent

很多系统其实只是固定 Workflow，不一定具备真正的动态决策能力。

### 3. 觉得 Agent 一定比 Workflow 高级

不一定。  
如果场景本来就稳定、明确，Workflow 反而更合适。

### 4. 觉得有了 Agent 就不需要 RAG

也不对。  
Agent 仍然可能需要外部知识来源，而 RAG 正是常见方案之一。

## 10. 面试怎么说

如果面试官问你“RAG、Workflow、Agent 有什么区别”，可以这样回答：

```text
我会把它们理解成三种不同层次的问题。
RAG 主要解决知识来源问题，也就是模型不知道的外部知识怎么引入上下文；
Workflow 主要解决固定步骤编排问题，适合流程明确、稳定重复执行的场景；
Agent 主要解决动态决策问题，适合目标明确但路径不固定、需要边做边判断的任务。
它们不是互斥关系，真实系统里经常会组合使用，比如 Agent 调用 RAG 获取知识，或者 Agent 在内部运行某个固定 Workflow。
```

这个回答的重点是：

- 讲清三者分别解决什么问题
- 讲清它们不是互斥关系
- 讲清真实系统里常常是组合使用

## 11. 复习时建议重点自测

- RAG 解决的核心问题是什么
- Workflow 和 Agent 的最大区别是什么
- 为什么很多知识库问答系统不一定需要 Agent
- 为什么 Agent 里经常还会用到 RAG 或 Workflow
- 什么场景下不该上 Agent
