# 01-Bug排查助手协作Demo

## Demo 目标

这个 demo 改成了更贴近真实使用场景的一版，不再只是“概念介绍卡片”。

它模拟的是一条更真实的协作链路：

- 用户提出线上 Bug 排查需求
- Agent 接收目标并负责调度
- LLM 负责判断下一步、组织总结
- Skill 负责把 Bug 分析方法固化下来
- MCP 负责向上层暴露日志、源码、接口文档等资源

这里的 `LLM` 和 `MCP` 都是 mock，但 mock 的方式尽量贴近真实调用职责，而不是只写一句说明文字。

## 你能从这个 demo 学到什么

- Agent 为什么不是“大模型本身”
- LLM 在 Agent 系统里常见承担什么职责
- Skill 为什么比 Prompt 更像“稳定任务能力”
- MCP 为什么是外部资源连接层，而不是简单插件列表
- 一次真实感更强的任务里，三者是怎么串起来的

## 页面结构

- 用户任务：展示真实排查请求
- 执行时间线：展示整个协作链路
- Mock LLM 调用：展示 prompt 和返回结果
- Mock Skill 输出：展示 Skill 的结构化分析
- Mock MCP 调用：展示资源列表、资源读取和搜索行为
- 当前资源内容：展示被读取的日志、源码和接口文档
- 最终结论：展示定位、原因、证据、修复建议

## 运行方式

使用任意静态服务打开当前目录，例如：

```bash
python3 -m http.server 8000
```

然后访问：

```text
http://localhost:8000/8-AI学习/03-AI-Demo/01-Bug排查助手协作Demo/
```

## 建议你怎么学

1. 先点击“开始排查”，完整看一遍调用链
2. 再点“重置”，改用“单步执行”观察每一步是谁在工作
3. 重点对比：
   - Agent 在决定什么
   - LLM 在生成什么
   - Skill 在稳定什么
   - MCP 在提供什么
4. 再去改 `app.js` 里的资源内容或字段名，看最终结论怎么变化

## 最关键的理解

这个 demo 想帮你建立一个更稳的认知：

- Agent 负责流程和任务推进
- LLM 负责推理、判断、总结
- Skill 负责把某类工作做成稳定套路
- MCP 负责把外部资源接进来

如果少了其中某一层，系统通常会变得不稳定、不透明，或者无法接触真实上下文。
