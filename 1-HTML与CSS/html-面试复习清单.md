# HTML 面试复习清单

## 使用方式

- 先看问题，尝试自己回答
- 再对照简答版和展开版检查表达是否完整
- 最后补常见追问和容易答偏的地方

### 1. 什么是 HTML 语义化？为什么要语义化？

**简答版答案**

HTML 语义化就是根据内容的实际含义选择合适的标签，而不是一律用 `div` 和 `span`。
这样做主要有三个价值：代码结构更清晰、利于 SEO、提升可访问性。

**展开版答案**

HTML 语义化的核心是“让标签本身表达内容角色和页面结构”。
比如头部区域可以用 `header`，导航可以用 `nav`，主要内容可以用 `main`，独立文章内容可以用 `article`，侧边信息可以用 `aside`，页脚可以用 `footer`。

这样做的好处不只是“代码看起来更规范”，而是会直接影响多个方面：

- 对开发者来说，页面结构更清楚，维护时更容易定位模块
- 对搜索引擎来说，更容易识别主体内容和页面层级
- 对屏幕阅读器等辅助工具来说，更容易理解页面区域和内容组织

所以语义化本质上是在增强页面结构的表达能力，而不是单纯换几个标签名。

**关键点**

- 语义化的本质是标签和内容含义匹配
- 价值不只在 SEO，还包括可访问性和可维护性
- 语义化标签主要体现结构含义，不等于样式差异

**常见追问**

- `section` 和 `div` 有什么区别？
- `article` 和 `section` 有什么区别？
- 如果都能实现效果，为什么不全用 `div`？

**追问答案**

- `section` 强调有主题、有分区语义，通常会配合标题使用；`div` 只是一个无语义的通用容器。
- `article` 更强调内容本身是独立完整、可单独分发的，比如文章、帖子、评论；`section` 更像页面中的一个主题分区。
- 全用 `div` 虽然能实现样式，但会损失结构表达能力，不利于可维护性、可访问性和 SEO。

**容易答偏的地方**

- 只回答“有利于 SEO”，忽略可访问性
- 把语义化理解成“样式上的区别”
- 只会背标签名，不会结合页面结构说明

### 2. `src` 和 `href` 有什么区别？

**简答版答案**

`src` 是把资源引入到当前文档里，资源会成为当前内容的一部分；`href` 是建立当前文档和外部资源之间的关联。

**展开版答案**

`src` 常见于 `script`、`img`、`iframe` 这类标签，它表示“把这个资源加载进来并参与当前页面内容”。
例如 `img` 的图片会显示在页面里，`script` 会下载并执行脚本，`iframe` 会嵌入另一个页面。

`href` 常见于 `a` 和 `link`：

- `a` 用来建立跳转链接
- `link` 用来建立当前文档和外部资源之间的关系，比如引入样式表

所以可以简单理解为：

- `src` 更像“把资源拿进来用”
- `href` 更像“建立引用或关联关系”

**关键点**

- `src` 会引入实际资源内容
- `href` 更偏引用关系或跳转关系
- 常见场景要会对应起来：`img/script` 用 `src`，`a/link` 用 `href`

**常见追问**

- 为什么 `link` 引入 CSS 用的是 `href`？
- `script src` 会不会阻塞解析？
- `img src` 加载失败会有什么表现？

**追问答案**

- `link` 的职责是建立当前文档和外部资源的关系，所以用 `href` 更符合“关联资源”的语义。
- 默认情况下 `script src` 会阻塞 HTML 解析；如果加上 `defer` 或 `async`，下载和执行时机会有所不同。
- `img src` 加载失败时图片不会正常展示，常见表现是出现破图图标，同时可以通过 `alt` 提供替代文本。

**容易答偏的地方**

- 只背“一个引用一个引入”，说不出具体场景
- 混淆 `link` 和 `script` 的加载行为
- 不知道 `src` 往往会触发资源下载和使用

### 3. `defer` 和 `async` 有什么区别？

**简答版答案**

两者都是为了解决脚本加载阻塞的问题。
`async` 是下载完成后立即执行，执行顺序不保证；`defer` 是等 HTML 解析完成后再按顺序执行。

**展开版答案**

默认情况下，浏览器解析 HTML 遇到 `script` 会暂停 HTML 解析，先下载并执行脚本，再继续往下解析，这会影响页面首屏表现。

给脚本加 `async` 或 `defer` 后，脚本可以和 HTML 解析并行下载，但执行时机不同：

- `async`
  - 下载完成后立即执行
  - 执行时会打断 HTML 解析
  - 多个 `async` 脚本执行顺序不确定
  - 适合独立脚本，比如统计脚本、埋点脚本
- `defer`
  - 也是并行下载
  - 但会等 HTML 解析完成后再执行
  - 多个 `defer` 脚本会按书写顺序执行
  - 更适合业务脚本或有依赖关系的脚本

所以面试里可以一句话概括：

- `async` 更强调“谁先下载完谁先执行”
- `defer` 更强调“等 DOM 解析完，再按顺序执行”

**关键点**

- 二者都能减少同步脚本的阻塞影响
- `async` 不保证顺序
- `defer` 保证顺序，并且在 HTML 解析完成后执行

**常见追问**

- `defer` 和 `DOMContentLoaded` 的关系是什么？
- 多个 `async` 为什么不能保证顺序？
- 业务代码更适合 `async` 还是 `defer`？

**追问答案**

- 一般情况下，`defer` 脚本会在 HTML 解析完成后、`DOMContentLoaded` 触发前执行。
- 多个 `async` 谁先下载完谁先执行，所以顺序取决于网络和资源情况，不由书写顺序保证。
- 业务代码通常更适合 `defer`，因为它更稳定，能保证顺序，也更适合依赖 DOM 或多个脚本之间有先后关系的场景。

**容易答偏的地方**

- 只说“都是异步加载”
- 没说清执行时机和顺序差异
- 把“下载时机”和“执行时机”混在一起

### 4. `meta viewport` 的作用是什么？

**简答版答案**

`meta viewport` 主要用于控制移动端页面的视口宽度和缩放行为，让页面能按预期进行响应式展示。

**展开版答案**

移动端浏览器默认会把页面放进一个虚拟布局视口里，如果不显式声明，页面往往会按桌面宽度缩放显示，导致布局和字体看起来很小。

常见写法是：

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0" />
```

这里最常用的两个配置是：

- `width=device-width`：让布局视口宽度等于设备宽度
- `initial-scale=1.0`：设置初始缩放比例为 1

它的意义在于让移动端页面的布局基准更符合设备宽度，这也是响应式布局的基础之一。

**关键点**

- 主要面向移动端适配
- 影响的是视口宽度和缩放行为
- 响应式布局通常依赖正确的 viewport 设置

**常见追问**

- 如果不写 `meta viewport` 会怎样？
- `initial-scale` 的作用是什么？
- 它和 `rem`、媒体查询是什么关系？

**追问答案**

- 不写的话，移动端浏览器往往会按较宽的默认布局视口渲染页面，导致页面整体被缩小，看起来字小、布局不对。
- `initial-scale` 用来指定页面初始缩放比例，常见设置是 `1.0`，表示按正常比例展示。
- `meta viewport` 决定的是移动端布局基准，`rem` 和媒体查询是在这个基础上继续做尺寸适配和响应式设计。

**容易答偏的地方**

- 只说“适配手机”但说不清具体作用
- 把视口和 CSS 单位问题混为一谈
- 不知道它本质上在控制移动端布局基准

### 5. 块级元素、行内元素、行内块元素有什么区别？

**简答版答案**

块级元素通常独占一行，可以设置宽高；行内元素通常不会独占一行，宽高默认由内容撑开；行内块元素既可以和其他元素同行排列，也可以设置宽高。

**展开版答案**

这三个概念本质上是在描述元素在文档流中的表现方式。

- 块级元素：
  - 默认独占一行
  - 宽度默认撑满父容器
  - 一般可以直接设置宽高
- 行内元素：
  - 不独占一行
  - 宽高更多由内容决定
  - 直接设置宽高通常不生效
- 行内块元素：
  - 不独占一行
  - 但可以设置宽高
  - 常用于既想并排显示又需要尺寸控制的场景

面试里通常还会延伸到：

- 哪些元素默认是块级，哪些默认是行内
- 为什么有些行内元素设置宽高不生效
- `display` 改变后表现会怎样变化

**关键点**

- 重点不是背分类，而是理解它们的布局表现
- 是否独占一行、能否设置宽高是核心区别
- 默认行为可以通过 `display` 改变

**常见追问**

- 常见块级元素和行内元素分别有哪些？
- 为什么 `span` 设置宽高不生效？
- `inline-block` 有什么使用场景？

**追问答案**

- 常见块级元素有 `div`、`p`、`h1`~`h6`、`ul`、`li`；常见行内元素有 `span`、`a`、`strong`、`em`。
- 因为 `span` 默认是行内元素，行内元素的宽高通常不按块级盒模型那样直接生效。
- `inline-block` 适合既想让元素并排显示，又希望可以设置宽高的场景，比如导航项、按钮组、标签项。

**容易答偏的地方**

- 只会背元素名单，不会总结规律
- 误以为“行内元素绝对不能设置宽高”
- 忽略 `display` 可以改变默认表现

### 6. `label` 有什么作用？为什么推荐和表单控件绑定？

**简答版答案**

`label` 用来描述表单控件。它和输入框绑定后，点击文字也能聚焦或选中对应控件，能提升可用性和可访问性。

**展开版答案**

`label` 的核心作用是给表单控件提供明确说明。
它有两种常见绑定方式：

1. 通过 `for` 对应控件的 `id`
2. 直接把表单控件包在 `label` 里面

绑定之后，点击 `label` 的文字区域，浏览器会自动把焦点交给对应输入框，或者切换单选框、复选框的选中状态。

它的价值主要体现在两个方面：

- 提升用户体验：点击范围更大，不需要只点很小的输入区域
- 提升可访问性：辅助设备更容易理解控件的用途

所以 `label` 不是可有可无的说明文字，而是表单语义的一部分。

**关键点**

- `label` 是表单可用性和可访问性的关键标签
- 点击 `label` 可以联动控件
- 常用于 `input`、`checkbox`、`radio`

**常见追问**

- `for` 和 `id` 的绑定方式怎么写？
- 为什么单选框和复选框更需要 `label`？
- `placeholder` 能不能代替 `label`？

**追问答案**

- 写法是 `label` 的 `for` 属性值等于表单控件的 `id`，比如 `<label for="username">用户名</label>` 对应 `<input id="username" />`。
- 因为单选框和复选框本身可点击区域很小，绑定 `label` 后点击文字也能操作，体验会明显更好。
- `placeholder` 不能完全代替 `label`，因为它更多是提示示例，不是稳定的控件说明，而且可访问性也不如 `label`。

**容易答偏的地方**

- 只说“用来显示标题”
- 不知道它和可访问性的关系
- 误以为 `placeholder` 可以完全代替 `label`

### 7. `link` 和 `@import` 有什么区别？

**简答版答案**

`link` 是 HTML 标签，用来在页面加载时引入外部资源；`@import` 是 CSS 语法，用来在样式文件里再引入其他样式。通常更推荐 `link`。

**展开版答案**

`link` 写在 HTML 里，浏览器在解析 HTML 时就能发现并尽早发起样式资源请求。
而 `@import` 写在 CSS 里，往往需要先下载并解析当前 CSS 后，浏览器才知道还要继续加载其他样式资源。

因此两者的主要区别包括：

- 使用位置不同：
  - `link` 属于 HTML
  - `@import` 属于 CSS
- 发现资源的时机不同：
  - `link` 更早被浏览器发现
  - `@import` 往往更晚触发
- 使用建议不同：
  - 页面主样式通常推荐 `link`
  - `@import` 更适合少量样式组织或特殊场景

面试里一般会进一步追问性能差异和实际使用建议。

**关键点**

- 一个是 HTML 标签，一个是 CSS 语法
- `link` 通常更有利于页面样式尽早加载
- 实际项目里一般优先推荐 `link`

**常见追问**

- 为什么说 `@import` 的性能通常不如 `link`？
- `link` 除了引入 CSS 还能做什么？
- 现代工程化项目里为什么很少直接手写 `@import`？

**追问答案**

- 因为 `@import` 往往要等当前 CSS 被发现并解析后，浏览器才知道还要继续请求其他样式资源，发现时机更晚。
- `link` 还可以用来声明图标、预加载资源、建立预连接等，它不只用于样式表。
- 现代工程化里样式依赖通常由构建工具统一处理，所以手写 `@import` 的场景少了，而且主样式入口也更推荐直接用 `link`。

**容易答偏的地方**

- 只说“语法不同”而不提加载时机
- 把它们理解成完全等价
- 不知道实际项目为什么更常用 `link`

### 8. attribute 和 property 有什么区别？

**简答版答案**

attribute 是写在 HTML 标签上的属性，property 是 DOM 对象上的属性。attribute 更偏初始声明，property 更偏运行时状态。

**展开版答案**

当浏览器解析 HTML 后，会把标签转换成 DOM 对象。
这时你在标签上写的属性，比如：

```html
<input value="hello" />
```

其中 `value="hello"` 是 HTML attribute。

但在 JS 运行时，通过 DOM 节点访问到的 `input.value` 是 property。
两者有联系，但并不完全等价。

比较常见的理解方式是：

- attribute：定义在 HTML 上，偏向初始值或结构声明
- property：定义在 DOM 对象上，偏向当前运行时状态

典型例子就是表单值：

- 初始 HTML 里写的 `value` 是 attribute
- 用户输入后，变化的通常是 DOM property

**关键点**

- attribute 属于 HTML 标记层
- property 属于 DOM 对象层
- 二者有关联，但运行时不一定始终同步

**常见追问**

- `input.value` 和 `getAttribute("value")` 有什么区别？
- 哪些场景下它们会不一致？
- `checked` 为什么也是常见考点？

**追问答案**

- `input.value` 读到的是当前输入框的实时值，`getAttribute("value")` 读到的更接近 HTML 上声明的初始值。
- 用户手动输入后，通常 DOM property 会变化，但原始 attribute 不一定同步变化，这时两者就可能不一致。
- `checked` 也是典型例子，因为复选框或单选框的“当前选中状态”属于运行时状态，特别适合拿来区分 attribute 和 property。

**容易答偏的地方**

- 只说“一个是属性一个也是属性”，没区分层级
- 不知道运行时状态和初始声明的区别
- 没举表单场景做说明

### 9. 什么是 HTML 可访问性？前端为什么要关注？

**简答版答案**

HTML 可访问性是指页面内容和交互能被更多用户正常理解和使用，包括依赖键盘、屏幕阅读器等辅助工具的用户。前端关注它，不只是为了规范，也是为了让产品真正可用。

**展开版答案**

可访问性可以理解为“让不同能力、不同使用方式的用户都能顺畅访问页面”。
在 HTML 层面，最基础的做法通常包括：

- 用正确的语义化标签表达结构
- 给图片补充有意义的 `alt`
- 给表单控件配好 `label`
- 需要按钮行为时用 `button`，需要跳转时用 `a`
- 在必要时使用 `aria-*` 补充辅助语义

前端要关注可访问性，原因主要有三个：

- 产品层面：让更多用户能正常使用
- 工程层面：语义更清晰，结构更规范
- 合规层面：部分业务场景会有无障碍要求

所以可访问性不是额外加分项，很多时候它和正确的 HTML 写法本来就是同一件事。

**关键点**

- 可访问性和语义化强相关
- `alt`、`label`、正确使用 `button` / `a` 都是基础能力
- 很多无障碍问题本质上是 HTML 结构用错了

**常见追问**

- `button` 和 `a` 应该怎么选？
- `alt` 应该怎么写才算合理？
- `aria-*` 是不是越多越好？

**追问答案**

- 需要触发页面行为、提交、弹窗、切换状态时优先用 `button`；需要跳转到另一个地址时用 `a`。
- `alt` 应该描述图片在当前上下文中的实际意义，而不是机械重复文件内容；如果图片纯装饰，可以考虑空 `alt`。
- 不是越多越好，优先应该先用正确的原生语义标签；只有原生语义不够表达时，再补充合适的 `aria-*`。

**容易答偏的地方**

- 把可访问性理解成“只给残障人士用”
- 只会背 `aria`，但忽略基础语义标签
- 不知道很多问题优先应该用原生标签解决

### 10. HTML 对 SEO 有哪些帮助？

**简答版答案**

HTML 对 SEO 的帮助主要体现在结构清晰、语义明确、标题和描述完整、内容可被搜索引擎更好理解。它不是 SEO 的全部，但属于非常基础的一层。

**展开版答案**

搜索引擎在理解页面时，不只是看文字内容，也会看页面结构和语义表达。
HTML 在 SEO 中的价值主要包括：

- 使用语义化标签，让页面主体结构更清晰
- 合理使用标题层级，比如 `h1` 到 `h6`
- 提供准确的 `title` 和 `meta description`
- 给图片补充合理的 `alt`
- 保证重要内容能在页面结构中被正常识别

从前端角度看，HTML 对 SEO 的帮助不等于“只要写了语义化标签就能有排名”，而是说它为搜索引擎理解内容提供了更好的基础条件。

**关键点**

- SEO 不是只看关键词，也看页面结构和语义
- `title`、标题层级、语义标签、`alt` 都属于基础项
- HTML 是 SEO 的基础层，不是全部

**常见追问**

- `title` 和 `meta description` 分别有什么作用？
- 为什么标题层级不能乱用？
- 单页应用为什么常被拿来讨论 SEO？

**追问答案**

- `title` 是搜索结果和浏览器标签上最核心的标题信息，`meta description` 更像页面摘要，都会影响搜索结果的展示表达。
- 标题层级体现内容结构，如果层级混乱，搜索引擎和辅助工具都会更难理解页面主次关系。
- 因为单页应用很多内容依赖 JS 运行后才渲染，如果首屏 HTML 内容过少，搜索引擎理解页面内容的效率和效果可能会受影响。

**容易答偏的地方**

- 把 SEO 完全等同于语义化
- 只会说“有利于搜索引擎抓取”，不会展开
- 不知道 `title`、标题层级、图片 `alt` 这些基础点
