# CSS 面试复习清单

## 使用方式

- 先看问题，尝试自己回答
- 再对照简答版和展开版检查表达是否完整
- 最后补常见追问和容易答偏的地方

### 1. 什么是标准盒模型？什么是 IE 盒模型？

**简答版答案**

标准盒模型里，`width` 和 `height` 只表示内容区大小；IE 盒模型也就是常见的 `border-box` 模式里，`width` 和 `height` 包含内容、内边距和边框。

**展开版答案**

CSS 盒模型描述的是一个元素在页面上实际占据空间的组成方式，通常包括：

- content
- padding
- border
- margin

在标准盒模型下，给元素设置的 `width: 200px`，只代表内容区宽度是 `200px`，元素真正占据的总宽度还要再加上左右 `padding` 和 `border`。

而在 IE 盒模型，也就是现在更常用的 `box-sizing: border-box` 模式下，设置的 `width: 200px` 表示整个盒子可见区域宽度就是 `200px`，内容区会自动扣除 `padding` 和 `border`。

所以两者最核心的区别在于：`width` 和 `height` 到底算内容区，还是算整个盒子。

**关键点**

- 盒模型的核心是理解内容区、内边距、边框、外边距的关系
- 标准盒模型下 `width` 不包含 `padding` 和 `border`
- `border-box` 更适合做实际页面布局

**常见追问**

- `box-sizing: border-box` 有什么作用？
- 元素最终宽高是怎么计算出来的？
- `width: 100%` 和实际占据空间为什么有时不一样？

**追问答案**

- `box-sizing: border-box` 会让 `width` 和 `height` 包含 `padding` 和 `border`，布局时更容易控制总尺寸。
- 标准盒模型下，总宽度通常是 `margin + border + padding + width`；`border-box` 下 `width` 已经包含了 `padding` 和 `border`。
- 因为 `width: 100%` 说的是内容区或盒子宽度基于父容器计算，但元素最终占据空间还可能受 `padding`、`border`、`margin` 影响。

**容易答偏的地方**

- 只会背“标准盒模型和 IE 盒模型”这两个名字
- 说不清 `width` 到底作用在哪一层
- 把 `margin` 也错误地算进 `box-sizing`

### 2. CSS 选择器优先级是怎么计算的？

**简答版答案**

CSS 优先级通常按内联样式、`id`、`class/属性/伪类`、标签/伪元素的顺序比较；优先级高的覆盖优先级低的，同级再看后写的规则。

**展开版答案**

样式冲突时，浏览器不会随机决定，而是按优先级和书写顺序计算最终结果。

通常可以这样理解优先级：

- 内联样式优先级最高
- `id` 选择器高于 `class`
- `class`、属性选择器、伪类属于同一层级
- 标签选择器和伪元素优先级更低

如果优先级完全相同，就看谁写在后面，后面的覆盖前面的。

另外 `!important` 会进一步提升声明权重，但它不适合滥用，因为会破坏样式覆盖的自然层次。

所以这个问题的重点不只是会背顺序，更要知道“为什么样式没生效”通常就是优先级或层叠顺序出了问题。

**关键点**

- 优先级比较是排查样式不生效的基础
- 同级优先级下，后写的规则会覆盖前面的规则
- `!important` 能提高权重，但不是推荐的常规解法

**常见追问**

- `!important` 的优先级表现是什么？
- 伪类和伪元素有什么区别？
- 哪些 CSS 属性会继承，哪些不会继承？

**追问答案**

- `!important` 会提升当前声明的优先级，但如果双方都用了 `!important`，还是要继续比较选择器优先级和书写顺序。
- 伪类描述元素状态，比如 `:hover`、`:focus`；伪元素描述元素的一部分，比如 `::before`、`::after`。
- 像 `color`、`font-size`、`line-height` 这类文本相关属性常常会继承，而盒模型和布局属性通常不会自动继承。

**容易答偏的地方**

- 只记一个模糊顺序，不知道同级还要比书写顺序
- 把伪类和伪元素混在一起
- 遇到样式冲突只会加 `!important`

### 3. 什么是 BFC？哪些情况会触发 BFC？

**简答版答案**

BFC 可以理解成一个独立的块级格式化上下文，里面的布局计算相对独立，不会随便和外部元素互相影响。常见触发方式包括 `overflow` 不为 `visible`、`display: flow-root`、浮动元素、绝对定位元素等。

**展开版答案**

BFC 的重点不是定义本身，而是它能解决什么布局问题。

你可以把 BFC 理解为一个相对封闭的布局环境，在这个环境里：

- 内部元素的布局尽量在自己的上下文里完成
- 外部浮动和内部布局的相互影响会减少
- 一些常见问题，比如高度塌陷、文字环绕、margin 重叠，可以通过 BFC 来控制

常见触发方式包括：

- `overflow: hidden/auto/scroll`
- `display: flow-root`
- 浮动元素
- 绝对定位元素
- `inline-block`

面试里通常不会只问“定义”，而是更关心两个点：

- 你知不知道哪些情况会形成 BFC
- 你知不知道它能解决什么实际问题

**关键点**

- BFC 是一个独立的块级格式化上下文
- 重点是理解它的隔离作用，而不是死记定义
- 常用于解决浮动影响、margin 重叠和高度塌陷问题

**常见追问**

- BFC 能解决哪些实际问题？
- 为什么会发生 margin 重叠？
- 父元素为什么会出现高度塌陷？

**追问答案**

- BFC 常见用途是包裹内部浮动元素、防止外部浮动影响自身布局、避免相邻块级元素垂直 margin 合并。
- margin 重叠通常发生在普通文档流中的块级元素垂直方向，相邻元素或父子元素的上下 margin 可能会合并。
- 当子元素都浮动后，它们会脱离普通文档流，父元素就可能感知不到它们的高度，从而表现出高度塌陷。

**容易答偏的地方**

- 只会背英文全称
- 说不清 BFC 解决的实际问题
- 把所有布局问题都归结成“加个 BFC 就行”

### 4. 浮动的本质是什么？它会带来什么影响？

**简答版答案**

浮动本质上是让元素脱离普通文档流的一部分，并尽量向左或向右排列。它早期常用于布局，但会带来父元素高度塌陷、周围内容环绕等副作用。

**展开版答案**

浮动最初的设计更偏向“图文环绕”场景，而不是今天常见的复杂页面布局。

元素设置 `float` 后，会出现几个典型变化：

- 元素会尽量向左或向右靠
- 后续普通文档流内容可能围绕它排布
- 它不再像普通块级元素那样参与父元素的高度计算

这也是为什么浮动常常会引出两个经典问题：

- 父元素高度塌陷
- 相邻内容位置异常或被环绕

以前做两栏、三栏布局会经常用浮动，但现在很多布局场景更适合用 Flex 或 Grid，因为它们语义更直接、控制也更稳定。

**关键点**

- 浮动本来不是为现代主布局而设计的
- 浮动元素会影响周围内容排布
- 浮动常和清除浮动一起出现

**常见追问**

- 浮动为什么会导致父元素高度塌陷？
- 清除浮动有哪些常见方式？
- 为什么很多场景下现在更推荐 Flex，而不是浮动布局？

**追问答案**

- 因为浮动元素脱离了普通文档流，父元素按普通流计算高度时可能感知不到它们。
- 常见方式有给父元素触发 BFC、使用 clearfix、添加清除浮动元素，或者直接改用 `display: flow-root`。
- Flex 是为一维布局设计的，表达更直接，也没有浮动那种需要额外处理高度塌陷和环绕问题的副作用。

**容易答偏的地方**

- 只记得“浮动能让元素左对齐右对齐”
- 不知道为什么要清除浮动
- 把浮动和定位混为一谈

### 5. `flex: 1` 到底是什么意思？

**简答版答案**

`flex: 1` 通常表示元素可以分配剩余空间，并且基础尺寸为 `0` 或接近按比例分配的效果。更完整地理解要结合 `flex-grow`、`flex-shrink`、`flex-basis` 一起看。

**展开版答案**

Flex 布局的核心是主轴空间分配。

`flex` 是一个缩写属性，通常对应：

- `flex-grow`
- `flex-shrink`
- `flex-basis`

很多场景下写 `flex: 1`，可以理解成“这个元素愿意去分剩余空间，并和同级元素按比例分配”。

如果多个子项都写 `flex: 1`，它们通常会平均分剩余空间；如果一个写 `flex: 2`，另一个写 `flex: 1`，前者通常会拿到更多的剩余空间。

Flex 最适合处理一维布局，比如横向排列一组元素、做自适应两栏、按钮组、导航栏、居中布局等。

面试里问 `flex: 1`，本质上是在看你是否真正理解 Flex 的空间分配模型，而不是只会背属性名。

**关键点**

- Flex 是一维布局模型
- `flex: 1` 不是魔法，要结合 grow、shrink、basis 理解
- Flex 很适合处理剩余空间分配和对齐问题

**常见追问**

- Flex 布局的主轴和交叉轴是什么？
- `justify-content` 和 `align-items` 分别控制什么？
- 什么场景适合用 Flex？

**追问答案**

- 主轴由 `flex-direction` 决定，默认横向；交叉轴和主轴垂直。
- `justify-content` 控制主轴上的排列方式，`align-items` 控制交叉轴上的对齐方式。
- 一维排布、按钮组、导航栏、自适应两栏、列表项对齐、水平垂直居中都很适合用 Flex。

**容易答偏的地方**

- 只会说“`flex: 1` 就是平分”
- 不知道它本质上和剩余空间分配有关
- 把主轴和交叉轴概念说反

### 6. Grid 和 Flex 的区别是什么？

**简答版答案**

Flex 更适合一维布局，主要处理一行或一列上的排布；Grid 更适合二维布局，可以同时控制行和列。

**展开版答案**

Flex 和 Grid 都是现代布局方案，但它们的关注点不同。

Flex 的核心是：

- 处理一个方向上的对齐和分配
- 更适合一维布局
- 常用于横向列表、纵向排列、居中、弹性分配空间

Grid 的核心是：

- 同时处理行和列
- 更适合二维布局
- 常用于页面区域划分、卡片网格、复杂布局模板

比如：

- 一组按钮水平排布，用 Flex 更自然
- 一个后台页面要同时定义头部、侧边栏、主内容区和底部，用 Grid 会更直观

所以不是谁替代谁，而是它们适合解决的问题维度不同。

**关键点**

- Flex 偏一维，Grid 偏二维
- Grid 对复杂区域布局更友好
- 两者可以在同一个页面里配合使用

**常见追问**

- 什么场景更适合 Grid？
- `grid-template-columns` 和 `grid-template-rows` 怎么理解？
- `fr` 单位是什么？

**追问答案**

- 需要明确控制行列关系、网格区域和整体版式时，更适合用 Grid。
- `grid-template-columns` 定义列轨道，`grid-template-rows` 定义行轨道，本质上是在声明网格结构。
- `fr` 可以理解成可用剩余空间的比例单位，适合做自适应网格分配。

**容易答偏的地方**

- 只会说“Grid 比 Flex 更高级”
- 不知道一维和二维的真正区别
- 误以为页面里只能二选一

### 7. `absolute`、`fixed`、`sticky` 有什么区别？

**简答版答案**

`absolute` 是相对包含块定位，通常脱离普通文档流；`fixed` 通常相对视口定位；`sticky` 则是在相对定位和固定定位之间切换，滚动到阈值后才表现出吸附效果。

**展开版答案**

定位相关问题的核心不是背五个单词，而是理解它们如何参与文档流、相对于谁定位、在滚动时怎么表现。

常见几个定位方式可以这样理解：

- `relative`
  - 仍在普通文档流中
  - 相对自身原本位置偏移
- `absolute`
  - 脱离普通文档流
  - 相对最近的定位祖先元素或包含块定位
- `fixed`
  - 通常脱离普通文档流
  - 默认相对视口定位
- `sticky`
  - 开始像相对定位
  - 滚动到指定阈值后像固定定位一样“吸附”

面试里经常会把这类问题和“包含块”一起问，因为真正难点在于“它到底相对于谁定位”。

**关键点**

- 定位问题本质上在问文档流和参考坐标系
- `absolute` 关注包含块
- `sticky` 生效依赖滚动容器和阈值设置

**常见追问**

- 绝对定位元素相对于谁定位？
- 什么是包含块？
- `sticky` 为什么有时不生效？

**追问答案**

- `absolute` 通常相对于最近的已定位祖先元素定位；如果没有，就继续往上找，直到更高层容器。
- 包含块可以理解为当前定位和尺寸计算的参考区域，不同定位方式下包含块的确定规则不同。
- 常见原因是没有设置 `top` / `left` 等阈值、父容器没有可滚动空间、或者祖先元素的布局环境影响了它的吸附行为。

**容易答偏的地方**

- 只会背“absolute 脱标、fixed 固定”
- 说不清“相对于谁定位”
- 不知道 `sticky` 生效需要条件

### 8. `z-index` 为什么有时不生效？

**简答版答案**

`z-index` 不生效，很多时候不是值不够大，而是元素不在正确的层叠上下文里，或者本身没有进入可比较层级的定位环境。

**展开版答案**

很多人遇到遮挡问题时第一反应是把 `z-index` 调大，但这并不总能解决问题。

原因是层叠顺序不仅由 `z-index` 决定，还和层叠上下文有关。

你可以把层叠上下文理解为一个独立的“叠放世界”：

- 同一个层叠上下文里的元素可以直接比较 `z-index`
- 不同层叠上下文之间先比较上下文整体，再比较内部元素

所以如果一个元素在更低层的上下文里，即使它内部 `z-index` 很大，也不一定能盖住另一个更高层上下文里的元素。

此外，`z-index` 往往要结合定位元素或其他会创建层叠上下文的属性一起看，不能单独理解成“数字越大越在上面”。

**关键点**

- `z-index` 的问题经常是层叠上下文问题
- 数值大不一定能跨上下文覆盖
- 遮挡问题通常要结合定位、层叠上下文和 DOM 结构一起看

**常见追问**

- 什么是层叠上下文？
- 哪些属性会创建层叠上下文？
- 为什么把 `z-index` 调得很大也不一定盖在最上面？

**追问答案**

- 层叠上下文可以理解成元素叠放时的独立比较环境，内部元素先在自己的上下文中比较，再和外部上下文整体比较。
- 常见触发包括定位元素配合 `z-index`、`opacity` 小于 `1`、`transform`、某些滤镜或混合模式等。
- 因为它可能被限制在一个更低层的上下文里，无法直接和另一个更高层上下文中的元素比较。

**容易答偏的地方**

- 只会说“把 `z-index` 调大”
- 不理解层叠上下文的隔离性
- 遇到遮挡问题不去检查父级结构

### 9. 响应式布局通常怎么做？

**简答版答案**

响应式布局通常是结合 `meta viewport`、弹性布局、相对单位和媒体查询，让页面能在不同屏幕尺寸下保持可用和可读。

**展开版答案**

响应式布局的目标不是“在手机上缩小页面”，而是让页面根据设备尺寸主动调整结构和尺寸。

常见手段包括：

- 正确设置 `meta viewport`
- 使用媒体查询在不同断点下调整布局
- 使用弹性布局，比如 Flex 和 Grid
- 使用相对单位，比如 `rem`、`vw`、百分比
- 让图片和容器具备自适应能力

常见单位理解可以这样记：

- `px`：固定像素单位
- `em`：相对当前元素字体大小
- `rem`：相对根元素字体大小
- `vw` / `vh`：相对视口宽高

面试里经常还会问 `rem` 方案和 `vw` 方案的差异，本质是在考你是否理解“尺寸基准”。

**关键点**

- 响应式布局是“按条件调整”，不是简单缩放
- 媒体查询、弹性布局、相对单位常常一起使用
- `viewport` 是移动端适配的基础前提

**常见追问**

- 媒体查询的常见用法是什么？
- `px`、`em`、`rem`、`vw`、`vh` 有什么区别？
- 如何理解 `rem` 方案和 `vw` 方案的区别？

**追问答案**

- 媒体查询通常按断点切换布局、字号、间距或显示策略，比如在手机和桌面下使用不同列数。
- `px` 更固定，`em` 和 `rem` 是相对字体大小的单位，`vw` 和 `vh` 是相对视口尺寸的单位。
- `rem` 更依赖根字体大小作为统一缩放基准，`vw` 更直接基于视口宽度变化，两者都能做适配，但设计思路不同。

**容易答偏的地方**

- 只会说“用媒体查询”
- 不知道 `viewport` 在移动端适配中的作用
- 把各种单位的参考基准说混

### 10. 回流和重绘有什么区别？

**简答版答案**

回流是布局相关信息发生变化，浏览器需要重新计算几何位置和尺寸；重绘是外观变化但布局不变，浏览器只需要重新绘制。通常回流成本更高。

**展开版答案**

这个问题本质上是在考你对浏览器渲染代价的理解。

简单来说：

- 回流：元素的位置、尺寸、布局关系变了，浏览器需要重新做布局计算
- 重绘：布局没变，只是颜色、背景、可见性等外观变了，浏览器重新画一遍

典型容易触发回流的情况包括：

- 修改元素宽高
- 增删 DOM
- 改变字体大小
- 读取某些会强制刷新布局的信息后又立即写样式

而像修改颜色、背景色、阴影等，更多只是重绘。

为什么常说 `transform` 和 `opacity` 更适合做动画？
因为它们通常不会频繁触发布局变化，更容易得到更平滑的渲染表现。

**关键点**

- 回流比重绘代价更高
- 性能优化重点常在减少不必要的回流
- 动画和频繁样式变更要尽量避开高成本布局属性

**常见追问**

- 哪些操作容易触发回流？
- 哪些操作通常只会触发重绘？
- 为什么 `transform` 和 `opacity` 更适合做动画？

**追问答案**

- 修改几何属性、增删节点、改布局相关样式、读写布局信息交替执行，通常都容易触发回流。
- 修改颜色、背景、边框颜色等外观属性，很多情况下更偏向只触发重绘。
- 因为它们通常不直接改变文档流中的布局关系，更容易减少高成本的布局计算和页面抖动。

**容易答偏的地方**

- 只会背“回流比重绘更耗性能”
- 不知道具体哪些操作会触发哪一种
- 把所有样式修改都笼统地说成“重绘”

- 先把这 10 个 CSS 问题过熟，至少做到简答版能直接讲出来
- 再结合自己的项目经验补充布局、性能优化场景
- 如果需要继续沉淀，可以把 Flex、定位、层叠上下文、性能单独拆成专题文档

### 11. 隐藏与显示

**简答版答案**

`display: none` 会让元素直接脱离布局且不渲染；`visibility: hidden` 元素不可见但仍占位；`opacity: 0` 只是变透明，元素依然在布局中，很多情况下也仍能响应事件。

**展开版答案**

这几个属性看起来都像“隐藏元素”，但它们影响的层面并不一样。

可以从三个维度理解：

- 是否占据布局空间
- 是否还能响应交互
- 是否还能被辅助技术感知

具体来说：

- `display: none`
  - 元素不参与布局
  - 页面上相当于没有这个节点的可视盒子
  - 通常也不会响应点击，因为连渲染盒都没有
- `visibility: hidden`
  - 元素仍然占据原来的位置
  - 只是视觉上隐藏
  - 常用于“隐藏但不希望布局跳动”的场景
- `opacity: 0`
  - 元素完全透明，但还在文档流中
  - 仍然可能响应点击、获取焦点
  - 常用于做渐隐动画

而 `overflow: hidden` 和上面三者也不同，它不是“把元素隐藏掉”，而是把超出元素盒子范围的内容裁掉。也就是说，元素本身仍然存在，只是溢出部分不显示。

面试里如果只回答“一个占位一个不占位”还不够，最好顺带补一句：透明不代表不可交互，真正要看是否仍在渲染树和交互链路里。

**关键点**

- `display: none` 是直接移出布局
- `visibility: hidden` 是隐藏但保留占位
- `opacity: 0` 更像“透明”，不是彻底消失

**常见追问**

- 这三种方式对事件响应有什么影响？
- `opacity: 0` 的元素还能点击吗？
- 什么场景适合用 `display: none`，什么场景适合用 `visibility: hidden`？

**追问答案**

- `display: none` 通常不会响应事件；`visibility: hidden` 和 `opacity: 0` 要结合具体交互环境理解，其中 `opacity: 0` 的元素很多时候仍可点击。
- 可以，`opacity: 0` 只是视觉透明，元素本身仍然存在，必要时要配合 `pointer-events: none`。
- 如果希望元素彻底不参与布局和显示，适合用 `display: none`；如果希望隐藏时页面结构不抖动，适合用 `visibility: hidden`；如果需要渐隐渐现动画，常用 `opacity`。

**容易答偏的地方**

- 认为“看不见”就一定不能点击
- 把 `overflow: hidden` 理解成隐藏元素本身
- 只回答占位问题，忽略交互和动画场景

### 12. 居中方案

**简答版答案**

居中方案没有唯一答案，要看是水平居中、垂直居中，还是水平垂直同时居中，也要看元素宽高是否已知。现在最常用、最稳的方案通常是 Flex，其次是 Grid、定位加位移等。

**展开版答案**

“居中”其实是很多小问题的统称，不同场景适合的方案不同。

常见情况可以这样拆：

- 行内内容水平居中
  - 父元素用 `text-align: center`
- 块级元素水平居中
  - 已知宽度时常用 `margin: 0 auto`
- 单个元素水平垂直居中
  - 父元素用 Flex：`display: flex; justify-content: center; align-items: center;`
- 二维布局下整体居中
  - 可以用 Grid：`display: grid; place-items: center`
- 已知或未知尺寸的绝对定位居中
  - `position: absolute; left: 50%; top: 50%; transform: translate(-50%, -50%)`

为什么现在面试里更推荐 Flex？
因为它表达语义直接，不依赖奇技淫巧，对已知和未知尺寸都比较友好，也更适合真实业务里的响应式布局。

以前很多垂直居中方案会依赖：

- `line-height`
- 绝对定位加负边距
- `table-cell`

这些方案不是不能用，而是适用范围更窄。面试时最好体现出：你知道旧方案，但现代开发优先用 Flex 或 Grid。

**关键点**

- 居中方案要结合场景和元素类型说
- Flex 是最常见的现代居中方案
- 已知宽高和未知宽高，适合的方法不完全一样

**常见追问**

- `margin: 0 auto` 为什么能水平居中？
- 为什么 Flex 做居中更常用？
- 绝对定位加 `translate` 的百分比是相对于谁？

**追问答案**

- 因为块级元素在有确定宽度时，左右外边距自动分配剩余空间，从而达到水平居中。
- 因为它同时解决对齐和空间分配问题，对未知尺寸更友好，代码也更清晰。
- `left: 50%` 和 `top: 50%` 是相对包含块计算，`translate(-50%, -50%)` 里的百分比则是相对元素自身尺寸计算。

**容易答偏的地方**

- 一上来只说 Flex，不区分不同居中场景
- 说 `margin: 0 auto` 时忘了前提是块级元素且通常要有宽度
- 把定位百分比和 `translate` 百分比的参考对象混淆

### 13. 包含块与百分比计算

**简答版答案**

包含块可以理解成元素定位和尺寸计算的参考区域。`absolute` 元素通常相对于最近的已定位祖先元素计算；百分比宽度通常相对包含块宽度，百分比高度是否生效则取决于父元素高度是否明确。

**展开版答案**

包含块是 CSS 里很容易被忽略，但定位题和百分比题经常会考到的核心概念。

你可以把它理解成：浏览器在计算当前位置、宽高、偏移量时，所依赖的那块参考区域。

几个高频点要分开记：

- 普通元素的百分比宽度
  - 通常相对父元素内容区宽度计算
- 百分比高度
  - 只有父元素有明确高度时，子元素的百分比高度才更容易生效
- 绝对定位元素
  - 通常相对于最近的已定位祖先元素确定包含块
- `top`、`left`、`right`、`bottom`
  - 百分比偏移量通常相对包含块尺寸计算
- `transform: translate(-50%, -50%)`
  - 这里的百分比相对的是元素自身尺寸，不是父元素

为什么面试喜欢问“百分比高度为什么不生效”？
因为很多人只记得“百分比是相对父元素”，却忽略了高度的前提条件。父元素如果高度是 `auto`，子元素的百分比高度通常就没法按预期算出来。

所以这一类题最重要的是分清：到底是在问布局尺寸，还是在问定位偏移；它们虽然都写百分比，但参考对象不一定一样。

**关键点**

- 包含块是定位和尺寸计算的参考区域
- 百分比宽度和高度的计算前提不同
- `translate` 的百分比相对自身，不相对父元素

**常见追问**

- 为什么百分比高度有时不生效？
- `absolute` 元素到底相对于谁定位？
- `left: 50%` 和 `translateX(-50%)` 为什么经常一起用？

**追问答案**

- 因为父元素高度如果没有明确值，子元素的百分比高度往往缺少稳定的计算基准。
- 一般相对于最近的已定位祖先元素；如果没有这样的祖先，就继续往上找更高层参考区域。
- 因为前者把元素左边缘放到包含块中线位置，后者再让元素自身向左回退自己宽度的一半，两者配合后才是视觉上的真正居中。

**容易答偏的地方**

- 笼统地说“百分比都相对父元素”
- 不知道高度百分比为什么和宽度百分比不一样
- 把定位偏移百分比和 `transform` 百分比说成同一个参考系

### 14. 伪类、伪元素与状态样式

**简答版答案**

伪类描述的是元素的状态，伪元素描述的是元素的一部分或额外生成的内容。像 `:hover`、`:focus` 这类常用于交互状态，`::before`、`::after` 常用于装饰性内容。

**展开版答案**

这组概念很容易背混，最稳的记法是先分“状态”和“部分”。

伪类的重点是：元素本身还在那里，只是处于某种状态，比如：

- `:hover`
- `:focus`
- `:active`
- `:checked`
- `:disabled`

它是在描述“这个元素现在怎么样”。

而伪元素的重点是：它像是生成了一个可被样式控制的额外片段，比如：

- `::before`
- `::after`
- `::first-line`
- `::first-letter`

它是在描述“元素的哪一部分”或“额外加出来的一部分”。

`::before` 和 `::after` 很常见，但它们通常要配合 `content` 使用。因为如果没有 `content`，这个生成内容往往就不会真正出现，样式也很难表现出来。

状态样式里，面试也常问可访问性相关区别：

- `:hover` 更偏鼠标悬停
- `:focus` 表示元素获得焦点
- `:focus-visible` 更强调键盘导航等更适合显示焦点指示的场景
- `:active` 表示激活瞬间

所以如果你能补一句“不要只给 `hover` 不给 `focus`，否则键盘用户体验会差”，回答会更完整。

**关键点**

- 伪类是状态，伪元素是部分或生成内容
- `::before`、`::after` 本质上不是 DOM 真实节点
- 状态样式要考虑交互，也要考虑可访问性

**常见追问**

- `:focus` 和 `:focus-visible` 有什么区别？
- 为什么伪元素通常要写 `content: ''`？
- 表单状态相关伪类有哪些？

**追问答案**

- `:focus` 只要获得焦点就可能匹配，`:focus-visible` 更偏向在需要明显焦点提示的场景下才显示，常见于键盘导航体验优化。
- 因为 `::before` 和 `::after` 是生成内容，没有 `content` 时通常不会按预期渲染出来。
- 常见有 `:checked`、`:disabled`、`:enabled`、`:valid`、`:invalid`、`:required` 等，可用于表单状态反馈。

**容易答偏的地方**

- 把伪类和伪元素都说成“伪标签”
- 只知道 `hover`，忽略 `focus` 和可访问性
- 以为 `::before`、`::after` 是真实 DOM 节点

### 15. 文本与排版

**简答版答案**

文本与排版题常考省略、换行、行高、图片基线和字体回退。单行省略号一般结合 `overflow`、`white-space`、`text-overflow`；多行省略常见做法是 `-webkit-line-clamp`；而图片底部空隙通常和行内元素基线对齐有关。

**展开版答案**

这类题虽然碎，但在实际开发里非常常见。

先看省略号：

- 单行省略号常见写法
  - `overflow: hidden`
  - `white-space: nowrap`
  - `text-overflow: ellipsis`
- 多行省略号常见写法
  - `display: -webkit-box`
  - `-webkit-line-clamp`
  - `-webkit-box-orient: vertical`
  - 再配合 `overflow: hidden`

再看几个容易混的文本属性：

- `white-space`
  - 控制空白是否合并、文本是否换行
- `word-break`
  - 控制单词或连续字符在边界处如何断开
- `overflow-wrap`
  - 当单词太长放不下时，是否允许换行打断

`line-height` 之所以常和“垂直居中”联系在一起，是因为它不仅影响行盒高度，也会影响文字在行内的视觉位置。比如单行按钮文字，很多人会直接把 `line-height` 设成和容器高度一样，形成近似垂直居中的效果。

图片底部空白间隙也是面试高频点。原因通常不是图片本身多了 margin，而是 `img` 默认按行内元素参与排版，要为基线对齐预留空间。常见处理方式有：

- 把图片设为 `display: block`
- 或修改 `vertical-align`

字体回退则主要是给 `font-family` 配一组候选字体，让浏览器按顺序匹配，确保不同系统下有较稳定的显示效果。

**关键点**

- 省略号实现要区分单行和多行
- 文本换行相关属性各自关注点不同
- 图片底部空隙往往是基线对齐问题，不是图片“自带空白”

**常见追问**

- 多行省略号有哪些限制？
- `white-space`、`word-break`、`overflow-wrap` 分别解决什么问题？
- 图片底部空白怎么消除？

**追问答案**

- 多行省略常见方案有一定兼容性和使用限制，通常更适合固定行数的展示场景。
- `white-space` 更偏整体换行策略，`word-break` 更偏单词或字符断开规则，`overflow-wrap` 更偏内容太长放不下时是否允许换行。
- 可以把 `img` 改成 `display: block`，或者设置合适的 `vertical-align`，本质上是处理行内基线对齐带来的留白。

**容易答偏的地方**

- 只会写单行省略，不知道多行怎么做
- 把几个换行相关属性混成一个概念
- 误以为图片底部空白是浏览器 bug

### 16. 动画与过渡

**简答版答案**

`transition` 更适合描述状态变化时的过渡，`animation` 更适合定义一段完整关键帧动画。动画是否流畅，核心取决于修改了哪些属性，以及这些变化会不会频繁触发高成本的布局和绘制。

**展开版答案**

这类题表面上在问 CSS 属性，实际上是在问浏览器渲染性能。

先区分两个概念：

- `transition`
  - 更适合从一个状态平滑过渡到另一个状态
  - 通常需要有触发条件，比如 `:hover`、类名切换
- `animation`
  - 可以通过 `@keyframes` 定义多个阶段
  - 更适合循环、连续、复杂节奏的动画

哪些属性更适合做动画？
通常更推荐：

- `transform`
- `opacity`

因为它们通常不会直接改变文档流中的布局关系，更容易避免频繁回流。

而不太适合高频动画的属性通常包括：

- `width`
- `height`
- `top`
- `left`
- `margin`

这些属性更容易引发布局变化，元素一多或者动画一频繁，就更容易卡顿。

为什么动画会卡？
常见原因包括：

- 动画属性选得不好，频繁触发回流或重绘
- 同时动画的元素太多
- 主线程还有大量脚本计算
- 图片、阴影、滤镜等绘制成本高

所以面试里如果能把“动画性能”和“浏览器渲染流程”连起来讲，会显得更扎实。也就是：样式变化不是免费的，浏览器可能要重新计算样式、布局、绘制，甚至合成，动画越高频，对这条链路的压力越大。

**关键点**

- `transition` 偏状态过渡，`animation` 偏完整动画编排
- 动画性能取决于属性类型和渲染代价
- `transform` 和 `opacity` 通常是更优先的动画属性

**常见追问**

- `transition` 和 `animation` 分别适合什么场景？
- 为什么 `transform` 和 `opacity` 更适合做动画？
- 动画卡顿通常从哪些方向排查？

**追问答案**

- 简单 hover 过渡、展开收起、按钮反馈常用 `transition`；需要关键帧、循环、节奏控制的效果更适合 `animation`。
- 因为它们通常不直接影响文档流布局，更容易减少高成本的布局计算和页面抖动。
- 可以先看是否改了布局属性，再看动画元素数量、脚本开销、图片和阴影等绘制成本，最后再结合浏览器性能面板定位瓶颈。

**容易答偏的地方**

- 只会背“用 `transform` 更好”，但说不出为什么
- 把所有卡顿都归因于“浏览器性能不好”
- 只讲属性语法，不讲渲染代价和性能关系

### 17. 说说设备像素、CSS 像素

**简答版答案**

设备像素是屏幕物理上的最小显示单元，CSS 像素是浏览器里的逻辑像素单位。它们不一定一一对应，在高密度屏幕上，1 个 CSS 像素往往会对应多个设备像素。

**展开版答案**

这个问题常常和移动端适配、高清屏、`viewport`、`dpr` 一起出现。

先把两个概念分开：

- 设备像素
  - 也叫物理像素
  - 是屏幕硬件上的真实像素点
- CSS 像素
  - 是浏览器用于布局和样式计算的逻辑单位
  - 我们写 `width: 100px`，这里的 `px` 通常说的是 CSS 像素

为什么它们不一样？
因为浏览器并不是简单地把 1 个 CSS 像素永远映射成 1 个物理像素，而是会结合设备密度做换算。

这时候就会引出 `devicePixelRatio`，也就是常说的 `dpr`。可以粗略理解成：

- `dpr = 设备像素 / CSS 像素`

比如在 `dpr = 2` 的屏幕上，1 个 CSS 像素通常会映射到 2 × 2 个设备像素来显示。这样做的结果是：

- 页面布局尺寸仍然按 CSS 像素来写，开发体验一致
- 但屏幕会用更多物理像素来呈现内容，显示更细腻

这也是为什么同样写一个 `100px` 宽的盒子，在普通屏和高清屏上视觉大小可以差不多，但清晰度不一样。

面试里常见的进一步考察是：

- 为什么 1px 边框在移动端有时看起来偏粗？
- 为什么图片在高倍屏上会发虚？

本质上都和 CSS 像素和物理像素之间的映射有关。比如图片资源如果分辨率不够，在高 `dpr` 屏幕上被放大显示后就容易模糊。

**关键点**

- 设备像素是物理概念，CSS 像素是逻辑概念
- 页面布局通常基于 CSS 像素开发
- 高密度屏下 1 个 CSS 像素可能对应多个设备像素

**常见追问**

- 什么是 `dpr`？
- 为什么高倍屏下图片容易发虚？
- 移动端 1px 问题是什么？

**追问答案**

- `dpr` 是设备像素比，可以理解成物理像素和 CSS 像素之间的比值。
- 因为图片资源本身像素密度不够，在高密度屏幕上显示时需要更多物理像素去呈现，放大后就容易不够清晰。
- 所谓 1px 问题，通常是指我们写的 `1px` 是 1 个 CSS 像素，在高 `dpr` 设备上显示出来可能对应多个物理像素，所以视觉上会显得比预期更粗。

**容易答偏的地方**

- 把设备像素和 CSS 像素当成同一个概念
- 只会背 `dpr`，但说不清它和页面显示效果的关系
- 不知道为什么高清屏下会出现图片发虚和 1px 边框问题
