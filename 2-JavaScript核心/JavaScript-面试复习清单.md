# JavaScript 面试复习清单

## 使用方式

- 先看问题，尝试自己回答
- 再对照简答版和展开版检查表达是否完整
- 最后补常见追问和容易答偏的地方

### 1. JavaScript 有哪些数据类型？`null` 和 `undefined` 有什么区别？

**简答版答案**

JavaScript 的数据类型通常分为基本类型和引用类型。
基本类型常见有 `number`、`string`、`boolean`、`undefined`、`null`、`symbol`、`bigint`，引用类型常见有 `object`，像数组、函数、日期本质上都属于对象。
`undefined` 更强调“没有被赋值”，`null` 更强调“有意地表示为空”。

**展开版答案**

JavaScript 早期常被简单说成“基本类型和对象”，但在面试里最好说得更完整一些。

基本类型包括：

- `number`
- `string`
- `boolean`
- `undefined`
- `null`
- `symbol`
- `bigint`

引用类型的核心是 `object`，数组、函数、正则、日期等都可以看作对象的具体表现。

`undefined` 一般表示变量声明了但还没有赋值，或者函数没有显式返回值，默认返回 `undefined`。
`null` 则更像开发者主动给出的“空值”，表示这里原本应该有对象或内容，但现在明确是空的。

比如：

```js
let a
let b = null
```

这里 `a` 是“未赋值”，`b` 是“主动置空”。

面试里有时还会追问 `typeof null`，它返回的是 `object`，这是 JavaScript 历史遗留问题，不代表 `null` 真的是对象。

**关键点**

- 数据类型至少要答到 `symbol` 和 `bigint`
- 引用类型本质上可以归到 `object`
- `undefined` 偏“未赋值”，`null` 偏“主动为空”
- `typeof null === 'object'` 是历史遗留问题

**常见追问**

- `typeof` 分别能返回什么？
- 怎么判断一个值到底是不是数组？
- 基本类型和引用类型在赋值时有什么区别？

**追问答案**

- `typeof` 常见会返回 `number`、`string`、`boolean`、`undefined`、`symbol`、`bigint`、`function`、`object`。
- 判断数组更可靠的方式是 `Array.isArray(arr)`。
- 基本类型赋值是值拷贝，引用类型赋值拷贝的是地址，所以多个变量可能指向同一个对象。

**容易答偏的地方**

- 漏掉 `symbol` 和 `bigint`
- 把数组、函数单独说成“不是对象”
- 说不清 `null` 和 `undefined` 的语义差异

### 2. `==` 和 `===` 有什么区别？为什么更推荐 `===`？

**简答版答案**

`==` 会先做类型转换再比较，`===` 不会做隐式类型转换，要求类型和值都相同才返回 `true`。
面试里更推荐 `===`，因为它规则更明确，能减少隐式转换带来的歧义和 bug。

**展开版答案**

`==` 叫宽松相等，比较时如果两边类型不同，JavaScript 会先尝试做隐式类型转换，再判断结果是否相等。
比如：

```js
0 == false // true
'' == false // true
null == undefined // true
```

这些结果本身不一定错，但它们的可读性和可预测性都比较差。

`===` 叫严格相等，它不会帮你做额外类型转换，只有当类型相同且值相等时才返回 `true`。
所以：

```js
0 === false // false
```

在业务代码里更推荐使用 `===`，因为它的判断规则更稳定，排查问题也更直接。
只有在你明确知道自己要利用宽松相等规则时，才会刻意使用 `==`，比如少数场景会用 `value == null` 同时判断 `null` 和 `undefined`。

**关键点**

- `==` 会触发隐式类型转换
- `===` 更严格、更可预测
- 绝大多数业务代码优先用 `===`
- `null == undefined` 是一个常见特殊点

**常见追问**

- `NaN === NaN` 为什么是 `false`？
- 如何正确判断 `NaN`？
- 对象比较时 `===` 比较的是什么？

**追问答案**

- 因为 `NaN` 表示“不是一个有效数字结果”，它和任何值比较都不相等，包括自己。
- 可以用 `Number.isNaN(value)` 判断。
- 对象比较时比较的是引用地址，不是结构内容。

**容易答偏的地方**

- 只背“一个严格一个不严格”，没说清类型转换
- 不知道 `null == undefined`
- 把对象比较误解成深比较

### 3. 什么是隐式类型转换？常见规则和过程是什么？

**简答版答案**

隐式类型转换就是 JavaScript 在某些运算或判断场景下，自动把一个值转成另一种类型再继续执行。
高频场景主要有三类：比较、运算、布尔判断。难点不在“会不会转”，而在“在什么场景下按什么顺序转”。

**展开版答案**

JavaScript 是弱类型语言，不同类型的值在一起参与比较或运算时，解释器经常会自动帮你做转换，这就是隐式类型转换。

这个机制本身不是错误，它是语言设计的一部分。真正容易出问题的是：

- 开发者不知道什么时候会触发转换
- 不清楚不同场景下的转换方向不一样
- 把少数特殊规则和一般规则混在一起记

面试里最值得先掌握的是三类高频场景：

- 比较
- 运算
- 布尔判断

比较场景里最常见的是 `==`。
它在两边类型不同的时候通常会尝试先转换再比较，比如：

- 布尔值先转数字，`true -> 1`，`false -> 0`
- 字符串和数字比较时，字符串通常先转数字
- 对象和原始值比较时，对象会先转成原始值，再继续比较
- `null` 和 `undefined` 是特例，`null == undefined` 为 `true`

运算场景里要特别注意 `+` 和其他运算符不一样：

- `+` 既可能是数字相加，也可能是字符串拼接
- `-`、`*`、`/` 往往会把参与运算的值转成数字

例如：

```js
'1' + 2 // '12'
'1' - 2 // -1
+true // 1
```

这里 `+` 在字符串存在时更容易走拼接，而减法会把两边都转成数字。

布尔判断场景常见于 `if`、`while`、三元运算符和逻辑运算。
JavaScript 会把值转成布尔值后再判断真假。

常见假值有：

- `false`
- `0`
- `-0`
- `0n`
- `''`
- `null`
- `undefined`
- `NaN`

除了这些，绝大多数值都是真值，包括空数组 `[]` 和空对象 `{}`。

理解隐式转换，最重要的不是死记所有题，而是形成一条稳定思路：

1. 先看当前是什么场景，是比较、运算还是布尔判断
2. 再看当前规则会把值往哪个方向转
3. 如果出现对象，继续追问它会先转成什么原始值
4. 最后再得到比较或运算结果

可以看一个经典案例：

```js
[] == false // true
```

拆解过程如下：

1. 这是 `==` 比较，而且两边类型不同，所以会发生隐式转换
2. 右边的 `false` 先转成数字 `0`
3. 左边的 `[]` 是对象，和原始值比较时，会先转原始值
4. 空数组转原始值后通常得到空字符串 `''`
5. 空字符串和数字比较时，空字符串再转成数字 `0`
6. 最后就变成 `0 == 0`，结果是 `true`

再看一个更容易混乱的例子：

```js
[] == ![] // true
```

拆解时不要一下子想太多，按顺序看：

1. `![]` 先执行
2. 空数组是对象，对象在布尔判断里是真值
3. 所以 `![]` 等于 `false`
4. 原式就变成 `[] == false`
5. 后面的过程和上一个案例一样，最后结果是 `true`

这类题之所以容易让人混乱，不是因为它特别高级，而是它把“布尔转换”和“宽松比较转换”叠在了一起。

再做一个对照：

```js
[] === false // false
```

因为 `===` 不做隐式类型转换，左边是对象，右边是布尔值，类型不同，直接就是 `false`。

**关键点**

- 隐式类型转换的高频场景是比较、运算、布尔判断
- `==` 和 `===` 最大差别之一就是前者可能触发转换
- `+` 要特别注意字符串拼接，`-`、`*`、`/` 更偏向数字转换
- 空数组和空对象在布尔判断里都是真值
- `null` 和 `undefined` 是特殊关系，不能和一般规则混着背

**常见追问**

- 对象转原始值时大致会经历什么过程？
- 为什么 `[]` 是真值，但 `[] == false` 又是 `true`？
- `if ('0')` 和 `if (0)` 的结果为什么不同？

**追问答案**

- 可以先理解成对象在需要参与比较或运算时，会优先尝试变成原始值，再进入后续比较或计算。数组常见会先变成字符串，比如 `[] -> ''`、`[1,2] -> '1,2'`。
- 因为这是两个不同场景。布尔判断里 `[]` 作为对象是真值，但 `[] == false` 先经历的是宽松比较规则，不是直接做布尔判断。
- `'0'` 是非空字符串，布尔判断里是真值；`0` 是数字零，属于假值，所以结果不同。

**容易答偏的地方**

- 把所有隐式转换都归结成“自动转数字”
- 只会背几个怪题答案，不会按场景拆过程
- 误以为空数组和空对象在布尔判断里是假值
- 把 `==` 的规则和 `if` 判断的规则混在一起

### 4. 什么是作用域？什么是作用域链？

**简答版答案**

作用域就是变量和函数可访问的范围。JavaScript 主要有全局作用域、函数作用域和块级作用域。
作用域链是在当前作用域找不到变量时，沿着外层词法环境一层层向上查找，直到全局作用域为止。

**展开版答案**

作用域的核心是“一个标识符在哪里可以被访问到”。
JavaScript 中常见的作用域有：

- 全局作用域
- 函数作用域
- 块级作用域

早期 `var` 主要形成函数作用域，而 `let` 和 `const` 会形成块级作用域，所以在 `if`、`for`、`{}` 里声明时访问范围更小。

作用域链强调的是查找规则。
当代码执行到某个变量时，会先在当前作用域找：

- 当前作用域有，就直接用
- 当前作用域没有，就去外层作用域找
- 一直找到全局作用域
- 还找不到，就抛出 `ReferenceError`

这个查找过程依赖的是词法作用域，也就是作用域在代码书写阶段基本就决定了，而不是在调用位置临时改变。

比如：

```js
const a = 1

function outer() {
  const b = 2
  function inner() {
    console.log(a, b)
  }
  inner()
}
```

`inner` 在自身作用域里找不到 `a` 和 `b`，就会沿着外层作用域向上找。

**关键点**

- 作用域描述“能访问哪些变量”
- JavaScript 是词法作用域
- 作用域链描述变量逐层向外查找的过程
- `let`、`const` 和 `var` 的作用域表现不同

**常见追问**

- 什么是词法作用域？
- `var`、`let`、`const` 在作用域上有什么区别？
- 暂时性死区是什么？

**追问答案**

- 词法作用域指变量可访问范围在代码定义时就基本确定，不是由调用位置决定。
- `var` 是函数作用域，`let` 和 `const` 是块级作用域。
- 暂时性死区指 `let` 和 `const` 声明前的区域虽然已经进入作用域，但不能访问对应变量。

**容易答偏的地方**

- 把作用域和执行上下文混为一谈
- 只会背全局作用域、局部作用域，不会解释查找过程
- 说不清 JavaScript 是词法作用域

### 5. 什么是闭包？闭包的应用场景和问题分别是什么？

**简答版答案**

闭包可以理解为“函数和它定义时所在词法环境的组合”。
即使外层函数已经执行完，内部函数依然能访问外层作用域里的变量。
它常用于封装私有变量、函数工厂、缓存和事件处理，但也可能带来内存占用和变量滥用问题。

**展开版答案**

闭包本质上不是一种语法，而是一种运行现象。
当一个函数内部再返回或引用另一个函数，而内部函数又访问了外层函数的变量时，就形成了闭包。

例如：

```js
function createCounter() {
  let count = 0
  return function () {
    count++
    return count
  }
}

const counter = createCounter()
```

这里 `createCounter` 执行结束后，按常规理解它的局部变量应该被销毁，但返回的内部函数还在引用 `count`，所以这个变量会继续保留。

闭包常见应用场景有：

- 封装私有状态
- 函数柯里化或工厂函数
- 缓存计算结果
- 事件回调中保留上下文

它的问题主要不在“闭包本身有错”，而在于不合理使用会让原本应该释放的数据继续被引用，造成内存占用增加，或者让变量状态变得难以追踪。

**关键点**

- 闭包的本质是函数访问并保留外层词法环境
- 闭包是 JavaScript 很核心的能力，不是单纯面试概念
- 价值在于状态保留和封装
- 风险在于引用不该长期保留的数据

**常见追问**

- `for` 循环里闭包为什么容易出问题？
- 闭包一定会导致内存泄漏吗？
- 闭包和私有变量有什么关系？

**追问答案**

- 因为多个回调可能共享同一个变量，尤其是 `var` 声明时更容易出现最终都拿到同一结果的问题。
- 不一定。闭包只是让变量活得更久，只有在不再需要却仍被引用时，才可能造成内存泄漏风险。
- 闭包能让外部无法直接访问某些变量，只能通过暴露的方法间接操作，所以常被用来模拟私有变量。

**容易答偏的地方**

- 只会说“函数套函数就是闭包”
- 把闭包妖魔化成“性能一定很差”
- 说不出具体应用场景

### 6. `this` 的指向到底由什么决定？

**简答版答案**

`this` 不是在函数定义时决定的，而是在函数调用时决定的。
面试里通常按四条规则理解：默认绑定、隐式绑定、显式绑定、`new` 绑定，其中 `new` 绑定优先级最高，箭头函数是特殊情况，它没有自己的 `this`。

**展开版答案**

很多人会把 `this` 理解成“函数属于谁，`this` 就是谁”，这其实不准确。
更准确的说法是：普通函数的 `this` 取决于调用方式。

常见规则可以这样记：

- 默认绑定：普通函数直接调用，非严格模式下指向全局对象，严格模式下是 `undefined`
- 隐式绑定：通过对象调用，`this` 指向调用它的对象
- 显式绑定：通过 `call`、`apply`、`bind` 指定 `this`
- `new` 绑定：通过 `new` 调用构造函数时，`this` 指向新创建的实例对象

例如：

```js
function fn() {
  console.log(this)
}

const obj = { fn }
obj.fn()
```

这里 `fn` 是通过 `obj.fn()` 调用的，所以 `this` 指向 `obj`。

如果多个规则同时出现，通常优先级是：

- `new` 绑定
- 显式绑定
- 隐式绑定
- 默认绑定

箭头函数是特殊情况，它没有自己的 `this`，而是继承外层作用域的 `this`。

**关键点**

- `this` 看调用方式，不看定义位置
- 重点掌握四种绑定规则和优先级
- 严格模式下默认绑定是 `undefined`
- 箭头函数没有自己的 `this`

**常见追问**

- 定时器里的 `this` 为什么经常出问题？
- 事件回调里的 `this` 指向谁？
- `bind` 之后还能被 `new` 改掉吗？

**追问答案**

- 因为回调真正执行时往往不是以原对象方法的形式调用，所以容易退化成默认绑定。
- 原生 DOM 事件中，普通函数回调里的 `this` 通常指向绑定事件的元素。
- 可以。`new` 调用时会创建新的实例绑定，优先级高于 `bind` 的显式绑定。

**容易答偏的地方**

- 说成“`this` 永远指向函数定义时的对象”
- 忽略严格模式差异
- 不知道箭头函数是例外

### 7. 箭头函数的 `this` 和普通函数有什么区别？

**简答版答案**

普通函数的 `this` 由调用方式决定，箭头函数没有自己的 `this`，它会继承外层最近一层非箭头函数或词法环境中的 `this`。
所以箭头函数适合需要固定上下文的场景，但不适合作为构造函数，也不适合滥用于对象方法。

**展开版答案**

箭头函数和普通函数最大的区别之一，就是 `this` 的处理方式不同。

普通函数：

- 调用方式不同，`this` 可能不同
- 可以用 `call`、`apply`、`bind` 改变 `this`
- 可以作为构造函数，用 `new` 调用

箭头函数：

- 没有自己的 `this`
- `this` 来自定义时外层作用域
- 不能作为构造函数
- `call`、`apply`、`bind` 不能真正改变它的 `this`

例如：

```js
const obj = {
  name: 'js',
  fn: () => {
    console.log(this.name)
  }
}
```

这里很多人以为会输出 `js`，但箭头函数没有自己的 `this`，它不会因为写在对象里就自动绑定到 `obj`，而是取决于外层作用域。

箭头函数很适合写在回调里，比如定时器、数组遍历、Promise 回调，因为它可以避免手动保存 `this`。

**关键点**

- 箭头函数没有自己的 `this`
- 它的 `this` 是词法继承，不是调用时决定
- 不能作为构造函数
- 不要把对象字面量里的箭头函数误当成普通方法

**常见追问**

- 为什么箭头函数不能当构造函数？
- 箭头函数有 `arguments` 吗？
- 类实例方法适不适合写成箭头函数？

**追问答案**

- 因为箭头函数没有自己的 `this`，也没有 `prototype`，不符合构造函数的行为模型。
- 没有自己的 `arguments`，通常需要依赖外层的 `arguments`，或者直接使用剩余参数。
- 看场景。若是为了稳定绑定实例 `this` 可以使用，但会带来每个实例都创建一份函数的问题。

**容易答偏的地方**

- 误以为箭头函数里的 `this` 指向当前对象
- 不知道 `call`、`apply` 对箭头函数不生效
- 把箭头函数和普通函数只差“写法更短”来理解

### 8. `call`、`apply`、`bind` 有什么区别？

**简答版答案**

三者都可以显式指定函数执行时的 `this`。
`call` 和 `apply` 会立即执行函数，区别在于参数传递形式不同；`bind` 不会立即执行，而是返回一个绑定好 `this` 的新函数。

**展开版答案**

这三个方法的核心作用都是改变普通函数执行时的 `this` 指向。

区别可以从两个维度看：

- 是否立即执行
- 参数如何传递

`call`：

- 立即执行
- 参数逐个传入

```js
fn.call(obj, 1, 2)
```

`apply`：

- 立即执行
- 参数以数组或类数组形式传入

```js
fn.apply(obj, [1, 2])
```

`bind`：

- 不立即执行
- 返回一个新的绑定函数

```js
const newFn = fn.bind(obj, 1, 2)
newFn()
```

在面试里还可以补一句：它们只能改变普通函数的 `this`，对箭头函数的 `this` 基本无效，因为箭头函数没有自己的 `this`。

**关键点**

- 都属于显式绑定 `this`
- `call`、`apply` 立即执行
- `bind` 返回新函数
- `call` 和 `apply` 主要区别是传参方式

**常见追问**

- `bind` 实现柯里化有什么用？
- `Math.max` 为什么常和 `apply` 一起出现？
- 手写 `bind` 需要注意什么？

**追问答案**

- `bind` 可以预置部分参数，形成更具体的新函数，常用于事件回调和参数复用。
- 因为早期需要把数组展开成参数列表传入，例如 `Math.max.apply(null, arr)`。
- 需要注意返回新函数、参数拼接、`this` 绑定，以及被 `new` 调用时的兼容处理。

**容易答偏的地方**

- 只记住名字，不知道谁立即执行
- 不知道 `bind` 返回的是新函数
- 误以为它们对箭头函数同样有效

### 9. 什么是原型？什么是原型链？

**简答版答案**

原型可以理解为对象共享属性和方法的一种机制。每个对象都可以通过原型关联到另一个对象，从而实现属性复用。
原型链就是对象查找属性时，先找自己，再沿着原型一层层向上查找的链式关系。

**展开版答案**

JavaScript 的对象系统不是传统类式继承，而是基于原型的。

当我们访问一个对象上的属性时，浏览器并不是只看对象本身：

- 先看对象自己有没有这个属性
- 如果没有，就顺着它的原型继续找
- 再没有，就继续往上找
- 直到找到 `Object.prototype`
- 再找不到就是 `undefined`

这条查找路径就叫原型链。

例如：

```js
function Person() {}
Person.prototype.sayHi = function () {
  console.log('hi')
}

const p = new Person()
p.sayHi()
```

`p` 本身没有 `sayHi`，但它会沿着原型链去 `Person.prototype` 上找，所以可以访问到这个方法。

原型机制的价值在于共享。
如果把方法都直接定义在实例对象上，每创建一个实例就会复制一份；放到原型上，多个实例可以共享同一套方法。

**关键点**

- JavaScript 是基于原型的对象系统
- 查找属性时会沿原型链向上查找
- 原型机制能实现方法共享
- 终点通常是 `Object.prototype`

**常见追问**

- `Object.create` 做了什么？
- 实例属性和原型属性同名时访问谁？
- 如何判断一个属性是对象自身的还是继承来的？

**追问答案**

- `Object.create(proto)` 会创建一个新对象，并把它的原型指向 `proto`。
- 优先访问实例自身属性，自身有就不会继续沿原型链查找。
- 可以用 `Object.hasOwn(obj, key)` 或 `hasOwnProperty` 判断。

**容易答偏的地方**

- 把原型理解成“类”
- 只背定义，不会解释查找过程
- 不知道原型的核心价值是共享和继承

### 10. `prototype` 和 `__proto__` 有什么区别？

**简答版答案**

`prototype` 通常是函数才有的属性，用来作为将来实例对象共享成员的原型对象。
`__proto__` 通常是对象上的内部原型访问入口，用来指向它的原型。
简单说，`prototype` 更偏“构造函数的原型对象”，`__proto__` 更偏“实例对象指向哪个原型”。

**展开版答案**

这两个概念很容易混，但它们不是一回事。

`prototype`：

- 主要存在于函数上
- 当函数作为构造函数使用时，实例对象会关联到这个 `prototype`
- 里面通常放共享方法

```js
function Person() {}
Person.prototype.sayHi = function () {}
```

`__proto__`：

- 通常存在于对象上
- 指向该对象的原型
- 本质上是对内部 `[[Prototype]]` 的非标准早期访问方式，现在更多作为理解概念使用

例如：

```js
const p = new Person()
console.log(p.__proto__ === Person.prototype) // true
```

面试里最好补一句：业务代码里不推荐直接依赖 `__proto__`，更推荐用 `Object.getPrototypeOf()` 和 `Object.setPrototypeOf()` 这类标准方式操作原型。

**关键点**

- `prototype` 主要在函数上
- `__proto__` 主要在对象上
- 实例对象的 `__proto__` 通常指向构造函数的 `prototype`
- 现代代码不建议依赖 `__proto__`

**常见追问**

- 为什么箭头函数没有 `prototype`？
- `constructor` 属性是什么？
- 如何安全获取对象原型？

**追问答案**

- 因为箭头函数不能作为构造函数使用，所以没有用于实例继承的 `prototype`。
- `constructor` 通常是原型对象上的一个属性，默认指回构造函数本身。
- 可以使用 `Object.getPrototypeOf(obj)`。

**容易答偏的地方**

- 直接说它们“都表示原型”
- 说不清谁在函数上，谁在对象上
- 把 `__proto__` 当成推荐写法

### 11. JavaScript 中几种常见继承方式分别是什么？更推荐哪种？

**简答版答案**

常见继承方式有原型链继承、借用构造函数继承、组合继承、寄生组合继承，以及 ES6 的 `class extends`。
如果从原理和面试表达来看，更推荐寄生组合继承；如果从现代开发实践来看，通常直接使用 `class extends`，因为它本质上也是基于原型链。

**展开版答案**

JavaScript 继承的核心目标是两件事：

- 继承父类实例属性
- 复用父类原型方法

不同方案就是围绕这两个目标做取舍。

原型链继承：

- 让子类原型指向父类实例
- 能继承原型方法
- 但引用类型属性容易被多个实例共享，不够理想

借用构造函数继承：

- 在子类构造函数里调用父类构造函数
- 能继承实例属性
- 但拿不到父类原型方法

组合继承：

- 原型链 + 借用构造函数结合
- 既有实例属性，也能继承原型方法
- 但父类构造函数会调用两次

寄生组合继承：

- 在组合继承基础上优化
- 通过 `Object.create` 继承父类原型
- 避免多调用一次父类构造函数

ES6 `class extends`：

- 是更友好的语法糖
- 本质底层仍然是基于原型链实现继承

**关键点**

- 继承方案的重点是实例属性和原型方法能否同时继承
- 寄生组合继承是经典面试答案
- `class extends` 是现代开发最常用写法
- 不要把 `class` 误解成彻底脱离原型的新机制

**常见追问**

- 为什么说寄生组合继承更好？
- `class extends` 和原型链是什么关系？
- 父类引用类型属性为什么会出问题？

**追问答案**

- 因为它兼顾实例属性和原型方法继承，又避免了父类构造函数多执行一次。
- `class` 只是语法层面的封装，底层依然依赖函数、原型和原型链。
- 如果多个实例共享同一个引用类型属性，某个实例修改后会影响其他实例。

**容易答偏的地方**

- 把各种继承方案混成一团
- 只会背名字，不知道各自解决什么问题
- 以为 `class` 不是原型继承

### 12. `new` 一个对象时，底层发生了什么？

**简答版答案**

`new` 的核心过程可以概括为四步：创建一个新对象、把新对象的原型指向构造函数的 `prototype`、用新对象作为 `this` 执行构造函数、最后返回这个对象；如果构造函数显式返回一个对象，就返回那个对象。

**展开版答案**

面试里这个问题通常是在考你对构造函数和原型链的理解。

`new Foo()` 大致会做这些事：

1. 创建一个空对象
2. 把这个空对象的内部原型指向 `Foo.prototype`
3. 以这个新对象作为 `this` 执行构造函数 `Foo`
4. 如果构造函数返回的是对象类型，就返回该对象；否则返回刚创建的新对象

例如：

```js
function Person(name) {
  this.name = name
}

const p = new Person('Tom')
```

执行后 `p` 会拿到 `name` 属性，同时 `p` 的原型会指向 `Person.prototype`，这样它才能访问原型上的共享方法。

这个问题经常和“手写 `new`”联动，所以最好能把这四步说顺。

**关键点**

- `new` 会创建新对象
- 会建立原型关联
- 构造函数里的 `this` 会指向这个新对象
- 返回值规则要记住

**常见追问**

- 如果构造函数返回基本类型会怎样？
- 为什么实例能访问到原型方法？
- 手写 `new` 要怎么做？

**追问答案**

- 如果返回基本类型，会被忽略，仍然返回新创建的实例对象。
- 因为实例对象的原型指向了构造函数的 `prototype`，访问属性时会沿原型链查找。
- 核心就是手动完成创建对象、绑定原型、执行构造函数和处理返回值这四步。一个常见的手写版本可以这样写：

```js
function myNew(Constructor, ...args) {
  const obj = Object.create(Constructor.prototype)
  const result = Constructor.apply(obj, args)
  return result !== null && (typeof result === 'object' || typeof result === 'function')
    ? result
    : obj
}
```

- 这段代码可以这样理解：
  - `Object.create(Constructor.prototype)` 用来创建一个新对象，并把它的原型指向构造函数的 `prototype`
  - `Constructor.apply(obj, args)` 用来让构造函数里的 `this` 指向这个新对象，并执行初始化逻辑
  - 如果构造函数显式返回了对象或函数，就返回这个结果
  - 否则返回前面创建的新对象
- 例如：

```js
function Person(name) {
  this.name = name
}

Person.prototype.sayHi = function () {
  return `Hi, ${this.name}`
}

const p = myNew(Person, 'Tom')

console.log(p.name) // 'Tom'
console.log(p.sayHi()) // 'Hi, Tom'
```

- 这个例子说明 `myNew` 不只是创建了对象，还正确建立了原型链，所以实例既能拿到构造函数里定义的属性，也能访问原型上的方法。

**容易答偏的地方**

- 只说“创建实例对象”
- 漏掉原型绑定过程
- 不知道构造函数返回对象时会覆盖默认返回值

### 13. JavaScript 是怎么实现继承的？

**简答版答案**

JavaScript 继承的底层本质是原型链。子对象之所以能访问父对象的方法，是因为查找属性时会沿着原型链向上查找。
现代开发里 `class extends` 只是更方便的语法写法，底层仍然离不开构造函数、原型和原型链。

**展开版答案**

从语言底层看，JavaScript 没有像某些传统面向对象语言那样真正独立的一套类继承模型，它更核心的机制是对象之间通过原型建立关联。

当子实例访问某个属性或方法时：

- 先在自己身上找
- 找不到就去自己的原型找
- 再找不到继续往上找

这就是继承效果的来源。

如果再结合构造函数调用，就能实现“既继承属性，又继承方法”的完整方案。
比如在构造函数中借用父类初始化实例属性，再把子类原型和父类原型关联起来。

ES6 的 `extends` 看起来像传统类继承，但本质上仍然是对这套原型机制的封装。
所以面试回答时，最好把“继承现象”和“原型链机制”绑定起来解释。

**关键点**

- 继承效果来源于原型链查找
- 构造函数主要解决实例属性初始化
- 原型主要解决方法共享
- `class extends` 只是更现代的写法

**常见追问**

- 属性继承和方法继承分别靠什么实现？
- `super` 的本质是什么？
- 为什么说 JavaScript 更偏基于对象而不是基于类？

**追问答案**

- 属性继承常靠父构造函数执行，方法继承常靠原型链。
- `super` 可以理解为对父类构造函数或父类原型方法的访问入口。
- 因为核心能力来自对象和原型之间的关联，而不是传统类系统本身。

**容易答偏的地方**

- 把继承只解释成 `extends`
- 只会写代码，不会说底层机制
- 没把构造函数和原型各自职责说清楚

### 14. 什么是 Promise？它解决了什么问题？

**简答版答案**

Promise 是对异步操作结果的一种统一抽象，它表示一个未来才会确定的结果。
它主要解决回调函数层层嵌套、错误处理分散、异步流程难以串联的问题。

**展开版答案**

在 Promise 出现之前，前端很多异步逻辑都依赖回调函数，比如请求、定时器、事件监听。
如果多个异步任务有先后依赖关系，就很容易写成层层嵌套的“回调地狱”，代码可读性和维护性都比较差。

Promise 的核心价值在于：

- 把异步结果统一封装成对象
- 提供链式调用能力
- 让成功和失败逻辑有更清晰的组织方式

例如：

```js
fetchData()
  .then((res) => handleRes(res))
  .catch((err) => handleErr(err))
```

它让异步流程更像“数据流转”而不是“层层回调”。

Promise 本身不是让异步变同步，而是让异步代码更可组合、更可管理。

**关键点**

- Promise 是异步结果的抽象
- 解决的是异步流程组织和错误传递问题
- 重点价值是链式调用和统一错误处理
- Promise 不是取消异步，也不是同步执行

**常见追问**

- Promise 一创建就会执行吗？
- Promise 和回调函数是什么关系？
- Promise 为什么能链式调用？

**追问答案**

- 会。`new Promise(executor)` 时执行器函数会立即执行。
- Promise 底层依然离不开回调，但它把回调管理得更规范、更可组合。
- 因为 `then` 会返回一个新的 Promise，从而支持链式继续处理。

**容易答偏的地方**

- 只说“Promise 是异步”
- 说不出它解决了什么具体痛点
- 误以为 Promise 能完全消灭回调

### 15. Promise 有哪几种状态？状态为什么不可逆？

**简答版答案**

Promise 有三种状态：`pending`、`fulfilled`、`rejected`。
一旦从 `pending` 变成成功或失败，就不能再改回去，这叫状态不可逆。这样做是为了保证异步结果稳定、可预测。

**展开版答案**

Promise 在刚创建时处于 `pending`，表示“还没有最终结果”。

后续只会走两种结果之一：

- `fulfilled`：成功完成
- `rejected`：失败结束

而且一旦确定，就不会再变化。
比如一个 Promise 已经 `fulfilled`，之后再调用 `reject` 不会生效。

状态不可逆非常重要，因为它保证了异步结果的单一性和稳定性。
如果一个 Promise 先成功后失败，或者先失败后成功，链式调用就会变得不可预测，代码也会很难推理。

这也是为什么 Promise 适合作为异步流程中的“承诺”对象：结果只会定一次。

**关键点**

- 三种状态要说全
- 只有 `pending` 能流向另外两种终态
- Promise 一旦决议就不可逆
- 不可逆是为了可预测性和一致性

**常见追问**

- `resolve` 后还能 `reject` 吗？
- `resolve(Promise)` 会发生什么？
- Promise 的状态和值是什么关系？

**追问答案**

- 不能，第一次状态确定后后续调用会被忽略。
- 当前 Promise 会“接管”传入 Promise 的最终状态。
- 状态表示当前阶段，值表示成功结果或失败原因，两者是相关但不同的概念。

**容易答偏的地方**

- 把状态和值混为一谈
- 只会说成功和失败，漏掉 `pending`
- 不知道为什么要不可逆

### 16. `Promise.then`、`catch`、`finally` 的执行和返回规则是什么？

**简答版答案**

`then` 用来处理成功结果，也可以接收失败回调；`catch` 用来处理异常和失败；`finally` 无论成功还是失败都会执行。
它们都支持链式调用，本质上是因为它们都会返回一个新的 Promise。

**展开版答案**

`then` 最常见的写法是：

```js
promise.then(onFulfilled, onRejected)
```

但实际开发里通常更推荐把错误处理交给 `catch`，因为语义更清晰。

返回规则是面试重点：

- 如果回调返回普通值，下一次 `then` 会拿到这个值
- 如果回调返回一个 Promise，后续链会等待这个 Promise 结果
- 如果回调抛出错误，后续会走失败链

`catch` 本质上可以看作 `then(undefined, onRejected)` 的语法糖，更方便集中处理错误。

`finally` 的特点是：

- 不接收上一步成功值或失败原因作为主要业务处理参数
- 更适合做收尾逻辑，比如关闭 loading、释放资源
- 它也会继续向后传递之前的结果，除非它内部抛错或返回一个失败 Promise

**关键点**

- `then`、`catch`、`finally` 都会返回新 Promise
- `catch` 更适合集中错误处理
- `finally` 适合收尾，不适合改主业务结果
- 回调返回值会影响后续链的状态和值

**常见追问**

- 为什么 `then` 能链式调用？
- `then` 第二个参数和 `catch` 有什么差别？
- `finally` 会拿到结果值吗？

**追问答案**

- 因为每次调用 `then` 都会生成一个新的 Promise。
- `catch` 更适合统一捕获前面链路中的异常，而 `then` 第二个参数更容易把错误处理分散。
- 通常不拿来处理结果值，它更像一个无论成败都执行的收尾钩子。

**容易答偏的地方**

- 只会说“`catch` 捕获错误”，不会说返回规则
- 不知道 `finally` 也会影响链路
- 误以为它们只是语法写法不同，没有本质联系

### 17. `Promise.all`、`Promise.race`、`Promise.allSettled`、`Promise.any` 有什么区别？

**简答版答案**

`Promise.all` 要全部成功才成功，有一个失败就整体失败。
`Promise.race` 取第一个落定的结果，不管成功还是失败。
`Promise.allSettled` 会等待所有任务都结束，并返回每个任务的结果状态。
`Promise.any` 只要有一个成功就成功，只有全部失败才失败。

**展开版答案**

这四个静态方法本质上都是在处理“多个 Promise 的组合”，区别在于它们对成功和失败的判定规则不同。

`Promise.all`：

- 适合“所有请求都必须成功”的场景
- 返回结果顺序和传入顺序一致
- 有一个失败就直接 reject

`Promise.race`：

- 谁先结束就采用谁的结果
- 不关心是成功还是失败
- 常用于超时控制、抢最快结果

`Promise.allSettled`：

- 不会因为单个失败而提前结束
- 等所有 Promise 都结束后，返回每项的 `status` 和结果
- 适合批量任务统计

`Promise.any`：

- 只关心第一个成功结果
- 只有全部失败才 reject
- 适合多个备用请求源兜底

面试里最好再补一句：不要把它们都简单理解成“并发请求工具”，重点是理解聚合规则。

**关键点**

- 四者最核心的区别在聚合成功/失败条件
- `all` 强调全部成功
- `race` 强调最先结束
- `allSettled` 强调全部完成
- `any` 强调任一成功

**常见追问**

- `Promise.all` 返回顺序和完成顺序有关系吗？
- `Promise.any` 全部失败时返回什么？
- 哪个更适合做请求超时控制？

**追问答案**

- 没关系，返回数组顺序由传入顺序决定。
- 会返回一个聚合错误，一般是 `AggregateError`。
- 通常会配合 `Promise.race` 来实现超时控制。

**容易答偏的地方**

- 把 `race` 说成“谁先成功用谁”
- 不知道 `allSettled` 不会短路失败
- 把 `any` 和 `all` 理解反了

### 18. `async/await` 本质上是什么？它和 Promise 的关系是什么？

**简答版答案**

`async/await` 本质上是 Promise 的语法糖，用更接近同步代码的写法组织异步流程。
`async` 函数执行后一定返回一个 Promise，`await` 会等待一个 Promise 决议后再继续往下执行。

**展开版答案**

`async/await` 并没有创造新的异步模型，它只是让 Promise 链式调用换了一种更易读的表达方式。

核心点有两个：

- `async` 修饰的函数一定返回 Promise
- `await` 后面通常接 Promise，也可以接普通值

如果 `await` 的是普通值，会直接包装成已成功的 Promise 来处理。
如果 `await` 的 Promise reject，通常需要配合 `try/catch` 捕获错误。

例如：

```js
async function getData() {
  try {
    const res = await fetchData()
    return res
  } catch (err) {
    console.log(err)
  }
}
```

它在阅读体验上更接近同步代码，但本质仍然是异步，不会阻塞整个 JavaScript 主线程。

**关键点**

- `async/await` 是 Promise 语法糖
- `async` 函数一定返回 Promise
- `await` 会暂停当前 `async` 函数内部后续逻辑
- 错误处理通常配合 `try/catch`

**常见追问**

- `await` 会阻塞线程吗？
- `await` 后面能不能跟普通值？
- 多个异步任务串行和并行怎么写？

**追问答案**

- 不会阻塞整个线程，它只是让当前 `async` 函数后续逻辑等待结果。
- 可以，普通值会被当作已成功结果处理。
- 有依赖关系就串行 `await`，无依赖关系通常先发起多个 Promise 再用 `await Promise.all()` 并行等待。

**容易答偏的地方**

- 说成“`async/await` 让异步变同步”
- 不知道 `async` 一定返回 Promise
- 不会结合 `try/catch` 说明错误处理

### 19. 什么是事件循环？宏任务和微任务分别有哪些？

**简答版答案**

事件循环是 JavaScript 处理同步任务、异步回调和任务调度的运行机制。
执行顺序通常是：先执行同步代码，当前调用栈清空后，优先处理微任务，再处理一个宏任务，再继续检查微任务，如此循环。

**展开版答案**

JavaScript 主线程一次只能做一件事，但实际开发中会有定时器、Promise、DOM 事件、网络请求等异步任务，所以需要一套调度机制，这就是事件循环。

可以把它理解成几个部分协作：

- 调用栈：执行当前同步代码
- 任务队列：存放等待执行的异步回调
- 事件循环：不断检查调用栈和任务队列，安排执行顺序

常见宏任务有：

- 整体脚本
- `setTimeout`
- `setInterval`
- I/O
- UI 渲染相关任务

常见微任务有：

- `Promise.then/catch/finally`
- `MutationObserver`
- `queueMicrotask`

执行顺序上，当前宏任务执行结束后，会先把本轮产生的微任务清空，再进入下一个宏任务。
所以很多题目里 Promise 回调会先于 `setTimeout` 执行。

**关键点**

- 事件循环解决的是异步调度问题
- 同步代码优先执行
- 微任务优先级通常高于下一轮宏任务
- Promise 回调属于微任务

**常见追问**

- 为什么 Promise 比 `setTimeout` 先执行？
- 浏览器和 Node 的事件循环完全一样吗？
- 如果微任务一直追加会怎样？

**追问答案**

- 因为当前宏任务结束后，会先清空微任务队列，再取下一个宏任务。
- 核心思想相似，但具体阶段和实现细节不完全一样。
- 可能导致后续宏任务迟迟得不到执行，出现任务饥饿问题。

**容易答偏的地方**

- 只背“宏任务微任务”，说不清整体流程
- 把 `Promise` 说成宏任务
- 不知道“当前宏任务结束后先清空微任务”

### 20. `setTimeout(fn, 0)` 为什么不是立即执行？

**简答版答案**

`setTimeout(fn, 0)` 的意思不是立刻执行，而是“最早在当前调用栈清空、并且轮到这个宏任务时再执行”。
所以它至少要等同步代码执行完，还要等前面的任务调度完成。

**展开版答案**

很多人会误以为 `setTimeout(fn, 0)` 等于“马上执行”，其实它只是把回调放进定时器对应的任务队列，并设置一个最小延迟时间。

它不能立即执行的原因主要有三个：

- 当前同步代码还在调用栈里，必须先执行完
- 事件循环要等调用栈清空后才会调度异步回调
- `setTimeout` 回调属于宏任务，还要排队等待前面的宏任务和微任务处理完

例如：

```js
console.log(1)
setTimeout(() => console.log(2), 0)
Promise.resolve().then(() => console.log(3))
console.log(4)
```

结果通常是：

```js
1
4
3
2
```

因为同步代码先执行，再执行微任务，最后才轮到 `setTimeout` 的宏任务。

**关键点**

- `0` 不代表立即执行
- `setTimeout` 回调是宏任务
- 调用栈清空前不会执行异步回调
- 微任务通常先于它执行

**常见追问**

- 浏览器定时器最小时间间隔一定是 `0` 吗？
- 为什么嵌套定时器会有最小延迟限制？
- 如何实现真正意义上的“尽快执行”？

**追问答案**

- 不一定，浏览器可能基于环境和策略做最小延迟限制。
- 因为浏览器会出于性能和调度考虑，对连续嵌套定时器做节流。
- 通常要看场景，很多时候微任务或更合理的调度方式比单纯用 `setTimeout(fn, 0)` 更合适。

**容易答偏的地方**

- 只说“因为 JS 是单线程”
- 不知道它属于宏任务
- 不能结合事件循环解释清楚

### 21. 深拷贝和浅拷贝有什么区别？常见实现方式有哪些？

**简答版答案**

浅拷贝只复制第一层属性，如果属性值还是对象，拷贝后仍然共享同一引用；深拷贝会递归复制所有层级，让新旧对象互不影响。
常见浅拷贝有展开运算符、`Object.assign`；常见深拷贝有 `structuredClone`、递归实现，以及特定场景下的库方案。

**展开版答案**

浅拷贝和深拷贝的核心区别在于：是否会继续复制嵌套对象。

浅拷贝示例：

```js
const obj1 = { a: 1, b: { c: 2 } }
const obj2 = { ...obj1 }
obj2.b.c = 3
```

这里虽然 `obj2` 是新对象，但 `b` 仍然和 `obj1` 指向同一个嵌套对象，所以修改会互相影响。

深拷贝则会继续处理嵌套层级，让内部对象也变成新的引用。

常见方式：

- 浅拷贝：`Object.assign`、展开运算符、数组的 `slice` / `concat`
- 深拷贝：`structuredClone`
- 手写递归深拷贝
- 借助第三方库

有些人会提 `JSON.parse(JSON.stringify(obj))`，它可以处理部分简单场景，但有明显局限，比如会丢失函数、`undefined`、`symbol`、循环引用以及特殊对象类型信息，所以不能作为通用深拷贝方案。

**关键点**

- 核心区别是嵌套引用是否被复制
- 展开运算符和 `Object.assign` 都是浅拷贝
- 深拷贝要考虑循环引用和特殊对象
- `JSON` 方案有明显限制

**常见追问**

- 为什么 `JSON` 方案不可靠？
- 手写深拷贝要考虑哪些边界？
- `structuredClone` 有什么优缺点？

**追问答案**

- 因为它会丢失部分类型信息，也处理不了循环引用。
- 需要考虑数组、对象、循环引用、日期、正则、Map、Set 等特殊类型。
- 它是原生方案，使用方便，能处理很多复杂数据，但也不是所有环境和所有类型都完全适配。

**容易答偏的地方**

- 把展开运算符说成深拷贝
- 无脑把 `JSON` 方案当万能答案
- 说不清“共享引用”到底会带来什么问题

### 22. 节流和防抖有什么区别？各自适合什么场景？

**简答版答案**

防抖是“频繁触发时，只在最后一次触发后执行一次”，适合输入搜索、表单校验这类希望等用户停下来再处理的场景。
节流是“频繁触发时，按固定时间间隔执行”，适合滚动、拖拽、窗口缩放这类持续触发但需要控制执行频率的场景。

**展开版答案**

这两个问题本质上都在解决高频事件带来的性能压力，但控制思路不同。

防抖：

- 每次触发都重新计时
- 只有在一段时间内不再触发，才真正执行
- 适合“结果只关心最后一次”的场景

比如搜索框输入联想，如果用户还在连续输入，就没必要每敲一个字都发请求。

节流：

- 无论事件触发多频繁，都按固定节奏执行
- 适合“持续触发但不能每次都执行”的场景

比如页面滚动监听、鼠标拖拽、窗口缩放，如果每一帧都做重计算，性能会很差。

所以可以简单记：

- 防抖看“停下来”
- 节流看“隔一段时间执行一次”

**关键点**

- 两者都是高频事件优化手段
- 防抖强调最后一次
- 节流强调固定频率
- 场景选择比定义本身更重要

**常见追问**

- 防抖和节流能不能同时支持立即执行？
- 手写时需要注意什么？
- 在 React 或 Vue 里怎么使用更合适？

**追问答案**

- 可以，很多实现会支持 `leading`、`trailing` 这类控制项。
- 需要注意 `this`、参数透传、定时器管理，以及取消能力。
- 一般会在事件处理函数层面封装，并注意组件卸载时清理定时器或引用。

**容易答偏的地方**

- 把两者定义说反
- 只背定义，不会举业务场景
- 不知道手写时要处理 `this` 和参数

### 23. CommonJS 和 ES Module 有什么区别？前端为什么要做模块化？

**简答版答案**

模块化是为了把代码按职责拆分，避免全局污染，提高复用、维护和协作效率。
CommonJS 是运行时加载，主要用于 Node.js 生态；ES Module 是语言层面的官方模块标准，支持静态分析，更适合现代前端构建和 tree shaking。

**展开版答案**

模块化的核心价值在于把原本散落在全局作用域里的代码按功能组织起来，让变量、函数、依赖关系都更清晰。

前端为什么需要模块化：

- 避免全局变量污染
- 提升代码复用和维护性
- 让多人协作边界更清楚
- 让打包、按需加载、tree shaking 等工程能力成为可能

CommonJS 的特点：

- 典型写法是 `require` 和 `module.exports`
- 更偏运行时加载
- 在 Node.js 里长期使用很多

ES Module 的特点：

- 典型写法是 `import` 和 `export`
- 属于 JavaScript 官方标准
- 支持静态分析
- 更利于打包优化和 tree shaking

前端现代项目通常以 ES Module 为主，但在一些工程链路或服务端环境里仍然会接触到 CommonJS。

**关键点**

- 模块化的本质是代码组织和依赖管理
- CommonJS 偏运行时
- ES Module 偏语言标准和静态分析
- 现代前端以 ES Module 为主

**常见追问**

- 什么是 tree shaking？
- CommonJS 为什么不利于静态分析？
- ES Module 一定是浏览器才能用吗？

**追问答案**

- tree shaking 是在打包时移除没有被使用的导出代码，减少最终产物体积。
- 因为 `require` 可以动态执行，依赖关系不一定在编译阶段完全确定。
- 不是。现代 Node.js 也支持 ES Module，只是使用方式和配置要符合环境要求。

**容易答偏的地方**

- 只会背 `import/export` 和 `require/export`
- 不知道模块化解决的是全局污染和依赖管理问题
- 说不清 ES Module 为什么更利于打包优化
