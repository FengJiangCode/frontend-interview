## 一、为什么需要事件循环

JavaScript 是单线程语言，同一时刻只能执行一段代码。但浏览器中充满了"需要等待结果"的操作（网络请求、定时器、用户交互）。如果用同步阻塞来处理，页面会卡死。

事件循环（Event Loop）是 JavaScript 运行时协调**调用栈**、**Web API / C++ 线程**和**任务队列**的调度机制，让单线程的 JS 能以非阻塞的方式处理异步任务。

## 二、核心组件

```
┌────────────────────────────────────────────────────────┐
│                      JS 运行时                          │
│                                                        │
│   ┌──────────┐    ┌───────────────────────────────┐   │
│   │  调用栈   │    │         Web APIs              │   │
│   │ Call     │    │  setTimeout / fetch /         │   │
│   │ Stack    │    │  DOM events / ...             │   │
│   └────┬─────┘    └──────────────┬────────────────┘   │
│        │                         │ 回调注册             │
│        │          ┌──────────────▼────────────────┐   │
│        │          │  任务队列（Task Queue）         │   │
│        │          │                               │   │
│        │          │  宏任务队列  [cb1] [cb2] ...   │   │
│        │          │  微任务队列  [cb3] [cb4] ...   │   │
│        │          └──────────────┬────────────────┘   │
│        │                         │                    │
│   ◄────┤    事件循环（Event Loop）├────────────────    │
│   当栈空时，从队列取回调          │                    │
└────────────────────────────────────────────────────────┘
```

| 组件 | 作用 |
|---|---|
| **调用栈** | 执行当前同步代码，函数调用形成栈帧，执行完弹出 |
| **Web APIs** | 浏览器提供的线程，处理定时器计时、网络请求等，完成后把回调推入任务队列 |
| **宏任务队列** | 存放等待执行的宏任务回调 |
| **微任务队列** | 存放等待执行的微任务回调，**优先级高于宏任务** |
| **事件循环** | 持续检查：调用栈空了 → 先清空微任务 → 取一个宏任务执行 → 重复 |

## 三、宏任务 vs 微任务

### 宏任务（Macrotask / Task）

| 来源 | 说明 |
|---|---|
| 整体脚本（script） | 第一个宏任务 |
| `setTimeout` / `setInterval` | 计时结束后进入宏任务队列 |
| `setImmediate` | Node.js 专有，check 阶段 |
| I/O 事件回调 | 文件读写、网络请求的回调 |
| UI 渲染 | 浏览器的渲染帧（不完全等同于宏任务，见第六节） |
| `MessageChannel` | postMessage 场景 |

### 微任务（Microtask）

| 来源 | 说明 |
|---|---|
| `Promise.then / catch / finally` | 最常见 |
| `async/await` | await 之后的代码等价于 `.then` 回调 |
| `queueMicrotask()` | 显式入队微任务 |
| `MutationObserver` | 浏览器 DOM 变化监听 |
| `process.nextTick` | Node.js 专有，**优先级高于 Promise.then** |

> **记忆口诀**：Promise 系列 + MutationObserver + queueMicrotask → 微任务；setTimeout/setInterval/I/O → 宏任务。

## 四、执行顺序——完整算法

```
1. 执行当前宏任务（第一个宏任务就是整体 <script>）
2. 调用栈清空后：
   a. 取出微任务队列中所有微任务，逐一执行
      （执行过程中新产生的微任务也立即追加到本轮清空）
   b. 微任务队列清空后，浏览器可能执行一次渲染
3. 从宏任务队列取出下一个宏任务，回到步骤 1
```

**关键点**：
- 每个宏任务结束后，必须先把**当前轮次产生的全部微任务**清空，才进入下一宏任务
- 微任务可以无限追加微任务，全部清空才算完——这也意味着微任务死循环会阻塞后续宏任务

## 五、高频面试题（逐题分析）

### 题 1：基础顺序

```js
console.log('A')
setTimeout(() => console.log('B'), 0)
Promise.resolve().then(() => console.log('C'))
console.log('D')
```

**答案：A D C B**

分析：
1. 同步：A → D
2. 调用栈清空，清微任务：Promise.then → C
3. 取下一宏任务：setTimeout → B

---

### 题 2：微任务中产生的微任务

```js
Promise.resolve()
  .then(() => {
    console.log(1)
    Promise.resolve().then(() => console.log(2))
  })
  .then(() => console.log(3))
```

**答案：1 2 3**

分析：
- 外层 `.then` 执行，输出 1，同时内部 `Promise.resolve().then(→ 2)` 加入微任务队列
- 此时微任务队列：`[→ 2, → 3]`（第二个 `.then` 在第一个 `.then` 完成后才入队，顺序是 2 先入）
- 清空：先 2，再 3

> **关键**：微任务执行期间产生的新微任务，会追加到**当前微任务队列尾部**，本轮一起清空。

---

### 题 3：async/await 拆解

```js
async function foo() {
  console.log(1)
  await bar()
  console.log(2)       // ← 等价于 Promise.then 回调
}

async function bar() {
  console.log(3)
}

foo()
console.log(4)
```

**答案：1 3 4 2**

分析：
- `foo()` 开始执行，输出 1
- 遇到 `await bar()`：先执行 `bar()`，输出 3；`bar()` 返回 `Promise.resolve(undefined)`
- `await` 将 `foo` 中 `await` 之后的代码（`console.log(2)`）包装成微任务挂起，控制权交还给调用方
- 同步继续：输出 4
- 调用栈清空，清微任务：`console.log(2)` → 2

---

### 题 4：await 嵌套层数影响（经典）

```js
async function async1() {
  console.log('async1 start')
  await async2()
  console.log('async1 end')
}

async function async2() {
  console.log('async2')
}

console.log('script start')
setTimeout(() => console.log('setTimeout'), 0)
async1()
new Promise(resolve => {
  console.log('promise1')
  resolve()
}).then(() => console.log('promise2'))
console.log('script end')
```

**答案：script start → async1 start → async2 → promise1 → script end → async1 end → promise2 → setTimeout**

分析：
1. 同步：`script start`
2. setTimeout 回调进宏任务队列
3. `async1()` 调用：`async1 start` → `async2()` → `async2` → `async2` 返回 resolved Promise → `async1 end` 作为微任务挂起
4. `new Promise` 构造器同步执行：`promise1` → `resolve()` → `.then(→ promise2)` 加入微任务
5. 同步：`script end`
6. 清微任务：`async1 end`（先入队）→ `promise2`
7. 宏任务：`setTimeout`

---

### 题 5：setTimeout 嵌套

```js
setTimeout(() => {
  console.log(1)
  Promise.resolve().then(() => console.log(2))
}, 0)

setTimeout(() => {
  console.log(3)
}, 0)
```

**答案：1 2 3**

分析：
- 第一个宏任务（setTimeout1）执行：输出 1，Promise.then 加入微任务
- 宏任务结束，清微任务：输出 2
- 取下一宏任务（setTimeout2）：输出 3

> 每个宏任务执行完都会清空微任务，所以 2 在 3 之前。

---

### 题 6：Promise 链与宏任务交替（综合题）

```js
console.log(1)

setTimeout(() => console.log(2), 0)

new Promise(resolve => {
  console.log(3)
  resolve()
  console.log(4)
}).then(() => {
  console.log(5)
  setTimeout(() => console.log(6), 0)
}).then(() => console.log(7))

setTimeout(() => console.log(8), 0)

console.log(9)
```

**答案：1 3 4 9 5 7 2 8 6**

分析：
1. 同步：1 → 3（Promise 构造器同步） → 4 → 9
2. 宏任务队列此时：`[→2, →8]`；微任务：`[→5]`
3. 清微任务：5（执行时 setTimeout6 进宏任务，第二个 `.then(→7)` 进微任务） → 7
4. 宏任务队列：`[→2, →8, →6]`，依次执行：2 → 8 → 6

---

### 题 7：queueMicrotask

```js
setTimeout(() => console.log('timeout'))
queueMicrotask(() => console.log('microtask'))
console.log('sync')
```

**答案：sync → microtask → timeout**

`queueMicrotask` 直接向微任务队列追加，效果等同于 `Promise.resolve().then()`，但语义更明确。

---

### 题 8：微任务饥饿（陷阱题）

```js
function loop() {
  Promise.resolve().then(loop)
}
loop()
setTimeout(() => console.log('never'), 0)
```

`setTimeout` 的回调**永远不会执行**。

原因：`loop` 每次执行都在微任务队列尾部追加新的微任务，微任务队列永远清不完，事件循环永远到不了下一个宏任务。这是真实的"微任务饥饿"。

## 六、渲染时机与 requestAnimationFrame

浏览器的渲染不是任务队列里的"宏任务"，而是在**每轮事件循环的微任务清空之后、下一个宏任务之前**，由浏览器自行决定是否执行渲染。

```
宏任务 → 清空微任务 → [浏览器渲染？] → 下一个宏任务
```

`requestAnimationFrame(callback)` 的回调在**浏览器渲染前**执行（即 style / layout / paint 之前），是修改 DOM 的最佳时机。

```
微任务清空 → rAF 回调 → 浏览器渲染（style/layout/paint）→ 下一宏任务
```

> **面试追问**：为什么用 rAF 做动画比 setTimeout 流畅？  
> 因为 rAF 与浏览器的刷新频率（通常 60fps = 16.7ms/帧）精确对齐，每帧只执行一次，不会出现 setTimeout 计时不准导致的丢帧或重复渲染。

## 七、Node.js 事件循环

Node.js 基于 libuv，事件循环有 **6 个阶段**，按顺序循环执行：

```
   ┌───────────────────────────┐
┌─>│         timers            │  ← setTimeout / setInterval 到期的回调
│  └──────────┬────────────────┘
│  ┌──────────▼────────────────┐
│  │     pending callbacks     │  ← 上一轮被延迟的 I/O 回调（如 TCP 错误通知）
│  └──────────┬────────────────┘
│  ┌──────────▼────────────────┐
│  │     idle, prepare         │  ← 内部使用
│  └──────────┬────────────────┘
│  ┌──────────▼────────────────┐
│  │         poll              │  ← 等待新的 I/O 事件；执行 I/O 回调
│  └──────────┬────────────────┘
│  ┌──────────▼────────────────┐
│  │         check             │  ← setImmediate 回调
│  └──────────┬────────────────┘
│  ┌──────────▼────────────────┐
└──┤     close callbacks       │  ← socket.on('close') 等
   └───────────────────────────┘
```

### 六个阶段详解

**阶段 1：timers**

执行 `setTimeout` 和 `setInterval` 到期的回调。

关键点：不是"到了时间就执行"，而是"到了时间才有资格在这个阶段被执行"。如果此时事件循环正在其他阶段，即使计时器已到期，也要等到下一轮 timers 阶段才执行。

```js
setTimeout(() => console.log('timer'), 100)
// 100ms 后进入 timers 队列，等到事件循环跑到 timers 阶段才真正执行
```

**阶段 2：pending callbacks**

执行上一轮被系统延迟通知的 I/O 回调（如 TCP ECONNREFUSED）。这些回调在 poll 阶段没能被及时处理，操作系统要求推迟到下一轮再通知。日常代码几乎感知不到这个阶段，属于内部机制。

**阶段 3：idle / prepare**

libuv 内部使用，Node.js 开发者无法介入，跳过即可。

**阶段 4：poll（最重要）**

执行 I/O 完成后的回调（文件读写、网络请求等），同时等待新的 I/O 事件。这是事件循环的核心阶段，大部分时间都花在这里。

> **关键理解**：I/O 操作本身由 libuv 线程池在后台执行，不阻塞事件循环。poll 阶段做的是询问操作系统哪些 I/O 完成了，把对应回调取出来执行。

```
进入 poll 阶段
  ├─ poll 队列有回调 → 逐一执行，直到队列清空
  └─ poll 队列为空
       ├─ 有 setImmediate 待执行 → 结束 poll，进入 check 阶段
       └─ 没有 setImmediate → 在这里阻塞等待，直到有新 I/O 事件或计时器到期
```

```js
fs.readFile('file.txt', (err, data) => {
  // 这个回调放在 poll 队列里
  console.log(data)
})

// 背后的执行流：
// fs.readFile 调用
//   ├─ 文件读取 → 交给 libuv 线程池（后台，不阻塞事件循环）
//   └─ 事件循环继续跑...
//        └─ poll 阶段：线程池读完了？有！→ 把回调放入 poll 队列 → 执行
```

**阶段 5：check**

执行 `setImmediate` 的回调。`setImmediate` 专门设计来在 poll 阶段结束后立即执行，所以它总是比下一轮 `setTimeout` 更早——这就是 I/O 回调内 `setImmediate` 必然先于 `setTimeout` 的原因：

```js
fs.readFile('file', () => {
  // 当前在 poll 阶段
  setTimeout(() => console.log('timeout'))     // 要等下一轮 timers
  setImmediate(() => console.log('immediate')) // poll → check，马上执行
})
// 始终输出：immediate → timeout
```

**阶段 6：close callbacks**

执行资源关闭的回调，如 `socket.on('close', ...)`、`process.on('exit', ...)`。

```js
const server = net.createServer()
server.on('close', () => console.log('server closed')) // 在这里执行
server.close()
```

### 微任务插在哪里

每两个阶段切换之间，Node.js 都会完整清空微任务，顺序固定：

```
[阶段N结束] → 清空 nextTick 队列 → 清空 Promise.then 队列 → [阶段N+1开始]
```

完整节奏：

```
timers → 微任务 → pending callbacks → 微任务 → idle/prepare → 微任务
→ poll → 微任务 → check → 微任务 → close callbacks → 微任务 → 下一轮...
```

> **面试一句话总结**：poll 阶段是核心，负责处理 I/O 并等待新事件；timers 在最前负责定时器；check 在 poll 后专属 setImmediate；每个阶段切换时都会清空 nextTick 和 Promise 微任务。

---

### Node.js 微任务优先级

在每个阶段切换时，Node.js 都会清空微任务，且顺序是：

```
process.nextTick 队列 → Promise.then 微任务队列
```

`process.nextTick` 的优先级**高于** `Promise.then`，这是 Node.js 独有的行为。

### setImmediate vs setTimeout(fn, 0)

```js
setTimeout(() => console.log('timeout'), 0)
setImmediate(() => console.log('immediate'))
```

**在主模块（非 I/O 回调内）**：顺序不确定，取决于进入事件循环时的计时误差。

**在 I/O 回调内**：`setImmediate` 总是先于 `setTimeout`，因为 I/O 完成后进入 poll 阶段，poll → check（setImmediate），计时器要等下一轮 timers 阶段。

```js
const fs = require('fs')
fs.readFile('file', () => {
  setTimeout(() => console.log('timeout'))
  setImmediate(() => console.log('immediate'))
})
// 始终输出：immediate → timeout
```

### process.nextTick 示例

```js
Promise.resolve().then(() => console.log('promise'))
process.nextTick(() => console.log('nextTick'))
console.log('sync')
```

**答案：sync → nextTick → promise**

## 八、高频追问 Q&A

**Q1：微任务和宏任务的本质区别是什么？**

不是"谁先谁后"的优先级问题，而是**调度粒度**不同。宏任务是"一次完整的任务单元"（每次事件循环只取一个），微任务是"当前任务的尾声"（必须在进入下一任务前全部清空）。微任务设计出来就是为了让某些逻辑"尽快"但"不急于当下"执行——Promise 的回调需要异步但又不能等太久。

**Q2：如果 Promise 回调里又创建了 Promise，会怎样？**

新的 Promise.then 追加到**当前微任务队列尾部**，会在本轮宏任务结束前全部清空。不会等到下一个宏任务。

**Q3：async/await 和 Promise 的关系？**

`async/await` 是 Promise 的语法糖：
- `async function` 返回一个 Promise
- `await expr` 等价于 `Promise.resolve(expr).then(继续执行后续代码)`

所以 `await` 之后的代码，本质上是一个微任务回调。

**Q4：为什么 `new Promise(executor)` 中的代码是同步的？**

Promise 构造器的 `executor` 函数在 `new` 调用时**立即、同步**执行，目的是让 Promise 的"触发逻辑"可以同步运行（比如 `new Promise(resolve => setTimeout(resolve, 1000))`，setTimeout 的注册是同步发生的）。只有 `.then` / `.catch` 的回调才是异步的。

**Q5：Node.js 中 process.nextTick 和 Promise.then 哪个先执行？**

`process.nextTick` 先。nextTick 队列在每个阶段间隙被优先清空，然后才是 Promise 微任务队列。这是 Node.js 的设计，目的是让 nextTick 能在 I/O 事件和定时器之前"抢占"执行，但滥用会导致 I/O 饥饿。

**Q6：浏览器和 Node.js 的事件循环最核心的区别？**

- 浏览器：两类任务（宏任务/微任务），每个宏任务后清空微任务，渲染穿插其中
- Node.js：6 个阶段的循环，每个阶段间清空 nextTick + Promise 微任务，`setImmediate` 有专属的 check 阶段，`process.nextTick` 优先级高于 Promise

**Q7：setTimeout(fn, 0) 的 0ms 是真正的 0ms 吗？**

不是。HTML 规范规定最小延迟为 **4ms**（嵌套超过 5 层时强制生效）。实际环境中受系统计时器精度、浏览器节流（后台标签页可能降至 1000ms）等影响，`setTimeout(fn, 0)` 只是"尽快执行"，不是"立即执行"。

**Q8：微任务会阻断渲染吗？**

会。渲染发生在微任务队列清空之后，所以只要微任务队列不清空，浏览器就不会渲染。微任务死循环（不断在微任务里追加新微任务）会让页面彻底冻结，效果等同于 `while(true)`。

```js
// 页面卡死，渲染永远不来
function loop() { Promise.resolve().then(loop) }
loop()
```

如果需要做大量异步计算又不想阻塞渲染，应该用 `setTimeout` / `MessageChannel` 分片，而不是微任务链。

**Q9：requestIdleCallback 是什么？和 rAF 有什么区别？**

`requestIdleCallback(callback)` 在浏览器**空闲时**（一帧渲染完、还有剩余时间时）执行回调，适合低优先级、可推迟的任务（如数据上报、预加载）。

```js
requestIdleCallback((deadline) => {
  while (deadline.timeRemaining() > 0 && tasks.length) {
    doTask(tasks.shift())   // 利用剩余帧时间处理任务
  }
  if (tasks.length) requestIdleCallback(...)  // 没做完，下一帧继续
}, { timeout: 2000 })        // 最长等 2s，超时强制执行
```

**"帧剩余时间"和事件循环的关系：**

浏览器以 60fps 运行时，每帧预算约 16.7ms，事件循环在这 16.7ms 内要完成：

```
┌──────────────────────────── 16.7ms ────────────────────────────┐
│  宏任务 → 清微任务 → rAF 回调 → 样式/布局/绘制 → [剩余时间？]  │
└────────────────────────────────────────────────────────────────┘
```

前面这些任务用完的时间越少，剩余的空闲窗口越大，`requestIdleCallback` 就在这个窗口里触发：

```
前置任务耗时 10ms  →  剩余 6.7ms  →  rIC 可用，timeRemaining() ≈ 6.7
前置任务耗时 17ms  →  已超帧预算  →  rIC 被推迟到下一帧
```

`deadline.timeRemaining()` 返回的就是"距离下一帧开始还剩多少毫秒"。rIC 不是独立的调度机制，它寄生在事件循环的帧间隙里——帧内任务越重，留给 rIC 的时间越少甚至直接跳过，这也是它不适合紧急任务的根本原因。

| | `requestAnimationFrame` | `requestIdleCallback` |
|---|---|---|
| 执行时机 | 每帧渲染**前**，必执行 | 每帧渲染**后**有剩余时间时 |
| 优先级 | 高，与渲染同步 | 低，可被推迟 |
| 适合场景 | DOM 动画、样式修改 | 分析、上报、预加载 |
| 可控性 | `deadline.timeRemaining()` 不可用 | 可查剩余时间，主动让出 |

**Q10：什么是时间分片？为什么用 MessageChannel 而不是 setTimeout？**

时间分片（Time Slicing）：把一个耗时长任务切成多个小片，每片执行完后主动让出主线程，让浏览器有机会渲染和响应用户输入，而不是阻塞到整个任务结束。

```js
function timeSlice(tasks) {
  function run() {
    const start = performance.now()
    while (tasks.length && performance.now() - start < 5) {
      tasks.shift()()           // 每次最多执行 5ms
    }
    if (tasks.length) channel.port1.postMessage(null)  // 让出后继续
  }
  const channel = new MessageChannel()
  channel.port2.onmessage = run
  channel.port1.postMessage(null)
}
```

用 `MessageChannel` 而不是 `setTimeout` 的原因：`MessageChannel` 的回调是宏任务，但**没有 4ms 最小延迟**，调度精度更高。React Scheduler 内部就是用 `MessageChannel` 来产生宏任务间隙的。

**Q11：Vue nextTick 的实现原理是什么？**

`nextTick` 的目标是"等本轮同步代码执行完、DOM 更新完之后再执行回调"。Vue 用微任务来实现这个"最近的异步时机"：

优先级降级策略（从高到低）：
1. `Promise.then`（微任务，现代浏览器）
2. `MutationObserver`（微任务，兼容旧版 iOS）
3. `setImmediate`（宏任务，Node.js / IE）
4. `setTimeout(fn, 0)`（宏任务，最后兜底）

为什么优先用微任务？因为微任务在当前宏任务结束前执行，能最早拿到更新后的 DOM，比宏任务方案延迟更小。

```js
// 使用场景：修改数据后立即读 DOM，需等 Vue 更新完
this.message = 'updated'
this.$nextTick(() => {
  console.log(this.$el.textContent)  // 能拿到更新后的 DOM
})
```

**Q12：Web Worker 有自己的事件循环吗？**

有。每个 Worker 运行在独立线程，有自己独立的事件循环（宏任务 + 微任务），与主线程完全隔离，不能访问 DOM。

主线程和 Worker 通过 `postMessage` 通信，消息传递是异步的（走各自的宏任务队列）。Worker 内部的 Promise、setTimeout 等行为和主线程规则完全一样。

适合场景：CPU 密集计算（图像处理、大数据解析），放入 Worker 后不会阻塞主线程的渲染和交互。

**Q13：为什么 React Fiber 要自己实现调度器？**

浏览器原生没有"任务优先级"和"可中断渲染"的能力——一旦开始渲染组件树，就必须跑完，期间无法响应用户输入。React Fiber 用 `MessageChannel` 产生宏任务间隙，在每帧剩余时间内尽量渲染，时间不够就中断、把控制权交还浏览器，下一帧再继续。

关键点：Fiber 把组件树的渲染从"一次性递归"改成"可中断的链表遍历"，配合自己的优先级队列（用户交互 > 动画 > 数据请求 > 后台任务），实现了并发模式（Concurrent Mode）。这本质上是在事件循环层面做的调度优化。

## 九、执行顺序速记框架

面对输出题，套用这个框架逐步分析：

```
第一步：找出所有同步代码，按顺序记录输出
第二步：找出微任务（Promise.then / await 后 / nextTick），按入队顺序排列
第三步：找出宏任务（setTimeout / setInterval），按入队顺序排列
第四步：按"同步 → 清微任务 → 宏任务1 → 清微任务 → 宏任务2 → ..."执行
注意：每个宏任务执行期间产生的微任务，在该宏任务结束后立即清空，不等下一轮
```

## 十、容易答偏的地方

1. **把"宏任务/微任务"当唯一知识点**。面试官真正想考的是你能不能追踪完整的执行流程，背概念不如能准确还原执行顺序。

2. **忘记 Promise 构造器是同步的**。`new Promise(executor)` 的 `executor` 立即执行，不是微任务。

3. **混淆 await 的暂停时机**。`await expr` 先把 `expr` 求值（同步），然后把"之后的代码"挂起为微任务。如果 `expr` 本身是同步值，等价于 `Promise.resolve(expr).then(...)`。

4. **认为 Node 和浏览器完全一样**。`process.nextTick`、`setImmediate`、6 阶段循环是 Node 独有的，浏览器没有。

5. **忽略微任务饥饿问题**。如果在微任务里无限追加微任务，后续宏任务（包括 setTimeout、I/O）永远得不到执行。
