## 高频追问 Q&A

**Q1：Formily 具体怎么实现字段联动的？**
Formily 有一套响应式状态管理（基于 `@formily/reactive`），字段之间通过 `reactions` 配置互相监听。比如"国家"字段变化时，触发"费率方案"字段的选项联动，全部通过 JSON Schema 描述，不写命令式代码。

**Q2：元数据平台是怎么与 Formily 结合的？**
元数据平台的职责是让非技术人员可视化配置表单，最终输出一份标准的 JSON Schema。Formily 的 `SchemaForm` 组件直接消费这个 JSON，解析并渲染出完整的表单 UI，包括校验规则和联动逻辑。我们做的核心工作是：定义 JSON Schema 规范 + 封装自定义业务组件 + 处理 Formily 与业务逻辑的边界。

**Q3：你负责重构，原有业务怎么保证稳定迁移？**
采用渐进式迁移策略。先选复杂度中等的签约流程做试点，新老方案双跑对比，确认功能对齐后再推进。同时制定了清晰的迁移规范，让其他同学也能独立完成迁移。

**Q4：团队接受新方案的阻力大吗？怎么推动的？**
有一定阻力，主要是 Formily 的学习成本。我的做法：先做出能跑的 Demo 降低感知门槛；再组织3次技术分享，结合实际业务案例讲，而不是照着文档念。最后用数据说话——第一个迁移场景上线后开发周期明显缩短，团队自然接受了。

**Q5：JSON Schema 的版本管理怎么做的？**
Schema 存在元数据平台，有版本记录和变更审计，不是散落在代码里的文件。表单渲染时拉取对应版本的 Schema，历史数据可回溯。如果 Schema 有破坏性变更，需要走评审流程。

**问题一：授信流程混乱**

**问题二：性能优化**
没有银弹，系统性诊断各个环节：
- 路由懒加载 + 代码分割，减少首屏 bundle 体积

**问题三：监控与SOP**
整合集团已有监控平台，接入核心链路的性能和错误指标；制定线上问题处理SOP：发现 → 定级 → 响应时限 → 排查路径 → 修复验证，让整个团队都能按流程处理，不依赖单个关键人。

## 高频追问 Q&A

**Q1：任务流引擎具体怎么设计的？**
核心是把"授信该走哪条链路"的决策从页面里抽出来，统一放在引擎里。每个授信环节定义为一个 Task（有唯一ID、执行函数、完成后的跳转目标）。引擎持有一份任务序列配置（JSON），根据用户渠道和实验版本选取对应序列，按顺序执行任务，任务完成后引擎决定下一步跳转。页面组件只关心"当前任务是什么"，不关心前后的流程逻辑。

**Q2：Node中间层做了哪些接口聚合？**
首页加载依赖多个接口（用户状态、贷款产品信息、AB实验分组等），原先在客户端串行/并行请求，任何一个慢都影响首屏。我在 Node 层把这些聚合成一个 `/api/init`，服务端并发请求、合并结果后返回给客户端，客户端一个请求拿到所有首屏数据。这是LCP改善最显著的单项优化。

**Q3：LCP从10秒到4秒，哪些手段贡献最多？**
接口聚合贡献最大，节省了多个串行请求的时间；其次是首屏代码分割，把非首屏路由的 JS 延迟加载；第三是将 AB 实验决策移到 Node 层，避免客户端等待实验接口返回才能渲染。

**Q4：带5人团队，遇到过什么管理挑战？怎么解决的？**
最典型的两个：代码质量参差不齐，以及需求估时不准。前者通过建立代码规范文档 + CR 流程解决，规范不是约束而是减少 CR 来回沟通的成本；后者通过迭代 buffer 机制（每个迭代预留弹性时间）+ 风险提前暴露来解决，全年做到无前端原因导致的延期。

**Q5：监控怎么衡量效果？**
核心指标是告警发现时长（从问题出现到告警触发）和 P3 事故数量。建立监控后，故障平均响应时间缩短了50%，全年无P3及以上。数据来自集团监控平台的月度报告。

## 展开版（5分钟+）

### 背景与挑战

国际化现金贷项目从国内金融部门迁移时，沿用国内统一组件库，实际落地遇到三个问题：

### 方案选型

**为什么选 vant-cli 而不是从零搭建？**
vant-cli 提供了完整的组件库工程化脚手架——文档站生成、构建打包、单测集成，省去了大量基础设施工作，让我们专注在组件设计和业务适配上。同时 vant 本身是成熟的移动端组件库，可以参考其 API 设计规范。

**核心设计决策：**
- **换肤**：用 CSS Custom Properties（CSS变量）实现主题层，组件内部只引用变量不写死数值，切换皮肤时替换根层变量值，无需重新加载资源
- **多语言**：集成 vue-i18n，组件内部文案全部走 i18n key，提供默认语言包，业务层可覆盖
- **BEM 规范**：统一 CSS 命名，避免样式污染，保证组件在不同项目中的可预期性

## 高频追问 Q&A

**Q1：换肤是怎么实现的？**
基于 CSS Custom Properties（CSS变量）。在组件库根层定义一套主题变量（颜色、字体大小、圆角等），组件内部只使用变量。切换皮肤时，动态替换根层的变量值即可，无需修改任何组件代码，也不需要重新加载资源，性能开销极小。

**Q2：多语言方案有什么挑战？**
主要挑战是组件库的多语言和业务层的多语言要解耦。组件库内部文案通过 i18n key 管理，默认提供几个语言包，业务层也可以覆盖特定 key。难点在于版本同步——组件库升级新增文案时，要确保业务侧语言包能对应更新，这需要在 changelog 里明确标注 i18n 变更。

**Q3：组件库如何保证跨项目的稳定性？**
三个层面：遵循语义化版本管理（breaking change 必须升 major，有明确 changelog）；CR 流程保证代码质量；单元测试正在推进中，是目前要补强的短板。破坏性变更需要走评审并同步各项目接入方。

**Q4：从0搭建组件库，遇到最难的部分是什么？**
不是技术实现，是设计规范的对齐。UI 习惯用 Figma 描述视觉，开发需要翻译成 CSS 变量和组件 API，前期沟通成本很高。我们的做法是建立"组件设计稿 + 对应开发规范文档"的双轨模式，走查时按文档比对，大幅减少了来回沟通，这也是走查效率提升30%的来源。

**Q5：为什么选择自研而不是改造开源库？**
改造开源库的问题是：改得越多，升级成本越高，最终变成一个难以维护的 fork。我们的业务需求（多语言、换肤、海外审美）足够有规律，自研可以完全按业务需要设计，而不是在别人的约束里做妥协。vant-cli 提供的工程化工具降低了自研的门槛，让自研的成本是可接受的。

## 容易答偏的地方

1. **不要说"就是封装了一套UI库"**。重点在于你解决了什么业务痛点（国际化支持缺失、自主维护迭代节奏），以及架构决策背后的思考（为什么用CSS变量换肤而不是多套样式文件，为什么选vant-cli而不是从零搭）。

2. **AI单测是加分项，不要过度渲染**。目前还在建设中，实事求是说"目标覆盖率80%，正在推进"，并能解释为什么选AI辅助（组件数量多，手写成本高）就足够了。
