## 文档结构（每份文件统一）

```
# 项目名
> 元信息（时间、角色、规模）

## 一句话版（10秒）
## 简答版（1-2分钟）
## 展开版（5分钟+）
  ### 背景与挑战
  ### 方案选型（为什么不选X，为什么选Y）
  ### 关键实现
  ### 结果与影响
## 高频追问 Q&A（4-6个）
## 容易答偏的地方
```

## 叙事策略

采用「冲突-决策-结果」线：先抛核心困境让面试官代入，再展示方案选型判断力，最后量化收益。技术深度定位中高级：讲清楚选型逻辑（为什么选 Formily/任务流引擎/自研组件库），不展开底层实现细节。

## 各项目核心叙事锚点

### 项目一：国际化外卖运营工作台
- 冲突：100+字段的复杂表单 + 6国业务规则差异 + 硬编码导致维护成本爆炸
- 决策：元数据平台 + Formily 配置化，JSON Schema 驱动，逻辑与业务解耦
- 结果：开发周期 5人/天 → 1人/天，代码风格一致率+80%，准时率95%+

### 项目二：巴西现金贷H5
- 冲突：前后端1:4人力比 + 多渠道多版本AB逻辑混乱 + LCP 10秒以上 + 无监控
- 决策：任务流引擎解耦授信流程 + Node中间层聚合接口 + 接入集团监控平台
- 结果：LCP 10s→4s，全年0起P3事故，响应时间缩短50%，无需求延期

### 项目三：金融组件库
- 冲突：国内组件库不支持国际化/多语言/换肤，跨部门依赖响应慢
- 决策：基于 vant-cli 自研，i18n + 换肤 + BEM + npm 包发布体系
- 结果：40+组件覆盖10+项目，设计走查效率+30%，推进AI单测目标覆盖率80%
