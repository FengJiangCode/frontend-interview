# 工程化面试复习清单

## 使用方式

- 先看问题，尝试自己回答
- 再对照简答版和展开版检查表达是否完整
- 最后补常见追问和容易答偏的地方

### 1. 什么是前端工程化？为什么工程化题里经常会聊到 Webpack、Vite、Babel、Git？

**简答版答案**

前端工程化本质上是把前端开发从“手写几个页面”升级成一套可协作、可维护、可交付的开发流程。面试里经常聊到 `Webpack`、`Vite`、`Babel`、`Git`，是因为它们分别代表了构建、开发体验、编译兼容、团队协作这几条核心主线。

**展开版答案**

前端工程化不是单独某个工具，而是一整套围绕项目开发效率和质量稳定性的方案。

如果项目规模很小，可能直接写几个 HTML、CSS、JS 文件也能跑起来。但只要项目开始变复杂，就会马上遇到这些问题：

- 代码怎么组织和拆分
- 不同模块怎么相互依赖
- 新语法怎么兼容不同浏览器
- 多人协作怎么减少冲突
- 本地开发和线上构建怎么统一

工程化就是在解决这些问题。

为什么工程化题里最常出现的是 `Webpack`、`Vite`、`Babel`、`Git`？

- `Webpack`
  - 代表传统构建工具体系
  - 重点是模块依赖分析、打包、资源处理、插件扩展
- `Vite`
  - 代表更现代的开发构建方案
  - 重点是开发阶段更快、生产环境仍然可以完成构建输出
- `Babel`
  - 代表编译转换链路
  - 重点是语法降级、兼容性处理、新旧环境适配
- `Git`
  - 代表团队协作流程
  - 重点是分支管理、代码合并、版本追踪和协作规范

所以面试里问工程化，通常并不是让你背工具清单，而是看你能不能讲清：

- 工具在解决什么问题
- 工具之间是什么关系
- 为什么项目里需要这些流程和约束

**关键点**

- 工程化是一套流程和体系，不是某一个工具
- 核心目标是提升开发效率、协作效率和交付稳定性
- `Webpack`、`Vite`、`Babel`、`Git` 分别对应不同工程化环节

**常见追问**

- 工程化和模块化有什么区别？
- 小项目也需要工程化吗？
- 工程化的收益主要体现在哪些地方？

**追问答案**

- 模块化更偏代码组织方式，工程化范围更大，还包括构建、编译、规范、协作和交付流程。
- 需要，但程度不一样。小项目不一定需要很重的配置，但基本的构建、版本管理和协作规范仍然有价值。
- 常见收益包括开发效率提升、代码质量更稳定、多人协作更顺畅、上线流程更可控。

**容易答偏的地方**

- 只会罗列工具，不会说它们分别解决什么问题
- 把工程化理解成“会配 Webpack”
- 只讲打包，不讲协作和兼容性

### 2. Webpack 的核心能力是什么？它的构建流程大致是怎样的？

**简答版答案**

`Webpack` 的核心能力是把前端项目看成一个模块依赖图，然后从入口出发分析依赖、处理资源，最后输出浏览器可运行的构建结果。它不只是打包工具，本质上还是一个可扩展的构建平台。

**展开版答案**

很多人会把 `Webpack` 简单理解成“把很多文件打成一个包”，但这个说法太窄了。

`Webpack` 更准确的理解方式是：

- 以入口文件为起点
- 递归分析模块依赖
- 在构建过程中处理 JavaScript、CSS、图片等资源
- 最后生成浏览器可以使用的产物

它的核心思想是“万物皆模块”。

也就是说，在 `Webpack` 看来，不只是 JS 可以成为模块，像 CSS、Less、图片、字体等资源，也可以通过相应处理方式被纳入构建流程。

它的大致流程可以这样理解：

- 确定入口 `entry`
- 从入口开始分析依赖
- 遇到不同类型资源时，交给对应规则处理
- 在构建过程中通过插件介入生命周期
- 最后根据输出配置生成产物

如果用更面试化的表达，可以概括成一句：

- 输入：源码和资源文件
- 处理：依赖分析、模块转换、插件扩展
- 输出：浏览器可部署的静态资源

所以 `Webpack` 的价值不只是“能打包”，而是它把前端项目的构建过程统一抽象成了一条可配置、可扩展的流水线。

**关键点**

- `Webpack` 的核心是模块依赖图，不只是文件拼接
- 它能统一处理多种前端资源
- 构建流程适合用“输入-处理-输出”来表达

**常见追问**

- 什么叫模块依赖图？
- 为什么说 Webpack 是可扩展的？
- Webpack 的入口和出口分别起什么作用？

**追问答案**

- 可以理解为一个从入口文件不断向外展开的依赖关系网络，哪些模块依赖哪些模块，都会被记录和处理。
- 因为它提供了 `Loader` 和 `Plugin` 两套扩展机制，可以对不同资源和不同构建阶段做自定义处理。
- 入口决定从哪里开始分析项目，出口决定最终构建产物输出到哪里、叫什么名字。

**容易答偏的地方**

- 只会说“Webpack 就是打包工具”
- 说不清它为什么能处理 CSS、图片等资源
- 流程表达混乱，只会背零散名词

### 3. Loader 和 Plugin 有什么区别？分别解决什么问题？

**简答版答案**

`Loader` 更像资源转换器，主要负责把某类文件变成 `Webpack` 能继续处理的模块；`Plugin` 更像构建流程扩展器，主要负责在构建生命周期中做更广义的能力增强。

**展开版答案**

这是 `Webpack` 面试里非常高频的一道题。

很多人会记成一句话：

- `Loader` 处理文件
- `Plugin` 扩展功能

这句话方向没错，但面试里最好再展开一点。

`Loader` 的重点是“转换”。

因为浏览器本身只能直接识别部分资源，而工程化项目里会有很多浏览器不能直接理解的内容，比如：

- ESNext 语法
- TypeScript
- Less / Sass
- CSS 模块化写法

这时就需要通过 `Loader` 把这些内容先转换成 `Webpack` 能继续打包的模块形式。

比如：

- `babel-loader` 负责把较新的 JS 语法交给 `Babel` 处理
- `css-loader` 负责让 CSS 也能参与模块依赖
- `style-loader` 负责把样式注入页面

而 `Plugin` 的重点不是单个文件转换，而是对整个构建过程做功能扩展。

比如：

- 自动生成 HTML
- 提取 CSS 文件
- 清理输出目录
- 定义环境变量
- 压缩构建产物

所以更稳的理解方式是：

- `Loader` 解决“某类资源怎么变成模块”
- `Plugin` 解决“整个构建流程还要增加什么能力”

**关键点**

- `Loader` 偏资源转换
- `Plugin` 偏构建流程扩展
- 两者共同组成了 `Webpack` 的可扩展能力

**常见追问**

- 为什么 CSS 处理通常会出现多个 Loader？
- Loader 执行顺序一般是怎样的？
- Plugin 一般通过什么方式介入构建？

**追问答案**

- 因为 CSS 处理往往不是一个动作，比如既要解析依赖，也可能要做预处理、模块化、再注入页面，所以会串联多个 Loader。
- 一般可以理解为从后往前执行，所以配置时需要注意链路顺序。
- Plugin 会基于 `Webpack` 提供的生命周期钩子，在合适阶段执行对应逻辑。

**容易答偏的地方**

- 把 Loader 和 Plugin 都说成“插件”
- 只会背定义，没有讲清两者解决的问题层级不同
- 不知道为什么一个资源处理链里会有多个 Loader

### 4. Webpack 的打包优化思路有哪些？常见的优化方向是什么？

**简答版答案**

`Webpack` 优化通常可以分成开发体验优化、构建速度优化和产物体积优化三类。面试里不要只背某个配置项，更重要的是先说你在优化什么，再说为什么这么做。

**展开版答案**

说 `Webpack` 优化时，最容易答乱，因为配置项很多。

更推荐的表达方式是先分目标，再讲方案。

常见可以分成三类：

- 提升开发阶段体验
- 提升构建速度
- 降低产物体积和运行成本

开发体验优化常见思路有：

- 使用 HMR，减少整页刷新
- 合理配置 Source Map，兼顾调试和速度
- 缩小监听和编译范围，减少无关文件处理

构建速度优化常见思路有：

- 减少不必要的 Loader 和 Plugin
- 对耗时转换做缓存
- 利用多进程或并行能力处理重任务
- 缩小 Babel 等转换工具的处理范围

产物优化常见思路有：

- 代码分割，避免首屏加载过大
- Tree Shaking，移除未使用代码
- 压缩 JS、CSS
- 图片和静态资源优化
- 抽离第三方依赖或做更合理的缓存策略

如果面试官继续追问，核心一定要回到两个判断标准：

- 你是在优化开发效率，还是在优化上线产物
- 你是在减少构建时间，还是在减少运行时负担

这样回答会比单纯背配置项更有条理。

**关键点**

- 优化要先分目标，再谈手段
- 开发阶段优化和生产环境优化不是一回事
- 代码分割、缓存、压缩、Tree Shaking 都是高频方向

**常见追问**

- 为什么优化 Babel 处理范围会提升构建速度？
- 代码分割主要解决什么问题？
- Tree Shaking 为什么能减少体积？

**追问答案**

- 因为 Babel 转换通常比较耗时，如果只处理真正需要转译的源码，而不是把无关目录也扫进去，构建时间会更可控。
- 主要是避免把所有代码都塞进一个大包里，让用户按需加载，降低首屏压力。
- 因为它会基于模块引用关系，把没有被实际使用的导出尽量去掉，从而减少无效代码进入最终产物。在 Webpack 里可以顺手补一句，Tree Shaking 常见会聊到两种模式：
  - `usedExports`
  - `sideEffects`
- `usedExports` 更偏“导出级别”的分析。它会基于 ES Module 的静态结构，标记哪些导出被使用了，哪些没有被使用。被标记为未使用的导出，后续通常再配合压缩工具一起移除。
- `sideEffects` 更偏“模块级别”的裁剪。它关注的是某个模块是否有副作用，如果项目里明确声明模块没有副作用，比如在 `package.json` 里配置 `sideEffects: false`，那么 Webpack 在发现这个模块没有被用到时，可以更激进地把整个模块都去掉。
- 两者的区别可以这样记：
  - `usedExports`：看模块里哪些导出没被用到
  - `sideEffects`：看整个模块能不能直接删除
- 面试里如果要举例，可以这样说：

```js
// math.js
export const add = (a, b) => a + b
export const minus = (a, b) => a - b
```

```js
// index.js
import { add } from './math'

console.log(add(1, 2))
```

- 这里 `minus` 没被使用，`usedExports` 会先把它标记出来；如果模块本身又没有副作用，后续就更容易被 Tree Shaking 干掉。
- 所以更完整的回答可以概括成一句：Tree Shaking 依赖 ES Module 的静态分析能力，Webpack 里常见会结合 `usedExports` 和 `sideEffects` 两种模式来做导出级别和模块级别的无效代码消除。

**容易答偏的地方**

- 一上来就堆配置名词，没有先说优化目标
- 把开发环境优化和生产环境优化混在一起
- 只会说“开启缓存、分包、压缩”，但说不清分别解决什么问题

### 5. Vite 为什么开发阶段更快？它和 Webpack 的核心差异是什么？

**简答版答案**

`Vite` 开发阶段更快，核心原因是它利用浏览器原生 `ES Module` 能力，启动时不需要像 `Webpack` 那样先把整个项目都打包一遍，而是按需加载和按需编译。它和 `Webpack` 的核心差异，主要就在开发阶段的处理模型不同。

**展开版答案**

很多人面试里会直接说：

- `Vite` 快，因为不用打包

这个说法方向对，但还不够完整。

更准确地说，`Vite` 在开发阶段采用的是一种更轻量的处理方式：

- 浏览器请求哪个模块，就按需返回哪个模块
- 源码转换也尽量按需进行
- 第三方依赖会提前做预构建，减少重复处理成本

如果面试官继续追问“那 Vite 在开发环境到底是怎么处理项目的”，可以按下面这条流程回答：

1. 先启动一个本地开发服务器，而不是先把整个业务代码打成一个 bundle
2. 先对 `node_modules` 里的第三方依赖做预构建
3. 浏览器访问页面时，会从 `index.html` 开始发起模块请求
4. 浏览器请求到哪个源码模块，Vite 就按需编译并返回哪个模块
5. 返回结果通常是浏览器可以直接执行的 `ES Module`
6. 文件改动后，Vite 再基于模块边界做更精准的 HMR 更新

这里最容易混淆的一点是：Vite 开发环境不是完全“什么都不构建”，而是不会像传统打包工具那样先把业务源码整体打成一个大包。它会先处理依赖，再对源码做按需编译。

依赖预构建这一步通常会借助 `esbuild`，目的主要有两个：

- 把部分第三方依赖转成更适合浏览器直接加载的 `ES Module`
- 减少开发阶段反复处理零散依赖的成本

而对业务源码来说，Vite 更像是在做“实时转换”：

- 遇到 `TS`、`JSX`、Vue 单文件组件、CSS 等内容时，再根据请求按需处理
- 处理完之后直接返回给浏览器，而不是先合并成一个完整 bundle

而传统 `Webpack` 开发模式通常是：

- 启动开发服务前，先构建一份比较完整的模块依赖关系
- 文件一多、依赖一复杂，启动和热更新时间就会越来越明显

所以两者最核心的差异，不只是“谁快谁慢”，而是开发阶段的思路不一样：

- `Webpack` 更偏统一打包后再提供服务
- `Vite` 更偏基于原生模块能力按需提供源码

当然，`Vite` 也不是完全不构建。

到了生产环境，它仍然需要把项目构建成适合部署的产物。只是它把“开发快”和“生产构建”这两个场景区分得更清楚。

**关键点**

- `Vite` 的快主要体现在开发阶段
- 核心原因是原生 `ES Module` + 按需加载 + 依赖预构建
- 不要把“开发快”误解成“完全不需要构建”

**常见追问**

- 什么是依赖预构建？
- 为什么项目越大，Webpack 启动慢的问题越明显？
- Vite 的 HMR 为什么通常体验更轻？

**追问答案**

- 可以理解为先把第三方依赖处理成更适合浏览器快速加载的形式，避免开发过程中反复转换。面试里如果要展开，可以补一句：它主要发生在开发环境启动早期，重点针对 `node_modules`，而不是把业务源码整体先打包一遍。
- 因为它需要维护更完整的依赖分析和打包处理链路，项目规模越大，这部分前置成本越高。
- 因为它更新时通常只处理变更影响到的模块，不需要重新构建过大的模块集合。也就是说，它的 HMR 更依赖模块边界和按需更新，而不是整包思路。

**容易答偏的地方**

- 只会说“Vite 比 Webpack 快”
- 把 Vite 说成完全不打包
- 不知道它快在开发阶段，而不是所有场景都绝对更快

### 6. Webpack 和 Vite 分别适合什么项目场景？怎么做技术选型？

**简答版答案**

`Webpack` 和 `Vite` 没有绝对谁更好，关键要看项目现状、生态兼容、团队成本和性能诉求。一般来说，新项目和现代前端栈更容易优先考虑 `Vite`，而历史项目、复杂定制构建链路或生态强依赖 `Webpack` 的场景，继续使用 `Webpack` 也很常见。

**展开版答案**

技术选型这类题，面试官往往不是在问你“背答案”，而是在看你有没有判断标准。

如果是新项目，尤其是：

- 技术栈比较现代
- 团队希望提升本地开发体验
- 构建链路不需要过多历史兼容

那很多团队会优先考虑 `Vite`，因为：

- 启动和热更新通常更轻快
- 配置相对更直观
- 现代框架生态对它支持已经比较成熟

但如果是下面这些场景，`Webpack` 仍然很常见：

- 老项目已经深度绑定 `Webpack`
- 有比较复杂的自定义 Loader / Plugin 体系
- 有大量历史配置和兼容性包袱
- 团队对现有链路已经非常熟悉

所以技术选型不要说成“新项目一定 Vite，老项目一定 Webpack”，而要从几个维度评估：

- 开发体验
- 历史迁移成本
- 生态兼容性
- 构建定制度
- 团队熟悉程度

如果这是面试，比较稳的表达通常是：

- 新项目优先考虑 `Vite`
- 老项目是否迁移，要综合收益和成本判断

这样既体现你知道趋势，也不会显得太绝对。

**关键点**

- 选型要看项目背景和团队成本
- `Vite` 更适合现代项目快速起步
- `Webpack` 在复杂历史项目里仍然很有价值

**常见追问**

- 老项目有必要迁移到 Vite 吗？
- 迁移构建工具最大的风险是什么？
- 面试里怎么表达选型取舍更稳妥？

**追问答案**

- 不一定。要看迁移收益是否足够覆盖改造成本，以及现有构建链路是否已经稳定。
- 最大风险通常是生态兼容和历史配置迁移，尤其是项目里有较多自定义构建逻辑时。
- 建议从“收益、成本、风险、团队熟悉度”四个维度来回答，不要只说趋势。

**容易答偏的地方**

- 把选型题答成站队题
- 一味强调新工具更先进，不讲迁移成本
- 只说快，不说兼容性和团队落地成本

### 7. Babel 的作用是什么？它和 Webpack、Vite 分别是什么关系？

**简答版答案**

`Babel` 的核心作用是把较新的 JavaScript 语法转换成目标环境更容易运行的代码。它本身主要负责“编译转换”，而 `Webpack`、`Vite` 主要负责“构建流程”，三者既有关联，也不是同一层面的工具。

**展开版答案**

`Babel` 经常和 `Webpack` 一起出现，所以很多人容易把它们混成一个工具链里的同一层。

其实更准确的理解是：

- `Babel` 负责代码转换
- `Webpack`、`Vite` 负责组织整体构建流程

比如你写了箭头函数、可选链、类字段等新语法，某些运行环境可能并不能直接很好地支持，这时就可以通过 `Babel` 做语法降级。

它的核心处理链路通常可以理解为：

- 先把源码解析成 AST
- 再根据规则做语法转换
- 最后生成新的代码

那它和 `Webpack`、`Vite` 的关系是什么？

和 `Webpack` 的关系通常比较直接：

- `Webpack` 负责构建
- `babel-loader` 把 JS 文件交给 `Babel` 处理

和 `Vite` 的关系则要分场景理解：

- `Vite` 自己能处理一部分现代前端开发转换需求
- 但在某些兼容性、语法插件或特定生态场景下，仍然可能结合 `Babel`

所以不要把 `Babel` 理解成“打包工具”，它更像构建链路中的一个编译转换环节。

**关键点**

- `Babel` 解决的是新语法兼容和代码转换问题
- `Webpack`、`Vite` 解决的是构建流程问题
- 三者可以协作，但职责不一样

**常见追问**

- Babel 为什么要先转成 AST？
- Babel 只能处理 JavaScript 吗？
- 现代项目里还有必要理解 Babel 吗？

**追问答案**

- 因为 AST 更适合做结构化分析和精确转换，字符串替换很难可靠处理复杂语法。
- 它主要面向 JavaScript 及相关生态语法转换，但本质思路是“解析 -> 转换 -> 生成”。
- 有必要。即使很多脚手架把它封装起来了，你也要知道兼容性和语法转换问题到底是谁在处理。

**容易答偏的地方**

- 把 Babel 说成打包工具
- 只会说“Babel 用来转 ES6”
- 不知道它和 Webpack、Vite 的职责边界

### 8. Babel 的语法转换和 Polyfill 有什么区别？

**简答版答案**

语法转换主要解决“代码写法太新，旧环境看不懂”的问题；`Polyfill` 主要解决“某些新 API 在旧环境里根本不存在”的问题。一个偏语法层，一个偏运行时能力补齐层。

**展开版答案**

这是 `Babel` 里非常容易混淆的一组概念。

很多人会笼统地说：

- `Babel` 可以兼容低版本浏览器

但如果面试官继续追问，最好能拆开说成两件事。

第一类是语法转换。

比如：

- 箭头函数
- class
- 模板字符串

这些本质上是“代码写法”的问题。旧环境可能不认识这些语法，所以需要先把它们改写成更基础的写法。

第二类是运行时能力缺失。

比如：

- `Promise`
- `Array.prototype.includes`
- `Map`

这些不是改改语法就能解决，因为旧环境里可能压根没有这些对象和方法。这时就需要 `Polyfill` 去补上对应能力。

所以更准确的表达是：

- 语法转换负责把“写法”变旧
- `Polyfill` 负责把“能力”补齐

面试里如果你能补一句，会更完整：

- 不是所有兼容性问题都能只靠语法转换解决

这通常会显得你理解得更扎实。

**关键点**

- 语法转换和 `Polyfill` 不是一回事
- 语法转换解决写法兼容
- `Polyfill` 解决运行时 API 缺失

**常见追问**

- 为什么 `Promise` 不能只靠语法转换解决？
- Polyfill 会不会增加包体积？
- 面试里怎么快速区分语法问题和 API 问题？

**追问答案**

- 因为 `Promise` 是运行时对象，不是语法糖。旧环境没有这个能力时，只改源码写法是不够的。
- 会，所以实际项目里通常会结合目标环境和按需策略来控制引入范围。
- 可以先判断它是“写法”还是“环境能力”。写法问题偏语法转换，能力问题偏 `Polyfill`。

**容易答偏的地方**

- 把语法转换和 `Polyfill` 混为一谈
- 只会举箭头函数，举不出运行时 API 的例子
- 误以为用了 Babel 就自动解决了所有兼容性问题

### 9. Git 常用工作流有哪些？团队协作里为什么需要分支规范？

**简答版答案**

`Git` 常用工作流本质上是在解决多人协作时的代码隔离、合并节奏和发布管理问题。团队需要分支规范，是为了降低冲突、减少混乱，并让开发、测试、上线流程更可控。

**展开版答案**

`Git` 面试题通常不会只问命令，还会追问协作思路。

为什么团队里要有工作流？

因为多人并行开发时，如果大家都直接往主分支提交代码，会很容易出现这些问题：

- 功能彼此干扰
- 代码状态不稳定
- 测试和发布边界不清晰
- 回滚和问题定位变困难

所以团队通常会基于 `Git` 设计一套分支协作规则。

常见思路包括：

- 主分支保持稳定可发布
- 功能开发在 feature 分支进行
- 合并前通过代码评审和测试
- 需要时再区分开发、测试、发布分支

具体工作流形式可能不止一种，比如：

- 比较轻量的 feature branch workflow
- 更强调发布节奏的 Git Flow
- 更偏持续集成的小步合并方式

面试里不一定要把每个工作流都展开到很细，但至少要说明：

- 为什么要有分支隔离
- 为什么主分支要保持稳定
- 为什么合并前要有 review 和验证

这样才能体现你理解的是协作机制，而不是只会 `git add`、`git commit`。

**关键点**

- 工作流的核心是协作秩序，不是命令堆砌
- 分支规范能降低冲突并提高发布稳定性
- 主分支通常应保持稳定和可回溯

**常见追问**

- 为什么很多团队不建议直接在主分支开发？
- 功能分支开发完一般怎么合并更合适？
- Git Flow 和更轻量的分支策略有什么差别？

**追问答案**

- 因为主分支通常承担稳定版本基线的角色，直接开发会让风险和不确定性集中到最关键的分支上。
- 一般会先经过自测、代码评审和必要验证，再决定是合并还是变基后合并，核心是保证历史和协作流程清晰。
- Git Flow 分支设计更完整，适合发布节奏明确的团队；轻量策略更简洁，适合持续迭代更快的项目。

**容易答偏的地方**

- 把 Git 面试题答成命令背诵题
- 只会说 feature 分支，却说不清为什么要这么分
- 不知道分支规范和上线稳定性之间的关系

### 10. Git 里 `merge` 和 `rebase` 有什么区别？面试里应该怎么解释取舍？

**简答版答案**

`merge` 是把两条分支历史合并到一起，通常会保留真实分叉痕迹；`rebase` 是把当前分支提交“挪到”另一条分支最新提交之后，让历史看起来更线性。两者没有绝对优劣，关键看团队对历史可读性和协作方式的要求。

**展开版答案**

这道题高频但也很容易答偏成“谁更高级”。

更稳的说法是先讲行为差异。

`merge` 的特点通常是：

- 保留原有分支历史
- 能看出分叉和汇合关系
- 在多人协作里更直观地反映真实开发过程

`rebase` 的特点通常是：

- 把提交接到新的基线上
- 让提交历史更线性
- 日志看起来更整洁

所以它们最大的区别，不只是命令行为，而是提交历史的表达方式不同。

那面试里怎么讲取舍？

比较常见、也比较稳妥的表达是：

- 本地整理自己分支历史时，可以适当使用 `rebase`
- 公共分支或多人共享分支，通常要更谨慎使用 `rebase`
- 分支合并到主线时，是否用 `merge` 或其他策略，要遵循团队规范

原因在于：

- `rebase` 会改写提交历史
- 如果已经推送并被多人协作依赖，再改历史会增加沟通和冲突成本

所以这道题最好的回答方式不是站队，而是体现你知道：

- 两者行为差异
- 各自适用场景
- 团队规范比个人习惯更重要

**关键点**

- `merge` 偏保留真实历史
- `rebase` 偏整理线性历史
- 是否使用要看协作场景和团队规范

**常见追问**

- 为什么说 rebase 会改写历史？
- 什么场景下更适合用 rebase？
- 如果团队里每个人习惯不一样，应该怎么处理？

**追问答案**

- 因为它不是简单生成一个新的合并点，而是把原来的提交基线改到新的位置上，所以提交指向会变化。
- 比如个人分支还没广泛共享时，想先整理提交顺序、减少无意义分叉，通常会更适合。
- 应该以团队约定为准，因为 Git 操作不只是个人习惯问题，还会直接影响协作成本和历史可读性。

**容易答偏的地方**

- 把 `rebase` 说成“比 merge 更高级”
- 只说历史更干净，不说改写历史的风险
- 不知道公共分支上滥用 `rebase` 为什么会带来问题
