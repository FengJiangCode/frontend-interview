# 浏览器原理面试复习清单

## 使用方式

- 先看问题，尝试自己回答
- 再对照简答版和展开版检查表达是否完整
- 最后补常见追问和容易答偏的地方

### 1. 从输入 URL 到页面展示，中间发生了什么？

**简答版答案**

从输入 URL 到页面展示，通常会经历 URL 解析、缓存判断、DNS 解析、建立连接、发送 HTTP 请求、服务端响应、浏览器解析资源、构建渲染树、布局和绘制等步骤。面试里重点不是把所有词背出来，而是能讲清主流程和每一步在做什么。

**展开版答案**

这是浏览器原理里最经典的一道串联题，本质上是在考你能不能把网络、浏览器渲染、缓存这些知识点串成一条完整链路。

大致可以按下面顺序理解：

- 输入 URL
  - 浏览器先解析 URL，判断协议、域名、路径等信息
- 检查缓存
  - 先看本地是否有可直接使用的缓存资源
- DNS 解析
  - 把域名解析成 IP 地址
- 建立连接
  - HTTP 通常基于 TCP 建立连接
  - HTTPS 还会在此基础上增加 TLS 握手
- 发送请求
  - 浏览器把请求报文发给服务器
- 服务器处理并返回响应
  - 返回 HTML、CSS、JS、图片等资源
- 浏览器开始解析页面
  - 解析 HTML 构建 DOM
  - 解析 CSS 构建 CSSOM
  - 合成渲染树
- 布局和绘制
  - 计算元素位置和大小
  - 把像素绘制到屏幕上
- 遇到外链资源继续请求
  - 比如 CSS、JS、图片、字体文件等

真正面试时不一定要求你讲到非常细，但至少要把“网络请求”和“页面渲染”这两大阶段讲出来。

如果再讲完整一点，可以补一句：渲染并不是等所有资源都回来后才开始，而是浏览器会边下载边解析、边构建页面。

**关键点**

- 这是一个“网络 + 渲染”的综合流程题
- 核心主线是：解析 URL -> 请求资源 -> 解析渲染
- 缓存、DNS、TCP/HTTPS、DOM/CSSOM、布局绘制都可能被追问

**常见追问**

- DNS 解析大致是怎么做的？
- 为什么页面不是等所有资源都回来才显示？
- CSS 和 JS 会不会阻塞页面渲染？

**追问答案**

- DNS 解析本质上是把域名映射成 IP，常见会经历浏览器缓存、系统缓存、本地配置和 DNS 服务器查询等过程。
- 因为浏览器是流式解析的，拿到一部分 HTML 后就会逐步开始构建 DOM 和页面。
- CSS 会影响渲染树构建，JS 在某些情况下也会阻塞 HTML 解析或影响渲染时机，所以常常会带来阻塞问题。

**容易答偏的地方**

- 只会背几个关键词，串不成完整流程
- 只讲网络请求，不讲后面的渲染过程
- 把所有细节混在一起，导致表达没有主线

### 2. 浏览器解析 URL 后，网络请求大致经历了哪些过程？

**简答版答案**

浏览器发起网络请求前后，通常会经历缓存判断、DNS 解析、建立 TCP 连接、HTTPS 握手、发送请求报文、服务器返回响应报文等步骤。重点是理解“如何找到服务器、如何建立通信、如何拿到响应”。

**展开版答案**

这道题比“输入 URL 到页面展示”更聚焦网络链路本身。

可以把它拆成几个连续阶段：

- 先判断是否命中缓存
  - 如果本地缓存可直接使用，请求可能都不需要真正发到服务器
- DNS 解析
  - 把域名解析成服务器 IP
- 建立连接
  - HTTP 常见是 TCP 连接
  - HTTPS 还要在 TCP 基础上做 TLS 握手
- 发送请求
  - 浏览器会把请求行、请求头、请求体等信息发给服务器
- 服务器处理请求
  - 路由、鉴权、业务处理、查询数据等
- 返回响应
  - 包括状态码、响应头、响应体
- 浏览器接收响应并处理
  - 根据内容类型决定怎么解析和后续如何加载资源

面试里如果题目问“网络请求流程”，一般最值得讲清的是：

- 域名怎么变成 IP
- HTTPS 为什么比 HTTP 多一步
- 请求和响应报文大致包含什么

如果用户第一次访问和后续访问不同，也可以顺带提一下缓存和连接复用的影响。

**关键点**

- 网络请求不是“浏览器直接找服务器拿数据”这么简单
- 关键步骤是缓存、DNS、连接建立、发送请求、返回响应
- HTTPS 相比 HTTP 多了安全协商过程

**常见追问**

- DNS 为什么会影响首屏时间？
- TCP 三次握手为什么需要三次？
- 长连接和短连接有什么区别？

**追问答案**

- 因为浏览器拿不到 IP 就没法真正发起请求，所以 DNS 解析会直接影响请求开始时间。
- 因为客户端和服务端需要相互确认发送和接收能力，三次握手是为了让双方都确认连接可用。
- 长连接会复用已有连接，减少重复建立连接的开销；短连接通常一次请求结束后就关闭连接。

**容易答偏的地方**

- 只讲 DNS，不讲后面的请求和响应过程
- 说 HTTPS 时只会说“更安全”，说不清多了什么步骤
- 把 TCP、HTTP、HTTPS 的层次关系混在一起

### 3. HTTP 和 HTTPS 有什么区别？

**简答版答案**

HTTP 是明文传输协议，HTTPS 是在 HTTP 基础上加了 TLS/SSL 的安全层。HTTPS 相比 HTTP 主要多了加密、身份认证和数据完整性校验，因此更安全。

**展开版答案**

HTTP 和 HTTPS 的核心差别，不只是“有没有证书”，而是通信是否安全。

HTTP 的特点是：

- 数据通常是明文传输
- 传输过程中容易被窃听或篡改
- 默认端口通常是 `80`

HTTPS 的特点是：

- 在 HTTP 基础上加入 TLS/SSL
- 数据传输前会协商密钥并加密
- 可以校验证书，确认服务端身份
- 能校验数据是否被中途篡改
- 默认端口通常是 `443`

所以 HTTPS 的核心价值通常可以概括成三点：

- 机密性
- 完整性
- 身份认证

在今天的 Web 开发里，HTTPS 基本已经是默认要求，尤其是登录、支付、用户信息、后台管理等场景，没有 HTTPS 风险会非常高。

**关键点**

- HTTP 明文传输，HTTPS 加密传输
- HTTPS 基于 HTTP + TLS/SSL
- HTTPS 提供机密性、完整性和身份认证

**常见追问**

- HTTPS 会不会更慢？
- HTTPS 的证书作用是什么？
- 用了 HTTPS 就绝对安全吗？

**追问答案**

- 相比 HTTP 会多一些握手和加解密成本，但现代浏览器和协议优化后，这部分通常是可接受的。
- 证书用来证明服务端身份，避免你连接到伪造的网站。
- 不是。HTTPS 主要解决传输链路安全问题，但不等于业务逻辑、账号体系、前端代码本身就没有安全风险。

**容易答偏的地方**

- 只会说“HTTPS 更安全”
- 把 HTTPS 理解成一种全新协议，而不是 HTTP 上加安全层
- 误以为有 HTTPS 就不会有任何安全问题

### 4. HTTPS 为什么更安全？TLS 握手大致做了什么？

**简答版答案**

HTTPS 更安全，是因为它在正式传输数据前会通过 TLS 握手协商加密方式、校验证书并生成会话密钥。之后的数据传输会基于这个密钥加密，从而降低窃听和篡改风险。

**展开版答案**

这道题通常是在 HTTP/HTTPS 区别之后继续深挖的。

HTTPS 的安全性并不是“直接把内容加密一下”这么简单，而是要先解决三个问题：

- 你连的是不是真正的目标服务器
- 双方用什么算法加密
- 后续通信使用什么密钥

TLS 握手可以粗略理解成一次安全协商过程，大致会做这些事：

- 客户端发起请求
  - 告诉服务端自己支持哪些 TLS 版本和加密套件
- 服务端返回证书
  - 让客户端验证自己的身份
- 客户端验证证书
  - 看证书是否可信、域名是否匹配、是否过期
- 双方协商生成会话密钥
  - 后续真正传输数据时主要使用这个对称密钥加密

为什么通常说 HTTPS 同时用了非对称加密和对称加密？

- 非对称加密更适合解决密钥交换和身份校验问题
- 对称加密更适合高效地传输大量数据

所以 HTTPS 更安全，不是因为“全程都用最复杂的加密算法”，而是因为它在安全性和性能之间做了组合设计。

**关键点**

- TLS 握手是 HTTPS 安全链路的核心前置步骤
- 握手要解决身份校验、算法协商、密钥协商
- 实际数据传输通常更多依赖对称加密

**常见追问**

- 为什么不直接全程用非对称加密？
- 证书校验失败会怎么样？
- 中间人攻击和 HTTPS 的关系是什么？

**追问答案**

- 因为非对称加密成本更高，做大规模数据传输效率较低，所以更适合用来协商密钥而不是全程承担数据加密。
- 浏览器通常会给出安全警告，提示当前连接不可信。
- HTTPS 的重要目标之一就是降低中间人攻击风险，通过证书校验和加密通信来减少被冒充和被篡改的可能。

**容易答偏的地方**

- 把 TLS 握手讲成极细的规范流程，反而失去主线
- 只会说“加密了”，不知道先要做证书和密钥协商
- 把对称加密和非对称加密的作用说反

### 5. HTTP 常见状态码有哪些？分别代表什么？

**简答版答案**

HTTP 状态码是服务器对请求处理结果的数字化表达。常见可以按五类记：`1xx` 信息、`2xx` 成功、`3xx` 重定向、`4xx` 客户端错误、`5xx` 服务端错误。面试里最常用的是 `200`、`301`、`302`、`304`、`400`、`401`、`403`、`404`、`500`。

**展开版答案**

状态码本质上是在告诉客户端：“这次请求结果怎么样，接下来该怎么处理”。

按类别可以这样记：

- `1xx`
  - 请求处理中，较少单独考
- `2xx`
  - 请求成功
- `3xx`
  - 重定向相关
- `4xx`
  - 客户端请求有问题
- `5xx`
  - 服务端处理出错

面试里最常见的几个状态码通常包括：

- `200 OK`
  - 请求成功
- `204 No Content`
  - 请求成功但没有返回实体内容
- `301 Moved Permanently`
  - 永久重定向
- `302 Found`
  - 临时重定向
- `304 Not Modified`
  - 资源未修改，通常和缓存协商有关
- `400 Bad Request`
  - 请求参数或格式有问题
- `401 Unauthorized`
  - 通常表示未认证或认证失效
- `403 Forbidden`
  - 已经识别身份，但没有权限访问
- `404 Not Found`
  - 资源不存在
- `500 Internal Server Error`
  - 服务端内部错误
- `502 Bad Gateway`
  - 网关或代理拿到上游错误响应
- `503 Service Unavailable`
  - 服务暂时不可用

真正面试时，不需要把所有状态码都背全，但要把高频状态码的语义说准，尤其是：

- `301` 和 `302`
- `401` 和 `403`
- `304` 和缓存

**关键点**

- 状态码是服务端处理结果的标准表达
- 五大类要知道，但高频码更重要
- 缓存、重定向、认证权限类状态码最常被追问

**常见追问**

- `301` 和 `302` 有什么区别？
- `401` 和 `403` 有什么区别？
- `304` 为什么不算真正的页面内容响应？

**追问答案**

- `301` 是永久重定向，`302` 是临时重定向，语义和缓存策略上通常会有差异。
- `401` 更偏未认证或认证失败，`403` 更偏没有权限访问。
- 因为 `304` 的核心含义是“资源没变，可以继续用缓存”，它通常不会重新返回完整资源内容。

**容易答偏的地方**

- 只记数字，不知道实际语义
- 把 `401` 和 `403` 混为一谈
- 不知道 `304` 和缓存强相关

### 6. HTTP/1.1 和 HTTP/2 有什么区别？

**简答版答案**

HTTP/2 相比 HTTP/1.1 的核心优化包括二进制分帧、多路复用、头部压缩和服务器推送。它的目标主要是减少连接浪费、降低延迟、提升并发传输效率。

**展开版答案**

HTTP/1.1 已经比更早版本做了不少优化，比如长连接、缓存控制等，但它在高并发资源加载场景下仍然有明显限制。

HTTP/1.1 的常见问题包括：

- 一个连接里的请求处理方式较受限制
- 容易出现队头阻塞问题
- 请求头重复发送带来额外开销

HTTP/2 的几个核心改进可以这样理解：

- 二进制分帧
  - 不再像 HTTP/1.1 那样主要依赖文本格式传输
- 多路复用
  - 多个请求可以复用同一个连接并行传输
- 头部压缩
  - 减少重复请求头带来的传输开销
- 服务器推送
  - 服务端可以主动推送一些客户端可能马上会用到的资源

所以 HTTP/2 的优势，不是“语法更高级”，而是它更适合现代网页里大量静态资源并发加载的场景。

不过也要注意，HTTP/2 并不等于一定就没有一切阻塞问题，它更多是从协议层面对传输效率做了优化。

**关键点**

- HTTP/2 重点优化的是传输效率和并发能力
- 多路复用是最常被提到的核心特性
- 头部压缩和二进制分帧也很重要

**常见追问**

- 什么是多路复用？
- HTTP/2 解决了哪些 HTTP/1.1 的问题？
- HTTP/2 和 HTTPS 是必须绑定的吗？

**追问答案**

- 可以理解成多个请求共享一个连接并并行传输，而不是每个请求都独占连接。
- 主要缓解了连接复用效率低、请求头重复、并发资源加载成本高等问题。
- 协议层面不是绝对强绑定，但现实里浏览器环境通常和 HTTPS 一起使用得更多。

**容易答偏的地方**

- 只会背“多路复用”
- 把 HTTP/2 说成只是“更快”而说不清为什么
- 误以为 HTTP/2 就彻底消灭了所有阻塞问题

### 7. 浏览器缓存有哪些？强缓存和协商缓存有什么区别？

**简答版答案**

浏览器缓存可以大致理解为本地对资源的复用机制。强缓存命中时浏览器直接使用本地资源，不会发送请求；协商缓存会先向服务器确认资源是否变化，如果没变就返回 `304` 继续使用缓存。

**展开版答案**

缓存题本质上是在考：浏览器怎样减少重复请求、加快页面访问。

常见可以先从两种缓存策略讲：

- 强缓存
- 协商缓存

强缓存的特点是：

- 浏览器自己判断缓存是否仍然有效
- 如果有效，直接使用本地资源
- 通常不会发请求到服务器

常见相关响应头有：

- `Expires`
- `Cache-Control`

其中现代开发里更常用 `Cache-Control`，因为它控制更灵活。

协商缓存的特点是：

- 浏览器会带上缓存标识去问服务器资源是否有变化
- 如果资源没变，服务器返回 `304`
- 浏览器继续使用本地缓存

常见相关字段有：

- `Last-Modified` / `If-Modified-Since`
- `ETag` / `If-None-Match`

所以两者最本质的区别是：

- 强缓存：本地直接决定是否可用
- 协商缓存：服务器参与确认资源是否变化

面试里也很常问优先级问题，通常可以回答：

- 先看强缓存
- 强缓存没命中，再走协商缓存

**关键点**

- 缓存的目的是真正减少重复请求和提升加载速度
- 强缓存直接用本地，协商缓存要问服务器
- `Cache-Control`、`ETag`、`304` 是高频关键词

**常见追问**

- `Cache-Control` 和 `Expires` 有什么区别？
- `ETag` 和 `Last-Modified` 有什么区别？
- 强缓存和协商缓存谁优先？

**追问答案**

- `Expires` 更早期，依赖绝对时间；`Cache-Control` 更现代，控制粒度更灵活。
- `Last-Modified` 基于修改时间，`ETag` 更像资源内容标识，通常能更精确判断资源是否变化。
- 一般先判断强缓存，强缓存未命中时再走协商缓存。

**容易答偏的地方**

- 只会背字段名，不知道整体策略差异
- 把 `304` 说成强缓存命中
- 说不清为什么缓存能减少请求成本

### 8. Cookie、Session、Token 有什么区别？

**简答版答案**

Cookie 是浏览器端保存并可随请求自动携带的小数据；Session 更偏服务端会话机制；Token 是服务端签发给客户端的身份凭证，后续请求通常由客户端主动携带。它们经常一起出现，但不是同一个层面的概念。

**展开版答案**

这道题很容易被答成“Cookie 和 Token 谁更安全”，但更准确的讲法是先分清层次。

`Cookie` 是一种浏览器端存储和传输机制：

- 可以存少量数据
- 同域请求时会自动带上
- 常被用于保存会话标识

`Session` 更偏服务端概念：

- 服务端保存一份用户会话数据
- 客户端通常只拿一个会话标识
- 服务端根据这个标识找到对应会话

`Token` 更偏身份认证凭证：

- 用户登录成功后，服务端签发一个令牌给客户端
- 客户端后续请求再主动携带这个令牌
- 服务端再据此判断身份是否合法

所以一个常见组合是：

- 浏览器 Cookie 中存 SessionID
- 服务端用 SessionID 找到 Session

而 Token 方案下，则常见是：

- 客户端保存 Token
- 请求时放在请求头里发给服务端

它们的核心区别不在于“谁能不能登录”，而在于：

- 状态保存在服务端还是客户端
- 请求时凭证如何携带
- 系统如何校验用户身份

**关键点**

- Cookie 是浏览器侧机制
- Session 更偏服务端会话状态
- Token 更偏客户端持有的认证凭证
- 它们经常配合使用，不是互斥概念

**常见追问**

- Session 一定要依赖 Cookie 吗？
- Token 一定更适合前后端分离吗？
- 为什么 Cookie 会被拿来做登录态？

**追问答案**

- 不一定，但 Cookie 是非常常见的承载方式，因为它能自动随请求发送。
- 很多前后端分离场景确实常用 Token，但具体方案还是要结合系统架构和安全要求决定。
- 因为它能稳定保存少量标识信息，并在请求时自动携带，适合做会话标识传递。

**容易答偏的地方**

- 把 Cookie、Session、Token 说成三个并列同级概念
- 一上来就讨论谁更安全，却说不清它们分别是什么
- 把 Session 理解成浏览器本地存储

### 9. Cookie、localStorage、sessionStorage、IndexedDB 有什么区别？

**简答版答案**

这几种都是浏览器侧常见的数据存储方式，但它们在容量、生命周期、是否自动随请求发送、适合的数据类型上差别很大。面试里通常要先分清“谁负责请求身份携带，谁负责本地持久化，谁适合大规模结构化数据”。

**展开版答案**

这道题和 JS 里的本地存储题有重合，但在浏览器原理里更适合从“浏览器如何保存数据”这个角度讲。

可以这样区分：

- `Cookie`
  - 存储量较小
  - 可设置过期时间
  - 会随着同域请求自动发送
  - 更适合保存少量标识类信息
- `localStorage`
  - 持久化存储
  - 不手动清理通常会一直保留
  - 不会自动随请求发送
  - 更适合前端长期本地配置
- `sessionStorage`
  - 生命周期通常跟当前标签页会话绑定
  - 标签页关闭后通常清空
  - 不会自动随请求发送
  - 更适合临时页面状态
- `IndexedDB`
  - 适合更复杂、更大规模的结构化数据
  - 支持索引、事务等能力
  - 更像浏览器里的本地数据库

如果从浏览器行为角度看，最重要的区别通常有四个：

- 是否自动参与请求
- 生命周期多长
- 容量有多大
- 能否胜任复杂数据存储

这也是为什么很多登录态问题会引出存储选择：不是看“哪个 API 用起来方便”，而是看数据是否敏感、是否要自动带给服务端、是否要长期保存。

**关键点**

- `Cookie` 的特殊点是会自动随请求发送
- `localStorage` 和 `sessionStorage` 更偏前端本地数据
- `IndexedDB` 更适合复杂结构化和大数据量场景
- 存储方式选择要结合安全性和使用场景

**常见追问**

- 为什么不建议把所有数据都放在 `localStorage`？
- 登录态放哪更合适？
- `sessionStorage` 和 `session` 是一回事吗？

**追问答案**

- 因为它更适合轻量本地数据，而且是浏览器侧同步 API，大量复杂数据并不合适。
- 没有绝对答案，要结合安全策略、服务端方案和业务场景综合决定。
- 不是。`sessionStorage` 是浏览器存储，`Session` 是服务端会话机制，它们不是一个概念。

**容易答偏的地方**

- 把浏览器存储和服务端会话机制混为一谈
- 只从容量大小回答，不谈请求行为和生命周期
- 把 `sessionStorage` 和 `Session` 误认为同一件事

### 10. 什么是同源策略？为什么浏览器需要同源策略？

**简答版答案**

同源策略是浏览器的一种安全限制，用来约束不同源之间的资源访问权限。所谓同源，通常要求协议、域名、端口都相同。它的核心目的是防止一个网站随意读取或操作另一个网站的敏感数据。

**展开版答案**

同源策略本质上是在回答一个安全问题：如果浏览器完全不做限制，会发生什么？

假设你登录了银行网站，同时又打开了一个恶意网站。如果浏览器允许后者随便读取前者的页面数据、Cookie、接口返回结果，那风险会非常大。

所以浏览器引入了同源策略，用来限制：

- 不同源页面之间的 DOM 访问
- 某些接口响应内容的读取
- 某些存储数据的共享访问

这里的“同源”通常指三个维度都相同：

- 协议相同
- 域名相同
- 端口相同

只要其中一个不同，通常就不算同源。

面试里回答这个问题时，重点不要只背定义，更要补一句：同源策略不是为了让开发更麻烦，而是为了保护用户在不同网站上的数据隔离。

**关键点**

- 同源策略是浏览器安全边界的一部分
- 同源判断通常看协议、域名、端口
- 它的核心价值是保护用户数据隔离

**常见追问**

- 图片、脚本、样式为什么很多时候又能跨域加载？
- 同源策略主要限制的是什么？
- 不同子域名之间算同源吗？

**追问答案**

- 因为浏览器对“资源加载”和“读取返回内容”这两件事的限制程度不完全一样，很多时候是能加载但不能随意读取结果。
- 重点是限制跨源读取敏感内容和跨源操作某些对象能力。
- 一般不算，因为域名不同就已经不满足同源条件。

**容易答偏的地方**

- 只会背“协议、域名、端口”
- 不知道同源策略的安全目的
- 说成“不同源之间完全什么都做不了”

### 11. 什么是跨域？常见跨域解决方案有哪些？

**简答版答案**

跨域本质上是浏览器里的跨源访问问题，不是服务器之间不能通信。常见解决方案包括 CORS、JSONP、反向代理和 `postMessage` 等，具体要看你是在解决接口访问、页面通信还是资源代理。

**展开版答案**

很多人第一次学跨域时会误以为“请求发不出去”，其实更准确的说法是：请求通常能发出去，但浏览器不一定允许前端脚本读取结果。

所以跨域问题本质上是浏览器基于同源策略做出的限制。

常见解决方式可以按场景理解：

- CORS
  - 最主流的跨域接口方案
  - 服务端通过响应头明确告诉浏览器是否允许跨域访问
- JSONP
  - 利用 `script` 标签不受同源限制的特点
  - 只适用于 GET 请求
  - 现在更多是历史面试题
- 反向代理
  - 由同源服务器转发请求
  - 前端看起来还是请求同源地址
- `postMessage`
  - 更适合不同窗口、iframe 之间通信

所以回答这道题时，最好先讲一句：跨域是浏览器限制，不是后端和后端之间的物理通信障碍。

**关键点**

- 跨域问题来自浏览器同源策略
- CORS 是最常见的现代解决方案
- JSONP 更偏历史方案
- 不同跨域方案解决的问题类型并不完全一样

**常见追问**

- CORS 是前端配还是后端配？
- JSONP 为什么只能 GET？
- 开发环境常见的代理方案本质是什么？

**追问答案**

- 主要是服务端通过响应头控制，浏览器再根据这些头决定是否放行前端读取结果。
- 因为它本质上依赖 `script` 标签加载资源，而不是标准 AJAX 请求。
- 本质上是让前端请求先打到同源开发服务器，再由它转发到目标服务。

**容易答偏的地方**

- 说成“跨域请求根本发不出去”
- 把 JSONP 当成现代接口跨域首选
- 不区分接口跨域和页面通信场景

### 12. 浏览器渲染流程是怎样的？DOM、CSSOM、Render Tree 分别是什么？

**简答版答案**

浏览器拿到 HTML 和 CSS 后，会先解析 HTML 构建 DOM，解析 CSS 构建 CSSOM，然后把两者结合生成 Render Tree，再进行布局和绘制。DOM 是内容结构，CSSOM 是样式规则，Render Tree 是真正参与渲染的结构。

**展开版答案**

这道题是浏览器渲染机制的核心题。

可以先把几个概念分清：

- DOM
  - HTML 解析后形成的文档结构树
- CSSOM
  - CSS 解析后形成的样式对象模型
- Render Tree
  - DOM 和 CSSOM 结合后得到的渲染树
  - 只保留真正需要渲染的节点和对应样式信息

渲染流程大致可以理解成：

1. 浏览器解析 HTML，构建 DOM
2. 浏览器解析 CSS，构建 CSSOM
3. 将 DOM 和 CSSOM 合并，生成 Render Tree
4. 根据 Render Tree 进行布局
5. 根据布局结果进行绘制
6. 必要时进行分层和合成显示

这里有两个高频点：

- DOM 不等于最终渲染结构
- 某些节点可能在 DOM 里存在，但不一定出现在 Render Tree 中

比如 `display: none` 的元素，通常不会进入最终渲染树。

另外，浏览器渲染流程也不是“整份 HTML 全部解析完，再统一渲染一次”。
更准确地说，它通常是流式解析、增量渲染：

- HTML 往往是一边下载一边解析
- 能先构建的 DOM 会先构建
- 对应的 CSS 准备好后，能渲染的部分会先尝试渲染
- 后面如果又来了新的 HTML、CSS，或者 JS 改了 DOM / 样式，相关渲染步骤还会继续局部重复

所以“浏览器一边解析一边渲染”的意思，不是所有步骤永远只做一次，而是浏览器会随着资源到达和页面变化，持续推进并按需重复后续渲染流程。

面试里如果能把“结构、样式、渲染”三层关系讲清楚，基本就已经很稳了。

**关键点**

- DOM 是结构，CSSOM 是样式，Render Tree 是渲染结果基础
- 渲染流程核心是解析、合成、布局、绘制
- 不是所有 DOM 节点都会进入 Render Tree

**常见追问**

- 为什么 CSS 会影响页面首次渲染？
- `display: none` 和 `visibility: hidden` 在渲染上有什么区别？
- JS 为什么可能阻塞页面渲染？
- 为什么说浏览器是一边解析一边渲染？

**追问答案**

- 因为浏览器构建渲染树时需要样式信息，CSS 未准备好会影响渲染结果确定。
- `display: none` 的元素通常不会进入渲染树，`visibility: hidden` 的元素仍可能参与布局，只是不可见。
- 因为 JS 可能修改 DOM 或样式，浏览器往往要在执行脚本后再决定后续解析和渲染策略。
- 因为浏览器通常采用流式解析和增量渲染，拿到一部分 HTML 和对应样式后就会先尝试构建和渲染这部分内容；后面如果新内容到来，或样式、脚本改变了已有结果，相关布局和绘制流程还会继续局部执行。

**容易答偏的地方**

- 把 DOM、CSSOM、Render Tree 混成一个东西
- 只会背流程，不知道每一步产物是什么
- 不知道 `display: none` 和渲染树的关系

### 13. 回流和重绘有什么区别？为什么操作 DOM 会影响性能？

**简答版答案**

回流是布局信息发生变化，浏览器需要重新计算元素位置和尺寸；重绘是布局不变但外观变化，浏览器只需要重新绘制。因为 DOM 和样式变更可能触发这两类工作，所以频繁操作 DOM 会带来性能开销，尤其是回流成本更高。

**展开版答案**

回流和重绘本质上是在考浏览器渲染的成本。

可以这样理解：

- 回流
  - 元素几何属性或布局关系变了
  - 浏览器需要重新计算布局
- 重绘
  - 布局不变，只是视觉表现变了
  - 浏览器重新绘制内容

常见容易触发回流的操作包括：

- 修改宽高
- 改变位置
- 增删节点
- 改变字体大小
- 读写布局信息交替进行

而像颜色、背景色、阴影等修改，很多时候更偏向重绘。

为什么说操作 DOM 可能影响性能？

因为 DOM 变更往往不会只影响一个点，它可能让浏览器重新走样式计算、布局、绘制甚至合成这整条链路。

所以真正的优化重点不只是“少操作 DOM”，而是：

- 避免频繁、零碎地改
- 尽量批量更新
- 尽量少改高成本布局属性

**关键点**

- 回流比重绘代价通常更高
- DOM 变更可能引发样式、布局、绘制一整链工作
- 性能优化关键是减少高成本布局变更

**常见追问**

- 哪些操作容易触发回流？
- 为什么读取布局信息有时也会影响性能？
- 为什么 `transform` 和 `opacity` 常被认为更适合做动画？

**追问答案**

- 修改几何属性、增删节点、改变布局结构等都容易触发回流。
- 因为浏览器可能需要先把之前的样式和布局计算结果同步刷新出来，再返回准确值。
- 因为它们通常不直接改变文档流布局，更容易减少高成本布局计算。

**容易答偏的地方**

- 只会说“回流比重绘更耗性能”
- 不知道具体是什么操作触发了什么代价
- 把所有页面变化都笼统地叫“重绘”

### 14. 浏览器有哪些主要线程？它们之间大致如何配合？

**简答版答案**

浏览器里并不是只有一个线程在工作。常见可以从主线程、GUI 渲染相关线程、JS 引擎执行、定时器、事件触发、网络请求等协作关系来理解。面试里重点不是背完所有线程名，而是讲清“为什么 JS 不能一边改 DOM 一边让渲染线程随便并行处理”。

**展开版答案**

浏览器原理里常见会说“浏览器是多线程协作的”，但这并不代表前端代码能随意并行改页面。

从面试角度，常见可以这样理解几个角色：

- GUI 渲染相关线程
  - 负责页面渲染、布局、绘制等工作
- JS 引擎线程
  - 负责执行 JavaScript 代码
- 事件触发相关线程
  - 某些事件准备好后，会把回调推进任务队列等待主线程处理
- 定时器相关线程
  - 计时结束后通知任务进入队列
- 网络请求相关线程
  - 请求完成后再通知对应回调进入调度流程

为什么经常说 JS 引擎线程和 GUI 渲染线程不能同时随意工作？

因为 JavaScript 可能直接操作 DOM、修改样式，如果渲染线程也同时改同一份页面状态，就容易出现结果不一致问题。

所以浏览器需要在调度上做协调：

- JS 执行时，可能影响渲染节奏
- 异步任务准备好后，不是立刻打断主线程，而是等待调度
- 页面渲染也要选择合适时机进行

这也是为什么前端很多性能问题，最终都会落到一个核心上：

- 主线程太忙
- 脚本执行太重
- 页面更新太频繁

所以这道题里最有价值的回答，不是罗列很多线程名字，而是讲清：

- 浏览器内部是协作模型
- JS、渲染、事件、网络不是一股脑同时乱跑
- 主线程调度压力会直接影响页面流畅度

**关键点**

- 浏览器内部有多种线程协作
- JavaScript 执行和页面渲染之间需要协调
- 主线程压力大时会直接影响交互和渲染表现

**常见追问**

- 为什么 JavaScript 执行会影响页面渲染？
- 定时器时间到了为什么不会立刻执行？
- 浏览器线程模型和事件循环有什么关系？

**追问答案**

- 因为 JS 可能操作 DOM 和样式，浏览器需要协调脚本执行和渲染时机，避免状态冲突。
- 因为时间到了只是说明任务可以进入调度等待，真正执行还要等主线程空闲并轮到它。
- 线程模型更偏浏览器内部角色分工，事件循环更偏主线程上的任务调度机制，两者相关但不是一个概念。

**容易答偏的地方**

- 只会死记线程名字，不知道它们为什么要分工
- 误以为浏览器里所有线程都能直接并行改 DOM
- 把线程模型和事件循环完全说成同一个概念
