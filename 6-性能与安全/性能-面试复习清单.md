# 性能面试复习清单

## 使用方式

- 先看问题，尝试自己回答
- 再对照简答版和展开版检查表达是否完整
- 最后补常见追问和容易答偏的地方

### 1. 什么是前端性能优化？面试里应该按什么框架来回答性能问题？

**简答版答案**

前端性能优化不是单点技巧堆砌，而是围绕用户体验去降低加载成本、提升交互流畅度和保证结果可验证。面试里比较稳的答法通常是按“发现问题、分析原因、优化方案、验证结果”这条主线来讲。

**展开版答案**

很多人一听到性能优化，就会开始罗列各种手段，比如压缩图片、开启缓存、代码分割、懒加载。但面试官真正想听的，通常不是你背了多少优化点，而是你是否建立了完整的性能思维。

更准确地说，前端性能优化是在不同阶段持续降低用户等待时间和运行卡顿感，核心目标一般包括：

- 更快看到内容
- 更快可交互
- 更稳定的页面布局
- 更流畅的交互反馈

所以性能优化不只是“让包更小”，还包括：

- 加载层面的优化
- 渲染层面的优化
- 运行时计算优化
- 监控和验证机制

面试里推荐按下面的框架去回答：

1. 先说怎么发现问题，比如看指标、看监控、看 DevTools
2. 再说问题出在哪，比如资源大、请求慢、阻塞渲染、长任务过多
3. 再说采取了什么优化方案
4. 最后说如何验证优化是否真的有效

这种表达方式的好处是，它比单纯背技术名词更有工程感，也更贴近真实项目。

**关键点**

- 性能优化的核心是用户体验，不只是包体积
- 回答时要有“发现 -> 分析 -> 优化 -> 验证”的闭环
- 性能问题通常要结合阶段和场景去看

**常见追问**

- 性能优化和用户体验是什么关系？
- 为什么说性能题不能只背优化手段？
- 项目里做性能优化时，最先该看什么？

**追问答案**

- 用户真正感受到的是打开快不快、卡不卡、布局稳不稳，所以性能优化本质上服务的是体验。
- 因为很多优化方案是有前提和取舍的，只有知道问题在哪，优化手段才有意义。
- 通常先看性能指标、监控数据和用户真实反馈，先定位主要瓶颈，再决定从哪里下手。

**容易答偏的地方**

- 一上来就背一堆优化点，没有主线
- 把性能优化只理解成“压缩资源”
- 不提验证结果，只说“我做过优化”

### 2. 前端性能优化一般分为哪些阶段？比如加载前、加载中、运行时、上线后分别优化什么？

**简答版答案**

前端性能优化通常可以按阶段拆成加载前、加载中、运行时和上线后监控四段。不同阶段关注点不同，前面更偏资源体积和请求链路，中间偏渲染与可交互，后面偏运行时成本和数据验证。

**展开版答案**

面试里如果被问“性能优化有哪些方向”，直接平铺一堆点很容易乱。更好的方法是按时间阶段来整理。

加载前，重点在于让资源更容易被浏览器拿到和处理：

- 资源压缩
- 图片格式优化
- 代码分割
- 缓存策略
- CDN

加载中，重点在于减少阻塞，尽快让页面能看到内容：

- 首屏关键资源优先加载
- 非关键资源延后加载
- 懒加载
- 预加载和预连接
- 减少阻塞渲染的 CSS / JS

运行时，重点在于页面渲染和交互是否流畅：

- 减少重绘回流
- 避免长任务阻塞主线程
- 节流和防抖
- 合理使用 Web Worker
- 控制频繁计算和不必要渲染

上线后，重点在于监控和验证：

- 采集性能指标
- 做异常上报
- 结合真实用户监控分析问题
- 对比优化前后的结果

这种分阶段表达的价值在于，面试官会觉得你对性能优化有系统认知，而不是记了几个零散技巧。

**关键点**

- 性能优化要按阶段理解，不要混着讲
- 加载前和加载中更偏资源与网络
- 运行时更偏渲染和主线程
- 上线后要靠监控验证

**常见追问**

- 为什么上线后监控也属于性能优化的一部分？
- 哪个阶段最重要？
- 怎么区分加载问题和运行时问题？

**追问答案**

- 因为没有监控和验证，就很难知道优化是否真的生效，也无法持续发现新问题。
- 没有绝对最重要，要看当前瓶颈在哪。首屏慢时先看加载链路，交互卡顿时先看运行时。
- 加载问题通常表现为打开慢、资源下载久；运行时问题更常表现为滚动卡、点击延迟、动画不流畅。

**容易答偏的地方**

- 把所有优化点混在一起
- 完全不提上线后的监控和验证
- 说不清不同阶段各自在解决什么问题

### 3. 什么是首屏优化？影响首屏速度的核心因素有哪些？

**简答版答案**

首屏优化就是让用户尽快看到页面主要内容，并尽快进入可交互状态。影响首屏速度的核心因素通常包括网络请求链路、资源体积、阻塞渲染资源、接口响应速度和客户端渲染成本。

**展开版答案**

首屏优化是性能题里最常见的一道题。它的本质不是“页面完全加载完”，而是用户什么时候能尽快看到有价值内容。

通常可以从这几个方面看首屏速度：

- HTML 到达得快不快
- CSS 是否阻塞渲染
- JS 是否过大、过早执行
- 首屏接口请求是否慢
- 图片和字体是否拖慢渲染
- 渲染本身是否过重

常见首屏优化思路有：

- 缩小首屏资源体积
- 非首屏模块延迟加载
- 关键资源优先加载
- 图片懒加载
- SSR 或预渲染
- 合理使用缓存和 CDN
- 减少首屏接口依赖链

面试里回答首屏优化，最好把“网络层、资源层、渲染层、接口层”一起提到，这样会更完整。

**关键点**

- 首屏优化重点是尽快看到核心内容
- 不等于等所有资源都加载完
- 影响首屏的因素往往是多层叠加的

**常见追问**

- 首屏优化和白屏优化有什么关系？
- CSR、SSR、预渲染对首屏有什么影响？
- 首屏优化怎么验证？

**追问答案**

- 白屏优化更偏“页面尽快有内容出现”，首屏优化范围更大，还会关心内容完整性和可交互时间。
- SSR 和预渲染通常能让首屏 HTML 更早可见，但也会带来服务端成本和工程复杂度。
- 可以看 FCP、LCP、接口耗时、首屏资源加载时长，也可以结合真实用户监控验证。

**容易答偏的地方**

- 只会说懒加载
- 把首屏优化理解成“把所有代码都压缩一下”
- 不提接口和渲染成本

### 4. 懒加载、预加载、预解析、预连接分别是什么？它们的适用场景有什么区别？

**简答版答案**

懒加载是“先不加载，等需要时再加载”；预加载是“后面会用到，所以提前加载”；预解析和预连接更偏网络层优化，分别用于提前解析域名和提前建立连接。它们的共同点都是优化加载体验，但切入点不同。

**展开版答案**

这几个概念很容易混，因为它们都和“资源什么时候加载”有关。

懒加载的重点是延后：

- 当前不需要的资源先不加载
- 典型场景是长列表图片、路由组件、非首屏模块

预加载的重点是提前：

- 明确知道某个资源马上会用到
- 提前下载，提高后续使用时机的响应速度

预解析通常指 DNS 预解析：

- 提前把域名解析成 IP
- 减少真正发请求时的等待

预连接则更进一步：

- 提前完成 DNS、TCP、TLS 等连接准备
- 适合即将访问的第三方域名或重要资源域名

所以可以简单记：

- 懒加载：延后做
- 预加载：提前下
- 预解析：提前找地址
- 预连接：提前把路打通

**关键点**

- 懒加载强调减少当前负担
- 预加载强调为即将使用的资源抢时间
- 预解析和预连接更偏网络准备

**常见追问**

- 图片懒加载一般怎么做？
- 预加载是不是越多越好？
- 预连接和预解析有什么区别？

**追问答案**

- 常见做法是结合 `loading="lazy"`、Intersection Observer 或框架路由懒加载能力。
- 不是。预加载太多会抢占带宽，反而影响真正关键资源。
- 预解析只提前做域名解析，预连接会继续提前建立更完整的网络连接。

**容易答偏的地方**

- 把懒加载和预加载说反
- 以为预加载越多越好
- 不知道预解析和预连接更偏网络层

### 5. 资源压缩和资源合并分别解决什么问题？在 HTTP/1.1 和 HTTP/2 下思路有什么变化？

**简答版答案**

资源压缩主要解决文件体积大、传输慢的问题；资源合并主要解决请求数量过多带来的开销。在 HTTP/1.1 下资源合并更常见，而在 HTTP/2 下多路复用更强，优化重点通常更偏合理拆分而不是机械合并。

**展开版答案**

压缩和合并都能提升加载性能，但它们解决的问题不是一回事。

资源压缩主要是把文件体积变小，比如：

- JS 压缩
- CSS 压缩
- Gzip / Brotli
- 图片压缩

它解决的是“单个资源传输成本高”的问题。

资源合并更像是减少请求次数，比如：

- 把多个脚本打进一个 chunk
- CSS 合并
- 雪碧图

它解决的是“请求数太多，连接和调度开销高”的问题。

在 HTTP/1.1 下，浏览器并发连接能力有限，很多小请求会更容易拖慢加载，所以资源合并常常比较有效。

但到了 HTTP/2，由于有多路复用，请求数的成本相对下降，所以优化思路会变化：

- 不再追求文件越少越好
- 更重视合理拆分和缓存复用
- 让首屏关键资源更早到达

所以现代项目里更常见的思路是：压缩仍然非常重要，但合并不能机械做，要结合协议、缓存和代码分割一起看。

**关键点**

- 压缩看体积，合并看请求数
- HTTP/1.1 更强调减少请求
- HTTP/2 更强调合理拆分和优先级

**常见追问**

- 为什么 HTTP/2 下不再强调“文件越少越好”？
- Brotli 和 Gzip 有什么关系？
- 雪碧图现在还常用吗？

**追问答案**

- 因为 HTTP/2 支持多路复用，请求数量本身的成本下降了，过度合并反而可能影响缓存和按需加载。
- 它们都是压缩方案，Brotli 在很多场景下压缩率更高，但也要看服务端和浏览器支持。
- 有些场景还会用，但没有以前那么高频了，因为现代协议、字体图标、SVG 和构建工具已经提供了更多替代方案。

**容易答偏的地方**

- 把压缩和合并当成一回事
- 还在机械强调“请求越少越好”
- 不知道协议升级会影响优化策略

### 6. 浏览器缓存有哪些常见策略？强缓存和协商缓存有什么区别？

**简答版答案**

浏览器缓存常见分为强缓存和协商缓存。强缓存命中时浏览器直接用本地资源，不会向服务器发起实际内容请求；协商缓存会先问服务器资源是否变化，没变就继续用本地缓存。

**展开版答案**

缓存是性能优化里非常高频的一条线，因为它直接影响用户是否需要重新下载资源。

强缓存常见依赖这些响应头：

- `Cache-Control`
- `Expires`

命中强缓存时，浏览器会直接使用本地缓存内容，通常不会重新请求资源实体，所以速度更快。

协商缓存常见依赖：

- `Last-Modified` / `If-Modified-Since`
- `ETag` / `If-None-Match`

协商缓存不会直接盲目使用本地缓存，而是先和服务器确认资源是否变化。如果没变化，服务器返回 `304 Not Modified`，浏览器再继续使用本地资源。

所以两者区别可以简单总结为：

- 强缓存：直接用，不问服务器
- 协商缓存：先问一下，再决定用不用

实际项目里，这两者通常是配合使用的，而不是二选一。

**关键点**

- 强缓存优先级通常更高
- 强缓存命中时请求成本最低
- 协商缓存更适合需要确认是否更新的资源

**常见追问**

- `Cache-Control` 和 `Expires` 有什么区别？
- `ETag` 和 `Last-Modified` 有什么区别？
- 为什么静态资源常配长缓存？

**追问答案**

- `Expires` 是绝对时间，`Cache-Control` 更现代、更灵活，通常优先参考后者。
- `Last-Modified` 基于修改时间，`ETag` 更像资源内容标识，通常更精细。
- 因为带 hash 的静态资源一旦内容变化，文件名通常也会变，适合配合长缓存提升复用率。

**容易答偏的地方**

- 把强缓存和协商缓存说成互斥关系
- 只会背响应头名字，不会说使用场景
- 不知道缓存策略要结合文件 hash

### 7. 图片优化有哪些常见手段？如何在质量、体积和加载体验之间做取舍？

**简答版答案**

图片优化常见手段包括选择合适格式、压缩体积、按尺寸输出、懒加载、占位图和 CDN 处理。真正难点不是“把图压小”，而是在清晰度、体积和体验之间做平衡。

**展开版答案**

图片往往是页面里最重的一类资源，所以优化空间很大。

常见手段包括：

- 选择合适格式，比如 `JPEG`、`PNG`、`WebP`、`AVIF`
- 压缩图片体积
- 按需输出尺寸，避免大图小用
- 使用响应式图片
- 懒加载非首屏图片
- 使用占位图、模糊图、骨架图提升感知体验
- 配合 CDN 做图片裁剪和转换

图片优化的难点不只是技术手段，而是取舍。
比如商品图、Banner 图、头像图，对清晰度和加载速度的要求就不完全一样。

面试里更好的回答方式是：先说方法，再补一句要结合业务场景权衡，比如首屏主图不能一味压缩到模糊，但长列表缩略图就更适合优先压体积。

**关键点**

- 图片优化不仅看体积，还看感知体验
- 格式选择和尺寸控制非常关键
- 非首屏图片更适合懒加载

**常见追问**

- `WebP` 和 `AVIF` 有什么价值？
- 为什么要避免大图小用？
- 图片优化怎么和首屏优化联动？

**追问答案**

- 它们通常能在保证质量的同时提供更好的压缩率，但要看兼容性和解码成本。
- 因为用户实际只展示很小区域，却下载了很大的资源，带宽和解码成本都会浪费。
- 首屏图片往往是 LCP 的重要候选资源，所以要重点关注加载优先级、尺寸和格式。

**容易答偏的地方**

- 只会说“压缩图片”
- 不知道图片格式和尺寸本身就是优化点
- 不提感知体验和业务取舍

### 8. 代码分割是什么？为什么它能优化性能？常见场景有哪些？

**简答版答案**

代码分割就是把原本一个大包拆成多个更小的 chunk，在不同场景按需加载。它能优化性能，核心原因是减少首屏下载和解析执行成本，让用户先拿到当前真正需要的代码。

**展开版答案**

如果所有页面和所有功能代码都打进一个大包，用户首次访问时就要一次性下载、解析和执行很多暂时用不到的代码，这会直接影响首屏。

代码分割的核心价值在于：

- 首屏只加载当前必须的代码
- 非首屏代码延后加载
- 提高缓存利用率

常见场景包括：

- 路由级拆分
- 组件级动态加载
- 第三方大库单独拆包
- 公共代码提取

为什么它能提升性能？
因为前端性能不只是下载，还包括解析和执行。包更合理地拆分后，不仅下载压力更小，主线程的解析执行成本也更可控。

**关键点**

- 代码分割的核心是按需加载
- 它优化的不只是下载体积，还有解析执行成本
- 典型场景是路由拆分和大模块延迟加载

**常见追问**

- 代码分割会不会导致请求变多？
- 什么场景不适合过度拆分？
- 公共代码抽离有什么价值？

**追问答案**

- 会，所以要平衡。拆得过细可能请求太多，拆得过粗又失去按需意义。
- 首屏强依赖的小模块、非常频繁共享的小片段，不一定适合过度拆分。
- 公共代码抽离有利于缓存复用，避免多个页面重复下载同一份逻辑。

**容易答偏的地方**

- 只会说“拆包”
- 不知道解析和执行成本也会影响性能
- 把代码拆得越细越好当成绝对原则

### 9. Tree Shaking、按需加载、动态导入分别解决什么问题？它们之间有什么关系？

**简答版答案**

Tree Shaking 解决的是“没用到的代码别进产物”；按需加载解决的是“当前不需要的代码别现在加载”；动态导入是实现按需加载的一种常见方式。它们都和“减少无效代码成本”有关，但作用阶段不同。

**展开版答案**

这三个概念经常一起出现，但不是同一层面的东西。

Tree Shaking 更偏构建阶段：

- 基于模块引用关系
- 移除没有被使用的导出
- 目标是让最终产物更干净

按需加载更偏运行时加载策略：

- 当前不用的内容先不加载
- 需要时再加载
- 目标是降低首屏负担

动态导入是实现按需加载的一种技术手段，比如：

```js
import('./detail.js')
```

它通常会让构建工具把相关代码拆成独立 chunk，等真正执行到这里时再加载。

所以它们的关系可以这样记：

- Tree Shaking：删没用的
- 按需加载：晚点再加载
- 动态导入：实现晚点加载的一种方式

**关键点**

- 三者目标相似，但作用阶段不同
- Tree Shaking 偏构建裁剪
- 动态导入常用于实现按需加载

**常见追问**

- Tree Shaking 为什么依赖 ES Module？
- 动态导入一定能优化性能吗？
- 按需加载和代码分割是什么关系？

**追问答案**

- 因为 ES Module 的依赖关系更静态，更适合构建阶段分析哪些导出被使用了。
- 不一定。如果拆分不合理，可能带来额外请求和体验抖动，所以要结合场景使用。
- 代码分割通常是按需加载的前提之一，先拆出来，后面才有机会按需取用。

**容易答偏的地方**

- 把三者当成同义词
- 不知道 Tree Shaking 发生在构建阶段
- 误以为只要动态导入就一定更快

### 10. 常见性能指标有哪些？`FCP`、`LCP`、`CLS`、`INP` 分别代表什么？

**简答版答案**

常见性能指标包括加载、渲染、交互和稳定性几类。`FCP` 关注第一次有内容绘制，`LCP` 关注主要内容何时可见，`CLS` 关注布局稳定性，`INP` 关注交互响应质量。

**展开版答案**

现代性能优化不能只说“感觉变快了”，而要尽量落到指标。

常见指标可以按维度理解：

- 加载相关
- 渲染相关
- 交互相关
- 稳定性相关

其中高频指标包括：

- `FCP`：First Contentful Paint，首次内容绘制
- `LCP`：Largest Contentful Paint，最大内容绘制
- `CLS`：Cumulative Layout Shift，累计布局偏移
- `INP`：Interaction to Next Paint，交互到下一次绘制

它们分别对应：

- 用户什么时候第一次看到内容
- 页面主要内容什么时候出来
- 页面会不会乱跳
- 用户交互后页面反应快不快

所以这些指标本质上是在量化用户体验，而不是单纯看技术实现。

**关键点**

- 指标是性能优化的重要依据
- `LCP` 往往和首屏关键内容关系很强
- `CLS` 看布局稳定性
- `INP` 更关注真实交互体验

**常见追问**

- 为什么不能只看一个指标？
- `LCP` 常受哪些因素影响？
- 为什么布局抖动也是性能问题？

**追问答案**

- 因为性能是多维度体验，只看一个指标容易误判，比如首屏快但交互很卡，仍然是问题。
- 大图、主内容资源、接口数据、阻塞渲染资源都可能影响 `LCP`。
- 因为布局突然跳动会直接影响用户体验和操作稳定性，所以 `CLS` 很重要。

**容易答偏的地方**

- 只会背缩写
- 不知道这些指标对应什么用户体验
- 把性能指标理解成只有加载时间

### 11. 如何定位前端性能问题？常用的分析工具和排查思路有哪些？

**简答版答案**

定位前端性能问题一般要结合浏览器 DevTools、Lighthouse、Performance 面板、Network 面板和线上监控数据一起看。思路上通常是先看现象和指标，再拆到网络、资源、渲染和运行时逐层定位。

**展开版答案**

性能优化最怕的不是不会优化，而是没有定位思路。

比较常见的排查方式是：

1. 先明确问题现象，比如打开慢、接口慢、滚动卡、点击延迟
2. 再看指标和监控数据
3. 再拆到具体层面去排查

常用工具包括：

- Chrome DevTools
- Network 面板
- Performance 面板
- Lighthouse
- Web Vitals
- 线上监控平台

如果是加载慢，重点看：

- 请求时长
- 资源体积
- 缓存命中
- 阻塞资源

如果是运行时卡顿，重点看：

- 长任务
- 主线程占用
- 重绘回流
- JS 执行热点

所以排查思路最好是分层，而不是一上来就猜“是不是图片太大”。

**关键点**

- 先看现象和指标，再看原因
- 工具只是辅助，关键是排查思路
- 要区分加载问题和运行时问题

**常见追问**

- DevTools 的 Performance 面板主要看什么？
- Lighthouse 适合做什么？
- 为什么线上监控和本地调试都需要？

**追问答案**

- 主要看主线程执行、渲染、长任务、帧率和各阶段耗时。
- 更适合做页面质量和性能的整体评估，帮助快速发现明显问题。
- 因为本地环境和真实用户环境差异很大，线上数据更接近真实场景。

**容易答偏的地方**

- 只会说“用 Lighthouse 看一下”
- 不知道加载问题和运行时问题的排查入口不同
- 完全不提线上真实数据

### 12. 重绘和回流是什么？为什么它们会影响页面性能？常见优化方式有哪些？

**简答版答案**

重绘是元素样式变化但不影响布局时，浏览器重新绘制；回流是布局相关属性变化导致浏览器重新计算几何信息。回流通常成本更高，因为它往往会引发后续绘制甚至影响更多节点。

**展开版答案**

回流和重绘是前端性能题里很经典的一组概念。

重绘一般发生在：

- 颜色变化
- 背景变化
- 可见性等不影响布局的样式变化

回流通常发生在：

- 宽高变化
- 位置变化
- DOM 结构变动
- 读取某些布局属性触发强制同步布局

为什么它们会影响性能？
因为浏览器需要重新计算布局、重新绘制，严重时还会阻塞主线程，导致卡顿。

优化思路常见有：

- 减少频繁修改布局属性
- 批量操作 DOM
- 避免频繁读写交替触发布局抖动
- 对动画优先使用 `transform`、`opacity`
- 合理使用节流和 `requestAnimationFrame`

**关键点**

- 回流通常比重绘成本更高
- 布局相关变化更容易引发回流
- 频繁读写 DOM 容易造成性能问题

**常见追问**

- 为什么 `transform` 通常更适合做动画？
- 什么是布局抖动？
- 读取 `offsetHeight` 为什么可能有成本？

**追问答案**

- 因为它通常不会直接触发布局变化，浏览器处理成本更可控。
- 指的是频繁读布局、改布局、再读布局，导致浏览器不断被迫重新计算。
- 因为浏览器可能需要先把当前布局计算出来，才能返回准确值。

**容易答偏的地方**

- 只会背定义
- 不知道回流为什么成本更高
- 不会把它和具体优化手段对应起来

### 13. JavaScript 运行时性能怎么优化？比如长任务、频繁计算、节流防抖、Web Worker 等分别适合什么场景？

**简答版答案**

JavaScript 运行时性能优化，核心是在保证功能正确的前提下减少主线程压力。常见思路包括拆分长任务、降低无效计算、控制高频事件触发，以及把适合并行的计算放到 Web Worker。

**展开版答案**

很多页面打开不慢，但一操作就卡，这往往是运行时性能问题。

常见问题包括：

- JS 长任务阻塞主线程
- 高频事件触发过于频繁
- 重复计算太多
- 大量数据处理集中在主线程

对应优化手段可以这样理解：

- 长任务：拆分执行、分批处理、避免一次做太多事
- 高频事件：节流、防抖
- 重复计算：缓存结果、减少不必要计算
- 大量纯计算：考虑 Web Worker

Web Worker 适合什么场景？

- 大数据量计算
- 复杂解析
- 不依赖 DOM 的耗时任务

但也要注意：

- Worker 不能直接操作 DOM
- 线程通信本身也有成本

所以面试里不要把 Web Worker 说成万能方案，而是要强调“适合把重计算从主线程搬走”。

**关键点**

- 运行时性能问题常表现为交互卡顿
- 优化核心是减轻主线程压力
- Worker 适合重计算，不适合 DOM 操作

**常见追问**

- 长任务为什么会影响交互？
- 节流和防抖在性能优化里的意义是什么？
- Web Worker 有没有成本？

**追问答案**

- 因为主线程被长时间占用后，用户输入、渲染和回调处理都会被延后。
- 它们能控制高频事件处理次数，减少无意义计算和渲染压力。
- 有，线程创建、数据传输和维护复杂度都会增加，所以不是所有场景都值得上。

**容易答偏的地方**

- 只会说节流防抖
- 说不清运行时卡顿和加载慢的区别
- 把 Web Worker 说成万能性能优化手段

### 14. 前端性能监控和异常上报一般怎么做？为什么性能优化不能只停留在“感觉更快”？

**简答版答案**

前端性能监控通常会采集关键性能指标、资源耗时、接口耗时和异常信息，再统一上报到监控平台。性能优化不能只停留在“感觉更快”，因为主观感受不稳定，必须用真实数据验证优化是否有效。

**展开版答案**

性能优化做到后面，一定会落到监控和验证。

通常会监控这些内容：

- 页面关键性能指标
- 首屏相关指标
- 资源加载耗时
- 接口请求耗时
- JS 异常
- Promise 异常
- 资源加载失败

上报方式通常是：

- 在浏览器侧采集数据
- 结合 `Performance API`、错误监听等能力
- 统一发送到监控平台

为什么不能只靠感觉？
因为：

- 开发环境和真实用户环境差异大
- 不同网络和设备下结果差异大
- 某些优化会改善一个指标，但可能伤害另一个指标

所以真实项目里更重要的是：

- 优化前有基线
- 优化后有对比
- 上线后持续监控

这样才能形成可验证的闭环。

**关键点**

- 性能优化最终要靠数据闭环
- 监控既包括性能，也包括异常
- 优化前后都要有数据对比

**常见追问**

- `Performance API` 能做什么？
- 资源加载失败也要上报吗？
- 为什么真实用户监控很重要？

**追问答案**

- 可以帮助采集页面加载、资源时序等性能相关信息，是前端监控的基础能力之一。
- 要，因为它会直接影响页面可用性和用户体验，很多时候也是性能问题的侧面表现。
- 因为实验室环境再标准，也不能完全代表用户真实设备、网络和访问路径。

**容易答偏的地方**

- 只会说“接个监控平台”
- 不知道监控和验证是性能优化闭环的一部分
- 把性能监控和错误监控完全割裂开
