# Vue 面试复习清单

## 使用方式

- 先看问题，尝试自己回答
- 再对照简答版和展开版检查表达是否完整
- 最后补常见追问和容易答偏的地方

### 1. Vue 的核心思想是什么？它和原生 DOM 开发的区别是什么？

**简答版答案**

Vue 的核心思想可以概括为：声明式开发、组件化、数据驱动视图。和原生 DOM 手动操作相比，Vue 更强调“描述页面该是什么状态”，而不是每次都手写如何更新页面。

**展开版答案**

Vue 之所以能提高前端开发效率，本质上不是因为它“封装了很多 API”，而是它改变了组织页面和更新页面的方式。

Vue 的几个核心思想通常可以这样理解：

- 声明式开发
  - 你更关注页面在某种状态下应该长什么样
  - 不需要频繁手动操作 DOM
- 数据驱动视图
  - 数据变化后，视图会按框架规则更新
- 组件化
  - 把页面拆成可复用、可维护的小单元

如果是原生 DOM 开发，很多时候你要自己做这些事：

- 查找节点
- 修改文本
- 改 class
- 控制显示隐藏
- 维护多个状态之间的同步

而 Vue 的思路更像是：

- 先把状态和模板关系描述清楚
- 再由框架接管视图更新

所以 Vue 的价值不只是“少写代码”，更重要的是让页面状态管理、组件复用和维护成本更可控。

**关键点**

- Vue 核心是声明式、组件化、数据驱动
- Vue 不是完全不操作 DOM，而是把 DOM 更新交给框架调度
- 相比原生 DOM，更强调状态和视图之间的映射关系

**常见追问**

- 什么叫声明式开发？
- Vue 为什么能减少手动 DOM 操作？
- 组件化解决了什么问题？

**追问答案**

- 声明式开发强调描述结果，而不是一步步手动编排过程，比如直接描述“数据是什么，页面就怎么显示”。
- 因为 Vue 会根据状态变化统一触发视图更新，不需要开发者每次都手写 DOM 更新逻辑。
- 组件化让页面结构更清晰，也更利于复用、协作和维护。

**容易答偏的地方**

- 只说“Vue 就是 MVVM”
- 只强调语法方便，不讲它解决了什么问题
- 误以为 Vue 完全不需要关心 DOM 更新成本

### 2. Vue2 和 Vue3 的主要区别是什么？为什么 Vue3 要做这些变化？

**简答版答案**

Vue3 相比 Vue2 的核心变化主要在于响应式实现、组合式 API、性能优化、类型支持和整体架构设计。Vue3 做这些变化的目的，是为了让大型项目更好维护、性能更好、逻辑复用更自然。

**展开版答案**

面试里问 Vue2 和 Vue3 的区别，通常不是让你死背功能清单，而是看你能不能说出“为什么要升级”。

几个高频差异可以这样记：

- 响应式实现不同
  - Vue2 主要基于 `Object.defineProperty`
  - Vue3 主要基于 `Proxy`
- 逻辑组织方式不同
  - Vue2 更常见的是 Options API
  - Vue3 引入了 Composition API
- 性能和包体积优化
  - Vue3 在初始化、更新、Tree Shaking 等方面做了更多优化
- TypeScript 友好度更高
  - Vue3 的设计对 TS 支持更自然

为什么 Vue3 要引入 Composition API？

因为在 Vue2 里，复杂组件经常会出现：

- data 很长
- methods 很多
- computed/watch 分散
- 同一块业务逻辑被拆在多个选项里

逻辑按“选项类型”分散，而不是按“业务功能”聚合，组件一复杂就难维护。Vue3 通过 Composition API，让相关逻辑可以按功能组织在一起。

所以 Vue3 的变化，不只是“语法升级”，本质上是在解决：

- 复杂组件逻辑组织困难
- 响应式能力限制
- 大型项目维护成本高

**关键点**

- Vue3 不是简单语法升级，而是架构层面的演进
- 响应式、组合式 API、性能优化是三大主线
- Vue3 更适合中大型项目和类型化开发

**常见追问**

- Composition API 比 Options API 好在哪里？
- Vue3 为什么用 `Proxy` 替代 `Object.defineProperty`？
- Vue2 项目有必要马上迁到 Vue3 吗？

**追问答案**

- 它更适合按功能组织逻辑，逻辑复用和拆分也更自然。
- 因为 `Proxy` 对对象整体代理能力更强，对新增属性、删除属性、数组等场景处理更自然。
- 不一定，要结合团队成本、生态依赖和项目生命周期决定。

**容易答偏的地方**

- 只背 `setup`、`ref`、`reactive`
- 只讲新语法，不讲背后的设计动机
- 把 Vue2 和 Vue3 说成完全割裂的两套东西

### 3. Vue 的响应式原理是什么？Vue2 和 Vue3 分别怎么实现？

**简答版答案**

Vue 的响应式原理，本质上是“在数据被读取时收集依赖，在数据变化时通知相关副作用重新执行”。Vue2 主要通过 `Object.defineProperty` 劫持属性，Vue3 主要通过 `Proxy` 代理对象。

**展开版答案**

响应式是 Vue 最核心的能力之一。

你可以把它简单理解成一句话：

- 数据变化了，相关视图为什么会自动更新？

背后本质上依赖两件事：

- 依赖收集
- 派发更新

Vue2 的实现思路大致是：

- 在初始化时遍历对象属性
- 用 `Object.defineProperty` 给每个属性加上 `getter` 和 `setter`
- 读取属性时做依赖收集
- 修改属性时通知依赖更新

这种方式的问题是：

- 需要初始化时递归遍历
- 对新增属性、删除属性处理不自然
- 对数组部分变更场景要做额外处理

Vue3 的实现思路则更现代：

- 使用 `Proxy` 直接代理整个对象
- 在 `get` 时收集依赖
- 在 `set` 时触发更新

这样带来的好处是：

- 对对象整体代理更自然
- 新增和删除属性更容易感知
- 数组和复杂结构的处理能力更强

所以面试里更稳的答法不是只说“Vue2 用 defineProperty，Vue3 用 Proxy”，而是顺带补一句：

- Vue2 能做响应式，但能力边界更多
- Vue3 响应式覆盖更完整，也更灵活

**关键点**

- 响应式的核心是依赖收集和触发更新
- Vue2 主要基于 `Object.defineProperty`
- Vue3 主要基于 `Proxy`
- Vue3 在对象和数组监听能力上更自然

**常见追问**

- 什么是依赖收集？
- 为什么 Vue3 的响应式更强？
- 响应式和虚拟 DOM 是一回事吗？

**追问答案**

- 可以理解为在读取数据时，把当前会用到这个数据的副作用或渲染逻辑记录下来。
- 因为 `Proxy` 是对对象整体代理，不再局限于初始化时把每个属性单独劫持。
- 不是。响应式解决“数据变化如何感知”，虚拟 DOM 更偏“视图如何高效更新”。

**容易答偏的地方**

- 只会背 `defineProperty` 和 `Proxy`
- 不知道响应式真正要解决的问题是什么
- 把响应式、双向绑定、虚拟 DOM 混成一个概念

### 4. 为什么 Vue2 对数组和对象的某些修改监听不到？

**简答版答案**

因为 Vue2 的响应式基于 `Object.defineProperty`，它只能劫持初始化时已经存在的属性。对于对象新增属性、删除属性，以及数组通过下标直接修改、改 `length` 等场景，默认不能被正常监听。

**展开版答案**

这是 Vue2 响应式限制里很经典的一道题。

Vue2 在初始化时会对已有属性做劫持，但它并不能天然知道：

- 后面又新增了哪些对象属性
- 某个属性被删除了
- 数组下标被直接改了
- 数组长度被直接重置了

所以常见失效场景包括：

- `obj.newKey = value`
- `delete obj.key`
- `arr[index] = value`
- `arr.length = 0`

为什么数组方法有些又能触发更新？

因为 Vue2 对部分会修改数组本身的方法做了额外处理，比如：

- `push`
- `pop`
- `shift`
- `unshift`
- `splice`

这些方法会被框架包一层，从而触发更新。

所以这道题的核心不是“Vue2 不支持数组响应式”，而是：

- Vue2 能处理一部分数组变更
- 但对某些直接赋值方式和动态属性场景有天然限制

**关键点**

- Vue2 的限制来自 `Object.defineProperty` 的能力边界
- 新增属性、删除属性、数组下标修改是高频坑
- 数组变更方法能触发更新是因为做了额外处理

**常见追问**

- Vue2 怎么解决这些监听不到的问题？
- Vue3 为什么没有这些限制那么明显？
- 为什么 `splice` 常被推荐？

**追问答案**

- Vue2 里常见会用 `Vue.set` / `this.$set`，数组场景常用 `splice`。
- 因为 Vue3 用 `Proxy` 代理对象，对属性新增、删除等变化感知更自然。
- 因为 `splice` 是 Vue2 能拦截到的数组变更方法之一。

**容易答偏的地方**

- 直接说“Vue2 数组不是响应式的”
- 不知道对象新增属性为什么有问题
- 只记解决方案，不知道为什么会这样

### 5. `computed` 和 `watch` 有什么区别？分别适合什么场景？

**简答版答案**

`computed` 更适合从已有状态推导出新值，强调有返回值和缓存；`watch` 更适合监听数据变化后执行副作用逻辑，比如发请求、做异步处理或手动调用某些方法。

**展开版答案**

这两个 API 很容易被混着用，但它们的设计目标其实不同。

`computed` 的特点是：

- 更适合做派生值
- 有返回值
- 基于依赖缓存
- 依赖不变时不会重复计算

比如：

- 根据商品列表算总价
- 根据状态生成展示文案
- 根据输入值做同步格式化展示

`watch` 的特点是：

- 更适合观察某个状态变化
- 执行副作用逻辑
- 常用于异步操作或联动处理

比如：

- 监听搜索关键词变化后发请求
- 监听路由参数变化重新拉数据
- 监听开关变化做本地存储同步

所以更稳的判断方式是：

- 如果你要“算一个值”，优先想 `computed`
- 如果你要“数据变了后做一件事”，优先想 `watch`

**关键点**

- `computed` 偏派生值，`watch` 偏副作用
- `computed` 有缓存特性
- `watch` 更常用于异步逻辑和监听联动

**常见追问**

- 为什么说 `computed` 有缓存？
- `watch` 能不能替代 `computed`？
- Vue3 里的 `watchEffect` 怎么理解？

**追问答案**

- 因为它会基于依赖变化决定是否重新计算，而不是每次访问都重新执行。
- 可以在某些场景下实现类似效果，但语义不合适，也容易让逻辑变得更绕。
- 可以理解成自动收集依赖并立即执行副作用的一种监听方式。

**容易答偏的地方**

- 只会背“一个监听一个计算”
- 不知道副作用是什么意思
- 用 `watch` 去做大量本该由 `computed` 处理的派生逻辑

#### 补充：Vue3 里 `watch` 和 `watchEffect` 有什么区别？

**简答版答案**

`watch` 更像“指定监听谁”，只有监听源变化时才触发，而且能拿到新值和旧值；`watchEffect` 更像“先执行这段副作用，再自动收集里面用到的依赖”，写法更省，但依赖边界没那么显式。

**展开版答案**

这两个 API 都能做副作用监听，但适用场景不完全一样。

`watch` 的特点是：

- 需要明确指定监听源
- 依赖关系更清晰
- 可以拿到 `newValue` 和 `oldValue`
- 可以控制是否立即执行、深度监听等行为

它更适合这种场景：

- 我明确知道要监听哪个 `ref`
- 我需要比较新旧值
- 我希望监听边界尽量可控

`watchEffect` 的特点是：

- 会先立即执行一次
- 执行过程中自动收集内部访问到的响应式依赖
- 不需要手动写监听源
- 更适合快速写一段依赖驱动的副作用

它更适合这种场景：

- 逻辑里会用到多个响应式值
- 我不想手动一个个声明依赖
- 我更关注“这段副作用依赖什么就跟着跑”

所以可以这么记：

- `watch` 是“显式监听”
- `watchEffect` 是“自动收集依赖后执行副作用”

**关键点**

- `watch` 依赖来源明确，`watchEffect` 依赖来源自动收集
- `watch` 能拿到新旧值，`watchEffect` 默认拿不到
- `watchEffect` 更省代码，但有时不如 `watch` 清晰可控

**常见追问**

- `watchEffect` 和 `computed` 的区别是什么？
- 为什么 `watchEffect` 会立即执行？
- 多个依赖场景下该优先选哪个？

**追问答案**

- `computed` 主要产出派生值，`watchEffect` 主要执行副作用，它们目标不一样。
- 因为它要先运行一次，才能在运行过程中知道自己依赖了哪些响应式数据。
- 如果你更在意依赖明确、可控和新旧值比较，优先用 `watch`；如果只是快速根据多个响应式值执行一段副作用，`watchEffect` 会更顺手。

**容易答偏的地方**

- 把 `watchEffect` 说成只是 `watch` 的简写
- 忽略 `watch` 可以拿到新旧值这一点
- 用 `watchEffect` 写过于隐式、难排查的业务监听

**使用代码**

`watch` 示例：

```js
import { ref, watch } from 'vue';

const keyword = ref('');

watch(keyword, async (newValue, oldValue) => {
  console.log('新值：', newValue);
  console.log('旧值：', oldValue);

  if (!newValue) return;
  // 这里可以发请求、做异步逻辑
  // await fetchList(newValue);
});
```

`watchEffect` 示例：

```js
import { ref, watchEffect } from 'vue';

const keyword = ref('');
const page = ref(1);

watchEffect(() => {
  console.log('当前 keyword：', keyword.value);
  console.log('当前 page：', page.value);

  // 这里访问到了哪个响应式数据，
  // watchEffect 就会自动收集哪个依赖
  // fetchList({
  //   keyword: keyword.value,
  //   page: page.value,
  // });
});
```

一句话记忆：

- 需要明确监听源、拿新旧值，用 `watch`
- 只想快速根据用到的响应式数据执行副作用，用 `watchEffect`

### 6. `v-if` 和 `v-show` 有什么区别？

**简答版答案**

`v-if` 是条件性地创建和销毁节点，`v-show` 是通过控制 CSS 的显示隐藏来切换可见性。频繁切换更适合 `v-show`，初次条件不一定成立或切换不频繁时更适合 `v-if`。

**展开版答案**

这道题看起来简单，但本质是在考你对渲染成本的理解。

`v-if` 的特点是：

- 条件为假时，节点可能根本不渲染
- 条件变化时，会创建或销毁对应节点
- 初始条件不满足时更省首屏开销

`v-show` 的特点是：

- 节点通常会先渲染出来
- 只是通过 `display` 控制显示隐藏
- 切换时不涉及频繁销毁和重建

所以一般经验是：

- 切换频繁，用 `v-show`
- 切换不频繁，或者初始不一定展示，用 `v-if`

面试时如果能补一句“这不是死规则，要结合节点复杂度和交互频率判断”，会更完整。

**关键点**

- `v-if` 更偏结构层面的控制
- `v-show` 更偏显示状态切换
- 选择时本质上要考虑初始开销和切换成本

**常见追问**

- `v-if` 和 `v-show` 会不会影响生命周期？
- 为什么 `v-show` 更适合频繁切换？
- 哪个更适合大组件弹窗？

**追问答案**

- `v-if` 由于涉及创建和销毁，通常会更明显地影响组件生命周期；`v-show` 更偏保留实例后切换显示。
- 因为它不需要频繁创建和销毁节点，切换成本通常更低。
- 如果弹窗频繁开关，通常更偏向 `v-show`；如果很少打开且内部内容复杂，也可能考虑 `v-if`。

**容易答偏的地方**

- 只会背“一个 display none，一个 add/remove”
- 不知道它们和切换成本的关系
- 误以为所有场景都固定选一个

### 7. `v-for` 为什么要加 `key`？为什么不推荐用 index 做 key？

**简答版答案**

`key` 的作用是帮助 Vue 在 diff 过程中更稳定地识别节点身份，从而正确复用和移动元素。用 `index` 做 key 在列表增删、排序时容易让节点复用错位，导致状态错乱或渲染异常。

**展开版答案**

Vue 更新列表时，不是每次都把整段 DOM 全删了再重建，而是会尽量复用已有节点。

这时框架要回答一个问题：

- 新旧列表里的这个节点，到底是不是同一个元素？

`key` 就是这个“身份标识”。

如果 `key` 稳定，Vue 更容易做到：

- 正确复用节点
- 正确移动节点
- 避免不必要的销毁和重建

为什么不推荐用 `index`？

因为 `index` 只代表当前位置，不代表节点本身身份。

一旦列表发生：

- 插入
- 删除
- 排序

位置就会变，原来的 `index` 也会变，这样框架就可能把“不是同一个节点”的内容误判成“可以复用”，从而导致：

- 输入框内容错位
- 组件内部状态错乱
- 动画或过渡异常

所以更推荐用业务上真正稳定的唯一标识，比如 `id`。

**关键点**

- `key` 是节点身份标识，不只是为了消除警告
- 稳定 `key` 有助于正确 diff 和节点复用
- `index` 在动态列表中不稳定

**常见追问**

- 什么场景下用 `index` 问题不大？
- 不写 `key` 会怎样？
- `key` 和性能有什么关系？

**追问答案**

- 如果列表完全静态、不涉及增删改排序，问题会小一些，但仍不算最佳实践。
- Vue 会使用默认策略处理，但在复杂列表更新中更容易出现复用不准确的问题。
- 稳定的 `key` 能帮助框架更正确地复用节点，减少错误更新和不必要的 DOM 操作。

**容易答偏的地方**

- 只会说“`key` 是为了性能”
- 不知道本质是节点身份识别
- 说不清为什么 `index` 在排序场景下会出问题

### 8. Vue 生命周期有哪些？常见生命周期适合做什么？

**简答版答案**

Vue 生命周期可以理解为组件从创建、挂载、更新到销毁的一系列阶段。面试里重点不是把名字全背完，而是说清“在哪个阶段做什么事更合适”。

**展开版答案**

生命周期本质上是在描述：

- 组件什么时候开始创建
- 什么时候模板已经渲染
- 什么时候数据更新完成
- 什么时候组件准备销毁

如果按常见阶段理解，大致可以分为：

- 创建阶段
- 挂载阶段
- 更新阶段
- 销毁阶段

在 Vue2 里常见会提到：

- `beforeCreate`
- `created`
- `beforeMount`
- `mounted`
- `beforeUpdate`
- `updated`
- `beforeDestroy`
- `destroyed`

实际面试中最常用的几个通常是：

- `created`
  - 适合做数据初始化、发起请求准备
- `mounted`
  - 适合操作真实 DOM、初始化第三方库
- `beforeDestroy`
  - 适合清理定时器、解绑事件

回答这题最重要的不是背顺序，而是知道每个阶段能拿到什么、适合干什么。

**关键点**

- 生命周期本质是组件不同运行阶段的钩子
- 高频重点通常在创建、挂载、更新、销毁四段
- 面试更关注“适合做什么”，不是单纯背名字

**常见追问**

- `created` 和 `mounted` 有什么区别？
- 为什么有些 DOM 操作要放到 `mounted`？
- 销毁阶段通常清理什么？

**追问答案**

- `created` 时数据和实例基本可用，但真实 DOM 还没挂载；`mounted` 时已经有真实 DOM 可以操作。
- 因为很多 DOM API 或第三方库初始化依赖真实挂载后的节点。
- 常见包括清理定时器、解绑全局事件、销毁订阅或第三方实例。

**容易答偏的地方**

- 只会背生命周期名字
- 不知道哪些阶段能不能操作 DOM
- 把 Vue2 和 Vue3 生命周期名字混在一起

### 9. Vue2 和 Vue3 生命周期名称上有什么区别？

**简答版答案**

Vue3 保留了生命周期阶段的大体含义，但部分命名发生了变化，尤其是销毁相关钩子由 `beforeDestroy` / `destroyed` 改成了 `beforeUnmount` / `unmounted`。同时在 Composition API 里，生命周期通常以 `onMounted`、`onUpdated` 这类形式使用。

**展开版答案**

这道题常出现在“你既会 Vue2 又会 Vue3 吗”的语境里。

Vue2 和 Vue3 生命周期的大体阶段没有本质变化，仍然是：

- 创建
- 挂载
- 更新
- 卸载

但 Vue3 的命名更偏向组件“挂载 / 卸载”语义，所以会看到：

- `beforeDestroy` -> `beforeUnmount`
- `destroyed` -> `unmounted`

另外，如果是 Composition API，用法也会变化，比如：

- `mounted` 对应 `onMounted`
- `updated` 对应 `onUpdated`
- `beforeUnmount` 对应 `onBeforeUnmount`

所以这题最稳的答法是：

- 含义主线没变
- 命名和使用方式更现代、更统一

**关键点**

- Vue3 主要是命名和调用方式变化，不是生命周期概念完全推翻
- 销毁相关变成了更语义化的 `unmount`
- Composition API 里常见 `onMounted` 这类写法

**常见追问**

- Vue3 为什么把 destroy 改成 unmount？
- Options API 和 Composition API 生命周期怎么对应？
- `setup` 和传统生命周期是什么关系？

**追问答案**

- 因为 Vue3 更强调组件挂载和卸载语义，命名更贴近组件树管理过程。
- 大体含义是一一对应的，只是 Composition API 使用的是函数式注册方式。
- `setup` 会比大多数传统生命周期更早执行，是组合式逻辑组织的入口。

**容易答偏的地方**

- 只背改名，不知道阶段本质没怎么变
- 把 `setup` 直接等同于某一个单独生命周期
- 说成 Vue3 生命周期和 Vue2 完全不同

### 10. Vue 组件通信有哪些方式？常见场景怎么选？

**简答版答案**

Vue 组件通信常见方式有 `props / emit`、`ref`、`provide / inject`、事件总线、状态管理等。选择方式的关键不是背 API，而是看组件关系：父子、兄弟、跨层级还是全局共享。

**展开版答案**

组件通信题本质上是在考你对组件关系和状态边界的理解。

常见方式可以按关系来记：

- 父传子
  - `props`
- 子传父
  - `emit`
- 父拿子实例
  - `ref`
- 跨层级传递
  - `provide / inject`
- 多组件共享状态
  - Vuex / Pinia

以前有些项目还会用事件总线，但现在更适合作为“了解过的旧方案”，不适合当成默认推荐。

所以一般可以这样选：

- 简单父子通信，优先 `props + emit`
- 祖先到后代的依赖传递，可考虑 `provide / inject`
- 全局共享状态，考虑状态管理
- 不要一上来就把所有通信都做成全局状态

面试里如果能补一句“通信方式选择本质上是在控制状态边界”，会更成熟。

**关键点**

- 组件通信方式很多，但要按关系选
- `props + emit` 仍然是最基本、最常见方案
- `provide / inject` 和状态管理适合不同层级问题

**常见追问**

- 兄弟组件怎么通信？
- `provide / inject` 能不能代替状态管理？
- 为什么不推荐事件总线滥用？

**追问答案**

- 常见可以通过共同父组件中转，或者使用状态管理统一维护。
- 在局部场景下可以，但它不等于完整的状态管理方案。
- 因为事件流向隐蔽、依赖关系不清晰，复杂项目里会提高维护成本。

**容易答偏的地方**

- 只会列 API，不会按场景选型
- 把所有共享状态都往全局 store 里塞
- 认为 `provide / inject` 能完全替代一切通信方式

### 11. 什么是 `nextTick`？它的作用是什么？

**简答版答案**

`nextTick` 的作用是让你在 Vue 完成一次异步 DOM 更新之后，再去拿更新后的 DOM 结果。因为 Vue 改数据后通常不会立刻同步更新 DOM，而是会把更新合并后异步处理。

**展开版答案**

很多人第一次接触 `nextTick`，会误以为它只是“延迟执行一个回调”，但它更准确的定位是：

- 在下一次 DOM 更新完成后执行逻辑

为什么需要它？

因为 Vue 为了性能，通常不会在你每次改数据时都立刻同步改 DOM，而是会：

- 先记录这次更新
- 合并同一轮中的多次变更
- 再统一更新 DOM

所以如果你刚改完数据就立刻去读 DOM，读到的可能还是旧结果。

这时就常用 `nextTick` 来确保：

- 等 DOM 更新完
- 再做依赖最新 DOM 的操作

典型场景包括：

- 更新列表后读取节点尺寸
- 打开弹窗后手动聚焦输入框
- 切换状态后获取最新渲染结果

**关键点**

- `nextTick` 不是单纯的“延迟一下”
- 它解决的是 DOM 更新时机问题
- 常用于需要依赖最新 DOM 的逻辑

**常见追问**

- 为什么 Vue 要异步更新 DOM？
- `nextTick` 和 `setTimeout` 有什么区别？
- Vue3 里怎么使用 `nextTick`？

**追问答案**

- 因为可以合并多次更新，减少重复渲染和 DOM 操作成本。
- `setTimeout` 是单纯把任务延后，`nextTick` 更强调 Vue 这轮 DOM 更新完成后的时机。
- Vue3 中通常直接从 `vue` 中导入 `nextTick` 使用。

**容易答偏的地方**

- 把 `nextTick` 理解成 Vue 版 `setTimeout`
- 不知道它和异步更新机制的关系
- 不知道它常用于读取更新后的 DOM

### 12. Vue 的异步更新机制怎么理解？为什么数据改了不会立刻更新 DOM？

**简答版答案**

Vue 数据变更后通常不会立即同步更新 DOM，而是会把同一轮事件循环里的多次更新合并，再统一执行渲染更新。这样做是为了减少重复渲染，提高性能。

**展开版答案**

这道题和 `nextTick` 强相关，但关注点更偏框架调度机制。

为什么 Vue 不直接“数据一改，DOM 就立刻改”？

因为真实项目里很容易出现：

```js
this.a = 1;
this.b = 2;
this.c = 3;
```

如果每改一次都马上更新 DOM，会导致很多无意义的重复渲染。

所以 Vue 通常会做这些事情：

- 收集这次状态变更
- 把相关 watcher 或副作用放进更新队列
- 去重和合并同一轮更新
- 在合适的异步时机统一刷新 DOM

如果把它和事件循环、`nextTick` 串起来，可以这样理解：

1. 同步代码里先修改响应式数据
2. Vue 感知到依赖变了，但不会立刻同步 patch DOM
3. 它会先把组件更新任务放进队列，并做去重
4. 然后借助微任务时机，在这轮同步代码执行完之后统一刷新队列
5. 刷新队列时，才会真正执行组件重新渲染和 DOM patch
6. DOM 更新完成后，`nextTick` 注册的回调再执行

所以从执行顺序上看，常见理解可以记成：

```text
改数据
-> 组件更新任务入队
-> 当前同步代码执行完
-> 微任务阶段刷新更新队列
-> Vue 完成 DOM patch
-> nextTick 回调执行
```

这也是为什么你经常会看到这种现象：

```js
state.count++;
console.log(el.textContent); // 这里可能还是旧 DOM

await nextTick();
console.log(el.textContent); // 这里通常才是新 DOM
```

因为前一个读取发生在“数据改了，但 DOM 还没真正 patch”的阶段；而 `nextTick` 会把逻辑放到这次 DOM 更新之后。

这样做的好处是：

- 减少重复更新
- 提高渲染效率
- 保证同一轮数据变更后的更新顺序更稳定

所以数据改了没立刻看到 DOM 变化，不是 Vue 失效了，而是框架在做批量调度优化。

**关键点**

- Vue 的异步更新机制本质是批量更新和调度优化
- 数据变化和 DOM 更新通常不是完全同步一对一
- `nextTick` 是拿到更新后 DOM 的常见方式

**常见追问**

- 异步更新机制为什么能提升性能？
- 队列去重是什么意思？
- 这和事件循环有什么关系？

**追问答案**

- 因为它把多次可能重复的 DOM 更新合并成更少的实际渲染动作。
- 指同一个响应式依赖或更新任务在同一轮中不会被无意义地重复加入很多次。
- Vue 的调度通常会借助微任务等异步时机，在当前同步逻辑结束后统一处理更新。数据变化后先入队，等队列刷新时再真正 patch DOM，而 `nextTick` 会在这次 DOM 更新完成后执行。

**容易答偏的地方**

- 只会说“Vue 更新是异步的”
- 不知道为什么异步更新反而更高效
- 不知道 `nextTick`、DOM patch、微任务之间的先后关系
- 把异步更新和网络异步混为一谈

**源码思路补充**

下面这两段不是完整源码，而是按 Vue2 / Vue3 的真实调度主线做的“核心代码提炼版”，重点只看：

- 更新任务入队
- 是否已经安排过本轮刷新
- 借助微任务统一 flush
- flush 时真正执行组件更新

`Vue2` 风格核心代码：

```js
const queue = [];
const has = {};
let waiting = false;

function queueWatcher(watcher) {
  const id = watcher.id;

  if (has[id]) return;
  has[id] = true;
  queue.push(watcher);

  if (!waiting) {
    waiting = true;
    nextTick(flushSchedulerQueue);
  }
}

function flushSchedulerQueue() {
  const flushingQueue = queue.slice();

  queue.length = 0;
  waiting = false;

  flushingQueue.forEach((watcher) => {
    has[watcher.id] = false;
    watcher.run();
  });
}

function nextTick(cb) {
  Promise.resolve().then(cb);
}

const watcher = {
  id: 1,
  run() {
    updateComponent();
  },
};

function updateComponent() {
  const vnode = render();
  patch(vnode);
}
```

这段代码对应的理解是：

- 数据变化后，不直接更新 DOM
- 而是先把对应的 `watcher` 放进队列
- `nextTick` 内部借助微任务，在当前同步代码结束后执行 `flushSchedulerQueue`
- `watcher.run()` 时才真正走 `render -> patch -> DOM 更新`

`Vue3` 风格核心代码：

```js
const queue = [];
let isFlushing = false;
let isFlushPending = false;
const resolvedPromise = Promise.resolve();

function queueJob(job) {
  if (!queue.includes(job)) {
    queue.push(job);
  }

  queueFlush();
}

function queueFlush() {
  if (isFlushing || isFlushPending) return;

  isFlushPending = true;
  resolvedPromise.then(flushJobs);
}

function flushJobs() {
  isFlushPending = false;
  isFlushing = true;

  try {
    for (const job of queue) {
      job();
    }
  } finally {
    queue.length = 0;
    isFlushing = false;
  }
}

const update = () => {
  const subTree = renderComponentRoot();
  patch(subTree);
};

queueJob(update);
```

这段代码对应的理解是：

- Vue3 不再主要讲 `watcher queue`，而更偏 `job queue`
- 组件更新函数会先入队
- `queueFlush` 保证本轮只安排一次微任务
- 到微任务阶段执行 `flushJobs`
- `job()` 执行时才真正重新渲染组件并 patch DOM

如果你要把 Vue2 和 Vue3 用一句话区分，可以这样记：

- Vue2 更常从 `watcher 入队 -> nextTick -> flushSchedulerQueue` 去理解
- Vue3 更常从 `job 入队 -> Promise 微任务 -> flushJobs` 去理解

它们表面 API 和实现细节不一样，但调度核心思想其实一致：

- 先收集更新
- 再批量调度
- 最后统一 patch DOM
- `nextTick` / 微任务逻辑发生在这套调度链路附近

### 13. Vue 的虚拟 DOM 和 diff 算法怎么理解？

**简答版答案**

虚拟 DOM 可以理解成真实 DOM 的一层 JavaScript 抽象表示。diff 算法的作用是比较新旧虚拟节点，尽量找出最小更新范围，再把必要变化同步到真实 DOM 上。

**展开版答案**

虚拟 DOM 不是 Vue 独有概念，但在 Vue 面试里很常被问。

它的核心作用不是“完全替代 DOM”，而是让框架可以先在 JavaScript 层描述和比较视图变化，再决定真实 DOM 怎么改。

为什么需要它？

因为真实 DOM 操作成本相对高，直接频繁手写更新逻辑也难维护。框架希望：

- 先根据状态生成新的虚拟节点树
- 再和旧树比较
- 最后只更新真正变化的部分

这就是 diff 的作用。

diff 的核心目标不是“算出绝对最优解”，而是：

- 在合理复杂度下尽量高效地找出变化

在 Vue 的实际更新里，`key`、节点类型、同层比较这些都会影响 diff 效果。

所以更稳的回答方式是：

- 虚拟 DOM 解决的是“如何在框架层组织视图更新”
- diff 解决的是“新旧视图如何高效比较”

**关键点**

- 虚拟 DOM 是视图结构的抽象表示
- diff 是比较新旧节点并确定最小更新范围
- 它不是完全不操作 DOM，而是让 DOM 更新更可控

**常见追问**

- 有了虚拟 DOM 就一定更快吗？
- 为什么 `key` 会影响 diff？
- Vue diff 是全量对比整棵树吗？

**追问答案**

- 不一定。虚拟 DOM 的价值更多是更新过程可控、跨平台和维护性提升，不是所有极端场景都绝对更快。
- 因为 `key` 会影响节点身份识别，直接关系到节点复用和移动策略。
- 不是简单暴力全量比较，通常会结合节点类型、层级和局部更新策略处理。

**容易答偏的地方**

- 只会说“虚拟 DOM 更快”
- 不知道虚拟 DOM 和 diff 各自解决什么问题
- 把虚拟 DOM 说成完全不需要真实 DOM

#### 补充：Vue2 和 Vue3 的 diff 有什么区别？

**简答版答案**

Vue2 和 Vue3 都是基于同层比较、尽量复用节点的思路做 diff，但 Vue3 在编译优化和运行时更新策略上更激进。简单说，Vue2 更偏“运行时做通用比较”，Vue3 更偏“编译阶段先告诉运行时哪里可能变，再结合更高效的 diff 策略去更新”。

**展开版答案**

面试里如果问 Vue2 / Vue3 的 diff 区别，重点通常不是背源码细节，而是说清楚“为什么 Vue3 更新更高效”。

先说共同点：

- 都不是全量暴力重建 DOM
- 都会尽量复用同类型节点
- 列表更新时都依赖 `key` 帮助识别节点身份
- diff 主要还是发生在同层节点之间

再说高频差异：

`Vue2` 的特点更偏：

- 运行时做较通用的 vnode 比较
- 子节点更新常见用双端比较思路
- 编译阶段给运行时的优化提示相对少
- 很多场景要到 patch 时才知道哪些地方需要细查

`Vue3` 的特点更偏：

- 编译阶段会产出更多优化信息
- 运行时可以借助 `PatchFlag`、动态节点收集等信息，跳过很多不必要比较
- 处理带 key 的子节点时，优化策略更强
- 在节点移动场景里会结合最长递增子序列思路，尽量减少真实 DOM 移动

所以 Vue3 的提升不只是“diff 算法本身更快”，而是：

- 编译器先帮运行时缩小比较范围
- 运行时再用更细的标记和更高效的节点移动策略做更新

如果用一句更面试化的话概括：

- Vue2 更偏“通用运行时 diff”
- Vue3 更偏“编译优化 + 更精准运行时 diff”

**关键点**

- Vue3 diff 的优势不只在运行时，还在编译阶段
- `PatchFlag`、静态提升、动态节点收集是 Vue3 常见优化关键词
- 列表重排场景里，Vue3 常会提到最长递增子序列优化移动次数

**常见追问**

- 什么是 `PatchFlag`？
- 最长递增子序列在 diff 里是干什么的？
- Vue3 是不是完全推翻了 Vue2 的 diff 思路？

**追问答案**

- 可以理解为编译阶段打给 vnode 的“变化标记”，告诉运行时哪些部分是动态的，更新时可以更有针对性。
- 它的作用是尽量减少节点移动次数，在列表重排时复用更多已有 DOM。
- 不是完全推翻。核心目标还是复用节点、减少不必要更新，只是 Vue3 把编译优化和运行时优化结合得更紧。

**容易答偏的地方**

- 只会说“Vue3 比 Vue2 快”
- 把 diff 区别全讲成最长递增子序列，忽略编译优化
- 以为 Vue3 diff 和 Vue2 完全没有继承关系

### 14. 为什么说 Vue 的模板编译不是简单的字符串替换？

**简答版答案**

Vue 的模板编译不是把模板直接粗暴替换成字符串，而是会先解析模板，生成语法树，再转成渲染函数或可执行结构。它本质上是一次“模板到渲染逻辑”的编译过程。

**展开版答案**

很多人会把模板语法理解成“最终就是字符串拼接”，这其实过于粗糙。

Vue 模板大致会经历：

- 模板解析
- 生成 AST
- 优化静态内容
- 生成渲染函数

也就是说，模板并不是浏览器天然认识的一套语言，而是 Vue 要先把它编译成更底层、更可执行的渲染逻辑。

为什么这很重要？

因为很多框架能力都依赖编译阶段：

- 指令解析
- 条件渲染
- 列表渲染
- 静态节点提升
- patch flag 等优化

所以说模板编译不是简单字符串替换，核心是想说明：

- Vue 模板背后是有编译链路和优化空间的

**关键点**

- 模板会经过解析、AST、代码生成等步骤
- 模板编译结果更接近渲染函数
- 编译阶段也是性能优化的重要入口

**常见追问**

- AST 是什么？
- 运行时和编译时有什么区别？
- Vue3 编译优化体现在哪？

**追问答案**

- AST 可以理解成源代码对应的抽象语法树，是编译过程中的中间表示。
- 编译时更偏模板分析和代码生成，运行时更偏响应式和实际渲染更新。
- 常见会体现在静态提升、patch flag、Block Tree 等优化思路上。

**容易答偏的地方**

- 只会说“模板最终会变成 render”
- 不知道中间还有解析和优化过程
- 把模板编译理解成普通文本替换

### 15. Vue Router 的原理是什么？hash 和 history 有什么区别？

**简答版答案**

Vue Router 本质上是监听 URL 变化，再根据不同路径映射到对应组件渲染。`hash` 模式基于 URL 中的 `#` 变化，`history` 模式基于 History API，看起来更自然，但服务端需要配合处理路由回退问题。

**展开版答案**

路由的核心问题其实是：

- URL 变了，页面为什么不用整页刷新也能切换内容？

Vue Router 做的事情大致是：

- 监听 URL 变化
- 找到当前路径对应的路由配置
- 渲染匹配的组件

常见两种模式：

- `hash`
  - URL 中会带 `#`
  - `#` 后面的变化通常不会触发传统整页请求
  - 部署更简单
- `history`
  - URL 更自然，没有 `#`
  - 基于 `pushState`、`replaceState` 等 History API
  - 刷新或直达子路径时，服务端要做兜底配置

所以它们的核心差别不只是“美不美观”，还包括：

- 路径表现形式
- 服务端是否要额外配合
- 部署和回退处理成本

**关键点**

- Vue Router 的核心是 URL 与组件映射
- `hash` 更简单，`history` 更自然
- `history` 模式通常需要服务端兜底

**常见追问**

- 为什么 `hash` 不容易 404？
- `history` 模式为什么刷新会有问题？
- 路由守卫通常用来做什么？

**追问答案**

- 因为 `#` 后面的内容通常不参与服务端路由匹配。
- 因为服务端如果没有把所有前端路由都回退到同一个入口文件，就可能找不到对应资源。
- 常见用于权限控制、登录校验、页面跳转前后的一些拦截处理。

**容易答偏的地方**

- 只会说“一个有 # 一个没 #”
- 不知道 `history` 模式为什么需要服务端配合
- 不知道路由原理本质是 URL 到组件的映射

#### 补充：Vue Router 两种实现方式的关键代码

**核心思路**

- `hash` 模式：监听 `hashchange`，从 `location.hash` 里拿路径
- `history` 模式：监听 `popstate`，通过 `pushState / replaceState` 改路径

`hash` 模式关键代码：

```js
class HashRouter {
  constructor(routes) {
    this.routes = routes;
    this.current = this.getPath();

    window.addEventListener('hashchange', () => {
      this.current = this.getPath();
      this.render();
    });

    window.addEventListener('load', () => {
      if (!location.hash) {
        location.hash = '/';
      }
      this.current = this.getPath();
      this.render();
    });
  }

  getPath() {
    return location.hash.slice(1) || '/';
  }

  push(path) {
    location.hash = path;
  }

  render() {
    const component = this.routes[this.current];
    console.log('渲染组件:', component);
  }
}
```

这段代码的关键点是：

- 路径信息放在 `#` 后面
- 改 `hash` 会触发 `hashchange`
- 浏览器不会把 `#` 后的内容当成服务端真实路径去请求

`history` 模式关键代码：

```js
class HistoryRouter {
  constructor(routes) {
    this.routes = routes;
    this.current = this.getPath();

    window.addEventListener('popstate', () => {
      this.current = this.getPath();
      this.render();
    });

    window.addEventListener('load', () => {
      this.current = this.getPath();
      this.render();
    });
  }

  getPath() {
    return location.pathname || '/';
  }

  push(path) {
    history.pushState({}, '', path);
    this.current = path;
    this.render();
  }

  replace(path) {
    history.replaceState({}, '', path);
    this.current = path;
    this.render();
  }

  render() {
    const component = this.routes[this.current];
    console.log('渲染组件:', component);
  }
}
```

这段代码的关键点是：

- 路径直接体现在 `location.pathname`
- 前进后退时会触发 `popstate`
- `pushState` 改了地址栏，但不会主动触发页面刷新
- 浏览器刷新时会真的拿这个路径去请求服务端，所以需要服务端兜底

一句话记忆：

- `hash`：改的是 `#`，监听 `hashchange`
- `history`：改的是路径，核心是 `pushState / popstate`

### 16. Vuex 和 Pinia 有什么区别？在 Vue2/3 里分别怎么选？

**简答版答案**

Vuex 是 Vue 生态里较早期的集中式状态管理方案，Pinia 是新一代更轻量、更贴近 Vue3 的状态管理方案。Vue3 项目通常更偏向 Pinia，Vue2 老项目则更常见 Vuex。

**展开版答案**

状态管理题里，面试官通常不是只问你 API，而是在问你有没有状态边界意识。

Vuex 的特点通常是：

- 体系完整
- 概念较多
- 早期 Vue2 项目里使用很多

Pinia 的特点通常是：

- 更轻量
- API 更直接
- 对 TypeScript 和 Vue3 更友好
- 使用体验更贴近 Composition API

为什么现在很多项目更推荐 Pinia？

因为它在多数业务场景下：

- 学习成本更低
- 模板代码更少
- 类型推导更自然

如果是 Vue3 新项目，通常更推荐 Pinia；如果是 Vue2 旧项目，面试里也要知道 Vuex 仍然很常见。

所以更成熟的回答方式不是“Pinia 一定比 Vuex 强”，而是：

- 新项目和新生态里，Pinia 更主流
- 老项目维护里，Vuex 仍然是重要语境

**关键点**

- Vuex 和 Pinia 都解决多组件共享状态问题
- Pinia 更轻量，也更贴近 Vue3
- Vue2 老项目仍然常见 Vuex

**常见追问**

- 什么状态才值得放到全局？
- Pinia 为什么更适合 Vue3？
- Vuex 和组件本地状态怎么划分边界？

**追问答案**

- 通常是多个组件共享、跨页面复用、具有全局意义的状态更适合放到全局。
- 因为它和 Composition API、类型系统、现代 Vue 生态配合更自然。
- 本地状态更适合只在当前组件内部使用的临时数据，全局状态更适合跨组件共享的数据。

**容易答偏的地方**

- 把所有状态都放进 store
- 只会背 API，不会讲选型理由
- 说成 Vue3 里 Vuex 完全不能用了

### 17. Vue 项目里常见性能优化手段有哪些？

**简答版答案**

Vue 项目性能优化通常可以从组件渲染、状态设计、路由加载、资源体积、列表更新和缓存策略等方向展开。面试里更重要的是能结合项目场景说“为什么要这样优化”，而不是只背一串技巧名词。

**展开版答案**

性能优化题如果只罗列概念，通常会显得比较空。更好的讲法是按几个层面拆：

- 组件层
  - 合理拆分组件，减少无意义更新
  - 避免把过多状态集中在单个大组件里
- 渲染层
  - 稳定使用 `key`
  - 减少高频无意义响应式更新
  - 大列表场景考虑虚拟列表
- 路由层
  - 路由懒加载
  - 按页面维度拆包
- 状态层
  - 不要把所有数据都做成全局响应式
  - 避免过深、过大的响应式对象
- 资源层
  - 图片压缩
  - Tree Shaking
  - 依赖按需加载

如果是 Vue 项目，还可以结合：

- `keep-alive`
- 异步组件
- 合理使用 `computed`
- 避免滥用 `watch`

真正面试里最加分的是：

- 不是只会背“懒加载、缓存、压缩”
- 而是能说清你在什么项目场景下为什么用了这些手段

**关键点**

- Vue 性能优化要分层讲：组件、渲染、路由、状态、资源
- 优化目标通常是减少不必要更新和降低首屏成本
- 结合真实项目场景最有说服力

**常见追问**

- `keep-alive` 适合什么场景？
- 为什么大列表要考虑虚拟列表？
- `computed` 为什么有时也算性能优化手段？

**追问答案**

- 常见适合页面切换后希望保留状态、减少重复初始化成本的场景。
- 因为真实渲染节点太多时，DOM 和布局成本会明显上升，虚拟列表可以减少同时存在的节点数量。
- 因为它能对派生值做缓存，减少重复计算和不必要的渲染开销。

**容易答偏的地方**

- 只会背“路由懒加载”
- 把所有问题都归结为打包体积大
- 不会结合 Vue 的响应式和组件更新机制去解释

### 18. 在真实项目里，如何结合 Vue2/3 讲你的框架能力和优化经验？

**简答版答案**

这道题重点不在背 Vue 知识点，而在于把你的框架理解落到真实项目里。比较稳的讲法是：先讲业务背景，再讲框架选型或技术难点，再讲你如何用 Vue2/3 的能力解决问题，最后讲结果和复盘。

**展开版答案**

面试里真正高阶一点的框架能力题，往往会从八股转到项目表达：

- 你在项目里怎么用 Vue？
- 你为什么这样设计？
- 你做过哪些优化？

这时建议你的表达结构固定一些，比如：

1. 业务背景
2. 具体问题
3. 技术方案
4. Vue2/3 在其中扮演什么角色
5. 最终效果

例如你可以从这些角度切入：

- 组件拆分与复用
- 表单和复杂交互状态管理
- 路由权限与页面缓存
- 大列表渲染优化
- 老 Vue2 项目到 Vue3 的迁移经验

回答时尽量避免：

- 一直讲框架 API
- 不结合业务目标
- 没有结果和收益

更加分的方式通常是：

- 能说明为什么这么选
- 能说明你踩过什么坑
- 能说明你如何权衡复杂度、维护性和性能

**关键点**

- 项目表达题核心是“框架能力如何落地”
- 用业务问题 -> 技术方案 -> 结果收益的结构更稳
- Vue2 / Vue3 的差异也可以通过项目经验自然带出来

**常见追问**

- 你在项目里做过最有代表性的 Vue 优化是什么？
- Vue2 升 Vue3 的经验能怎么讲？
- 怎么证明你的优化真的有效？

**追问答案**

- 可以优先讲一个你自己主导过、收益明确、链路完整的优化点，比如页面首屏优化、组件重构或状态管理治理。
- 可以从迁移动机、兼容问题、团队成本、代码收益几个方面展开。
- 最好结合具体指标，比如加载时间、交互流畅度、错误率、维护成本或开发效率变化来说明。

**容易答偏的地方**

- 只会讲 API，不会讲项目场景
- 一直说“做了优化”，却说不清优化前后差异
- 把项目题答成背八股
